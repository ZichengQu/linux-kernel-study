# 0. 这组文档的核心主线

这组文档不是单纯讲“有哪些锁”，而是在讲 Linux 内核面对并发时的完整解决体系：

```text
多个 CPU / 多个内核线程 / 中断 / softirq / 用户线程
        同时访问共享数据
              ↓
        可能发生竞态、乱序、可见性问题
              ↓
        需要同步机制保证正确性
```

你要先把内核同步分成三类问题：

## 0.1 第一类：原子性问题

例如：

```c
counter++;
```

看起来是一条语句，实际可能拆成：

```c
tmp = counter;
tmp = tmp + 1;
counter = tmp;
```

如果 CPU0 和 CPU1 同时执行，就可能出现：

```text
初始 counter = 0

CPU0 读到 0
CPU1 读到 0
CPU0 写回 1
CPU1 写回 1

最终结果是 1，不是 2
```

这就是原子性问题。

解决方法：

* atomic 操作；
* spinlock；
* mutex；
* rwsem；
* RCU；
* seqlock；
* Per-CPU 分散访问。

---

## 0.2 第二类：可见性问题

CPU0 写了：

```c
data = 100;
ready = 1;
```

CPU1 读：

```c
if (ready)
    printk("%d\n", data);
```

你以为 CPU1 看到 `ready == 1` 时，一定能看到 `data == 100`。

但在多核 + cache + store buffer + 编译器优化下，不一定。

这就是 memory barrier 要解决的问题。

Linux 官方 memory barrier 文档也明确说明，内核里的锁操作、读写锁、mutex、semaphore、rwsem 等 acquire/release 操作都隐含一定的 barrier 语义。([内核官网][2])

---

## 0.3 第三类：生命周期问题

这个是 RCU 最典型。

比如一个链表节点：

```text
reader 正在读节点 A
writer 把节点 A 从链表删除并 kfree(A)
reader 继续访问 A
```

这就变成 use-after-free。

RCU 解决的不是简单的互斥，而是：

> 允许读者继续读旧对象，写者发布新对象，但旧对象必须等所有旧读者离开后才能释放。

Linux RCU 文档也把 RCU 的核心描述为：把破坏性操作拆成两部分，一部分让别人看不到旧对象，另一部分真正销毁旧对象，中间必须经过 grace period。([内核官网][3])

# 2. 第一组：基础概念

## 2.1 为什么内核需要同步机制？

内核不是单线程程序。

同一份内核数据可能被这些执行流访问：

```text
进程上下文
内核线程
硬中断
软中断
timer
workqueue
多个 CPU 上并行执行的内核代码
```

比如调度器里的 runqueue：

```c
struct rq {
    raw_spinlock_t __lock;
    struct cfs_rq cfs;
    struct rt_rq rt;
    ...
};
```

每个 CPU 有自己的 rq，但跨 CPU 负载均衡、唤醒迁移、pull/push task 时，其他 CPU 也可能访问这个 rq。

所以需要锁。

再比如网络协议栈：

```text
一个 CPU 正在收包
另一个 CPU 正在关闭 socket
另一个 CPU 正在 timer 超时重传
```

如果没有同步，很容易出现：

* 链表损坏；
* 引用计数错误；
* use-after-free；
* double free；
* 死锁；
* 数据读到一半被改；
* 一个 CPU 看不到另一个 CPU 写入的数据。

---

## 2.2 临界区是什么？

临界区就是：

> 访问共享数据、并且不能被其他执行流同时破坏的代码区域。

例如：

```c
spin_lock(&lock);

list_add(&node->list, &head);
count++;

spin_unlock(&lock);
```

这里 `list_add()` 和 `count++` 合起来就是临界区。

重点是：临界区不一定只有一条语句，它通常是一组必须保持一致的操作。

---

## 2.3 同步机制大体分两类

### 第一类：忙等锁

拿不到锁时不睡眠，而是在 CPU 上转圈等。

典型：

```text
spinlock
rw spinlock
seqlock 写侧
queued spinlock
```

特点：

```text
优点：不能睡眠的上下文也能用，比如中断上下文
缺点：等待期间浪费 CPU
适合：临界区很短
```

---

### 第二类：睡眠锁

拿不到锁时当前 task 进入睡眠，让出 CPU。

典型：

```text
mutex
semaphore
rwsem
futex 慢路径
```

特点：

```text
优点：等待时间长也不浪费 CPU
缺点：不能用于中断上下文
适合：临界区较长，或者可能睡眠
```

mutex 官方文档也明确说，mutex 是 sleeping lock，拿不到时会睡眠，并且不能在 interrupt context 中使用。([内核文档][4])

---

# 3. 原子操作 atomic

wowotech 原子操作文章从一个普通变量修改开始解释：对内存变量的修改通常包括读内存、修改寄存器、写回内存三个步骤。([Wowotech][5])

## 3.1 atomic 解决什么问题？

解决单个变量的原子修改。

例如：

```c
atomic_t refcnt = ATOMIC_INIT(1);

atomic_inc(&refcnt);
atomic_dec(&refcnt);
```

它保证：

```text
读 old value
计算 new value
写回 new value
```

这几个动作不会被其他 CPU 插进来导致丢更新。

---

## 3.2 atomic 最常见用途：引用计数

比如一个对象：

```c
struct obj {
    atomic_t refcnt;
    ...
};
```

获取对象时：

```c
atomic_inc(&obj->refcnt);
```

释放对象时：

```c
if (atomic_dec_and_test(&obj->refcnt))
    kfree(obj);
```

意思是：

```text
如果 refcnt 减 1 后变成 0，说明没人用了，可以释放。
```

---

## 3.3 atomic 和普通 int 的区别

普通 int：

```c
int count;
count++;
```

可能被打断。

atomic：

```c
atomic_t count;
atomic_inc(&count);
```

保证这次修改是原子的。

---

## 3.4 atomic 和 spinlock 的区别

atomic 适合这种：

```c
counter++;
counter--;
test_and_set_bit();
```

也就是单变量、单动作。

spinlock 适合这种：

```c
if (!list_empty(&head)) {
    node = list_first_entry(&head, struct node, list);
    list_del(&node->list);
}
```

因为这里有多步：

```text
判断链表非空
取第一个节点
删除节点
更新链表指针
```

这不是一个 atomic 操作能保护的。

---

## 3.5 atomic 的限制

atomic 不是万能锁。

它不能保护复杂结构的一致性。

例如：

```c
struct pair {
    int a;
    int b;
};
```

如果要求：

```text
a 和 b 必须同时更新
读者不能看到 a 是新值，b 是旧值
```

那 atomic 不够，需要锁、seqlock 或 RCU。

# 4. Per-CPU 变量

wowotech Per-CPU 文章的核心问题是：共享内存上的原子操作会导致硬件层面的总线/cache 一致性开销，Per-CPU 变量通过“每个 CPU 一份副本”来减少共享竞争。([Wowotech][6])

## 4.1 Per-CPU 变量解决什么问题？

普通共享变量：

```text
CPU0 CPU1 CPU2 CPU3 都修改同一个 counter
```

问题：

```text
同一条 cacheline 在多个 CPU 间来回迁移
需要原子操作
性能差
```

Per-CPU 变量：

```text
CPU0 修改 counter[0]
CPU1 修改 counter[1]
CPU2 修改 counter[2]
CPU3 修改 counter[3]
```

这样每个 CPU 修改自己的副本，避免竞争。

---

## 4.2 典型场景

### 场景 1：统计计数

比如网络收包计数：

```text
CPU0 收包 1000 个
CPU1 收包 800 个
CPU2 收包 1200 个
```

没必要每次都全局 atomic 加一。

可以每 CPU 一个计数器：

```c
DEFINE_PER_CPU(unsigned long, rx_packets);
```

收包时：

```c
this_cpu_inc(rx_packets);
```

读取总数时再汇总：

```c
for_each_possible_cpu(cpu)
    total += per_cpu(rx_packets, cpu);
```

---

### 场景 2：调度器数据

Linux 调度器大量使用 per-cpu 数据。

例如：

```text
每个 CPU 有自己的 runqueue
每个 CPU 有自己的 load
每个 CPU 有自己的 tick 状态
```

这样本地调度不需要全局大锁。

---

## 4.3 Per-CPU 的核心思想

它不是“保护共享数据”，而是：

> 通过拆分数据，让数据尽量不共享。

这是非常重要的内核优化思想。

也就是：

```text
加锁是下策
减少共享才是上策
```

---

## 4.4 使用 Per-CPU 变量的注意点

### 注意 1：当前任务可能迁移 CPU

如果你这样写：

```c
p = this_cpu_ptr(&var);
/* 当前 task 被迁移到另一个 CPU */
*p = 1;
```

就可能出问题。

所以访问 per-cpu 数据时，很多场景要确保：

```text
当前执行流不会迁移 CPU
```

常见方式：

```c
preempt_disable();
...
preempt_enable();
```

或者使用内核提供的 this_cpu 操作，它内部根据语义处理。

---

### 注意 2：Per-CPU 适合局部统计，不适合强一致读

如果你要读一个完全精确的全局值，Per-CPU 变量可能要遍历所有 CPU 汇总。

汇总期间其他 CPU 还在更新，所以这个值可能只是近似快照。

---

# 5. memory barrier：内存屏障

wowotech memory barrier 文章和后续 Why Memory Barriers/perfbook 翻译，是这个分类里最难但也最重要的一组。memory barrier 文章从 CPU cache、store buffer、invalidate 等角度解释为什么 CPU0 写了变量，CPU1 不一定马上以你想象的顺序看到。([Wowotech][7])

## 5.1 为什么需要 memory barrier？

你写的代码：

```c
a = 1;
b = 1;
```

你以为一定是：

```text
先写 a
再写 b
```

但实际有三层可能改变顺序：

```text
1. 编译器优化重排
2. CPU 指令乱序执行
3. cache/store buffer 导致其他 CPU 看到的顺序不一致
```

所以 CPU0 的执行顺序、CPU0 对外可见顺序、CPU1 观察到的顺序，可能不是同一个东西。

---

## 5.2 最经典例子

```c
int data = 0;
int ready = 0;
```

CPU0：

```c
data = 100;
ready = 1;
```

CPU1：

```c
if (ready == 1)
    printk("%d\n", data);
```

你期待输出：

```text
100
```

但如果没有内存屏障，理论上 CPU1 可能看到：

```text
ready == 1
data == 0
```

原因是：

```text
CPU0 对 ready 的写先被 CPU1 看见
CPU0 对 data 的写还停在 store buffer 或没有被 CPU1 看见
```

---

## 5.3 memory barrier 不是锁

这个很重要。

锁解决：

```text
谁可以进入临界区
```

barrier 解决：

```text
内存访问顺序如何对其他 CPU 可见
```

例如：

```c
smp_wmb();
smp_rmb();
smp_mb();
```

这些不会阻止两个 CPU 同时执行代码。

它只是约束内存访问顺序。

---

## 5.4 编译器乱序 Compiler Reordering

wowotech 还有一篇专门讲“编译乱序”，强调编译器会为了优化性能调整代码顺序。([Wowotech][8])

比如你写：

```c
x = 1;
y = 2;
```

如果编译器认为不影响单线程语义，它可能换顺序。

但内核是并发程序，另一个 CPU 可能正在观察 `x` 和 `y`。

所以内核里有：

```c
barrier();
READ_ONCE(x);
WRITE_ONCE(x, v);
```

---

## 5.5 READ_ONCE / WRITE_ONCE 的意义

```c
READ_ONCE(x)
WRITE_ONCE(x, v)
```

主要解决：

```text
防止编译器把访问优化掉
防止编译器把一次访问拆成多次
防止编译器合并多次访问
让代码明确表达“这个变量可能被并发修改”
```

但注意：

```text
READ_ONCE/WRITE_ONCE 本身不是完整内存屏障。
```

它们更多是编译器层面的访问约束和单次访问保证。

---

## 5.6 常见 barrier 类型

| API                   | 作用          |
| --------------------- | ----------- |
| `barrier()`           | 编译器屏障       |
| `smp_rmb()`           | 读屏障         |
| `smp_wmb()`           | 写屏障         |
| `smp_mb()`            | 全屏障         |
| `smp_store_release()` | release 语义写 |
| `smp_load_acquire()`  | acquire 语义读 |

---

## 5.7 acquire/release 怎么理解？

release：

```text
我在 release 之前做的写入，必须先对外可见，
然后别人才能看到 release 这个动作。
```

acquire：

```text
我 acquire 成功之后，后面的读写不能跑到 acquire 前面。
```

典型模型：

CPU0：

```c
WRITE_ONCE(data, 100);
smp_store_release(&ready, 1);
```

CPU1：

```c
if (smp_load_acquire(&ready))
    printk("%d\n", READ_ONCE(data));
```

这样 CPU1 看到 `ready == 1` 后，就能看到 CPU0 在 release 前写入的 `data = 100`。

---

## 5.8 锁和 memory barrier 的关系

很多锁操作本身隐含 barrier。

Linux 官方 memory barrier 文档说明，spin locks、R/W spin locks、mutexes、semaphores、R/W semaphores 的 acquire/release 操作都有相应内存屏障语义。([内核官网][2])

所以：

```c
spin_lock(&lock);
x = 1;
spin_unlock(&lock);
```

不用你手动在里面加 `smp_mb()`，锁本身会提供必要的顺序保证。

但在无锁代码、RCU、atomic 状态机、唤醒标志位里，就经常需要手动考虑 memory order。

---

# 6. spinlock：自旋锁

wowotech spinlock 文章讲的是 Linux 内核中最基础的忙等锁。文章强调 spinlock 是“死等”的锁机制：资源冲突时，要么死等，要么挂起当前进程；spinlock 选择前者。([Wowotech][9])

## 6.1 spinlock 解决什么问题？

保护短临界区。

比如：

```c
spin_lock(&lock);
list_add(&node->list, &head);
spin_unlock(&lock);
```

当另一个 CPU 也想进来时：

```text
它不会睡眠
它会在 CPU 上循环等待
直到锁释放
```

---

## 6.2 为什么 spinlock 不能用于长临界区？

因为拿不到锁的 CPU 会一直空转：

```text
while (lock is busy)
    cpu_relax();
```

如果临界区很长，比如里面做 I/O、等待内存、睡眠，那其他 CPU 就白白浪费时间。

所以 spinlock 的原则：

```text
临界区必须短
不能睡眠
不能做可能阻塞的操作
```

---

## 6.3 spinlock 为什么适合中断上下文？

中断上下文不能睡眠。

例如硬中断 handler 中不能：

```c
mutex_lock(&lock);
```

因为 mutex 可能睡眠。

但可以用 spinlock，因为 spinlock 不睡眠。

---

## 6.4 spinlock 的常见 API

```c
DEFINE_SPINLOCK(lock);

spin_lock(&lock);
spin_unlock(&lock);
```

如果临界区可能和中断共享数据：

```c
unsigned long flags;

spin_lock_irqsave(&lock, flags);
/* critical section */
spin_unlock_irqrestore(&lock, flags);
```

如果只需要关本地 softirq：

```c
spin_lock_bh(&lock);
spin_unlock_bh(&lock);
```

---

## 6.5 spin_lock 和 spin_lock_irqsave 的区别

### `spin_lock()`

只拿锁，不关中断。

适合：

```text
不会被本 CPU 中断 handler 重新访问同一把锁
```

---

### `spin_lock_irqsave()`

做两件事：

```text
1. 保存当前中断状态
2. 关闭本地 CPU 中断
3. 获取 spinlock
```

释放时恢复：

```text
1. 释放 spinlock
2. 恢复之前中断状态
```

为什么需要？

假设进程上下文：

```c
spin_lock(&lock);
/* 正在操作 shared_data */
```

此时本 CPU 来了一个中断，中断 handler 也要：

```c
spin_lock(&lock);
```

就死锁了。

因为：

```text
进程上下文已经持锁
中断打断了进程上下文
中断 handler 又等这把锁
但锁只能等进程上下文继续执行后释放
可是进程上下文被中断 handler 打断了
```

所以如果锁会被中断 handler 使用，进程上下文拿锁时必须关本地中断。

---

## 6.6 spinlock 的隐含语义

拿 spinlock 时通常会：

```text
禁止内核抢占
```

否则当前 task 拿着 spinlock 被调度走，其他 task 在同 CPU 上可能浪费等待，甚至带来复杂问题。

所以 spinlock 不只是一个变量，它还和：

```text
preempt_disable
irq disable
memory barrier
架构原子指令
cacheline 竞争
```

强相关。

---

## 6.7 spinlock 的底层实现直觉

最简单的 spinlock 可以想象成：

```c
while (test_and_set(&lock->value) == locked)
    cpu_relax();
```

解锁：

```c
lock->value = unlocked;
```

但真实 Linux 复杂很多，因为要考虑：

```text
公平性
NUMA
cacheline 抖动
虚拟化
PV spinlock
queued spinlock
PREEMPT_RT
```

# 7. Linux 内核的自旋锁：更现代视角

wowotech 后面还有一篇“Linux 内核的自旋锁”，这是 2019 年的文章，通常比早期 spinlock 文章更适合作为补充阅读。([Wowotech][10])

这类文章重点一般会更贴近：

```text
ARM64 原子指令
ticket lock / queued lock
spin_lock 的架构实现
内存屏障
锁竞争路径
```

## 7.1 ARM64 上 spinlock 的典型思想

ARM64 上常见原子更新会基于独占访问思想：

```text
LDXR / LDAXR 读取并标记独占
STXR / STLXR 尝试写回
如果写回失败，说明期间被别人改过，重试
```

伪代码：

```c
do {
    old = exclusive_load(lock);
    if (old != 0)
        continue;
} while (exclusive_store(lock, 1) failed);
```

解锁：

```c
store_release(lock, 0);
```

这里 release 很重要，它保证临界区内写入在释放锁前对其他 CPU 可见。

# 8. Read/Write spin lock：自旋读写锁

wowotech rw spinlock 文章的核心是：普通 spinlock 不区分读者和写者，而读写锁允许多个 reader 并发，但 writer 必须独占。文章检索结果里也列出了 DEFINE_RWLOCK、rwlock_init 等 API。([Wowotech][11])

## 8.1 为什么需要 rw spinlock？

普通 spinlock：

```text
读者和读者互斥
读者和写者互斥
写者和写者互斥
```

但很多数据结构是读多写少。

比如：

```text
1000 次查询
1 次更新
```

查询之间其实不需要互斥。

所以 rw spinlock 允许：

```text
多个 reader 同时进入
writer 进入时必须没有 reader，也没有其他 writer
```

---

## 8.2 rw spinlock 的语义

| 当前状态     | 新 reader 能进吗 | 新 writer 能进吗 |
| -------- | -----------: | -----------: |
| 没人持锁     |           可以 |           可以 |
| 有 reader |           可以 |          不可以 |
| 有 writer |          不可以 |          不可以 |

---

## 8.3 常见 API

```c
DEFINE_RWLOCK(lock);

read_lock(&lock);
read_unlock(&lock);

write_lock(&lock);
write_unlock(&lock);
```

中断场景也有：

```c
read_lock_irqsave(&lock, flags);
read_unlock_irqrestore(&lock, flags);

write_lock_irqsave(&lock, flags);
write_unlock_irqrestore(&lock, flags);
```

---

## 8.4 rw spinlock 的问题

它不是读多写少的终极答案。

问题包括：

```text
1. writer 可能被大量 reader 饿死
2. reader 虽然并发，但还是要修改读计数，仍有 cacheline 竞争
3. 读侧仍然有锁开销
4. 对极致读性能场景，RCU 更合适
```

所以现代内核里，很多读多写少场景更倾向于 RCU，而不是 rwlock。

---

# 9. Seqlock

wowotech seqlock 文章说得很清楚：reader 侧在访问数据过程中检测是否有并发 writer，如果检测到，就重新读；不断 retry，直到读到一致的数据。检索结果也总结了 seqlock 特点：只允许一个 writer 进入；没有 writer 时，reader 可以随意进入；reader 不阻挡 reader。([Wowotech][12])

## 9.1 seqlock 解决什么问题？

它解决的是：

```text
读很多
写很少
读者不想拿锁
但读者可以接受重试
```

典型场景：

```text
时间数据
小型统计数据
一组需要一致读取的变量
```

---

## 9.2 seqlock 的核心思想

seqlock 里有一个序列号 `seq`。

writer 更新前：

```text
seq 变成奇数
```

writer 更新后：

```text
seq 变成偶数
```

reader 读取：

```text
1. 先读 seq
2. 如果 seq 是奇数，说明 writer 正在写，重试
3. 读取数据
4. 再读 seq
5. 如果两次 seq 不同，说明读期间发生写，重试
6. 如果相同且是偶数，说明数据一致
```

---

## 9.3 seqlock 伪代码

writer：

```c
write_seqlock(&seq_lock);

x = new_x;
y = new_y;

write_sequnlock(&seq_lock);
```

reader：

```c
do {
    seq = read_seqbegin(&seq_lock);

    local_x = x;
    local_y = y;

} while (read_seqretry(&seq_lock, seq));
```

---

## 9.4 为什么 seqlock 适合读时间？

比如系统时间可能由多个字段组成：

```c
struct time_data {
    u64 seconds;
    u64 nanoseconds;
};
```

writer 更新时间时，可能先写 seconds，再写 nanoseconds。

reader 如果读到一半 writer 改了，就可能看到：

```text
seconds 是新的
nanoseconds 是旧的
```

seqlock 可以让 reader 发现“不一致”，然后重读。

---

## 9.5 seqlock 的限制

seqlock 有几个非常重要的限制：

### 限制 1：保护的数据不能包含会被释放的指针

因为 reader 不加锁，writer 不能在 reader 读的时候释放对象。

例如：

```c
p = global_ptr;
use p->field;
```

如果 writer 释放了 `p`，reader 就 use-after-free。

这种场景应该用 RCU。

---

### 限制 2：reader 可能无限重试

如果 writer 很频繁，reader 可能一直读不到稳定版本。

所以 seqlock 适合：

```text
写少
写快
读多
数据小
```

---

### 限制 3：读侧不能有副作用

因为 reader 可能重试，所以读侧代码最好只是复制数据，不要做：

```text
分配内存
修改状态
发送消息
增加引用计数
```

否则重试会导致副作用重复发生。

---

# 10. RCU 基础

RCU 是这组文档最核心、最值得深入的一部分。wowotech 有 RCU 基础、RCU synchronize、多个历史版本实现、perfbook RCU 翻译、RCU 概述和使用方法等多篇。RCU 基础文章说明它一部分讲原理，一部分讲 Linux kernel 中的实现；RCU synchronize 文章关注 callback、grace period 和 synchronize_rcu 等机制。([Wowotech][13])

## 10.1 RCU 是什么？

RCU = Read-Copy-Update。

核心思想：

```text
Read：读者直接读，不阻塞
Copy：写者复制旧数据
Update：写者修改副本，然后发布新版本
最后：等待旧读者退出，再释放旧版本
```

Linux 官方文档也说，RCU 优化的是 read-mostly 场景，读路径非常轻量，更新侧更复杂。([内核文档][14])

---

## 10.2 RCU 和 rwlock 的区别

rwlock：

```text
reader 拿读锁
writer 拿写锁
writer 等 reader 全部退出
reader 也可能被 writer 阻塞
```

RCU：

```text
reader 不阻塞 writer
writer 不阻塞 reader
旧 reader 看旧数据
新 reader 看新数据
旧数据延迟释放
```

---

## 10.3 RCU 的餐厅菜单类比

旧菜单正在被顾客看。

老板要更新菜单：

```text
1. 不把旧菜单从顾客手里抢走
2. 先印一份新菜单
3. 后来的顾客看新菜单
4. 等正在看旧菜单的顾客都看完
5. 再回收旧菜单
```

对应 RCU：

```text
旧菜单 = old object
新菜单 = new object
顾客看菜单 = RCU read-side critical section
等待所有旧顾客看完 = grace period
回收旧菜单 = kfree old object
```

---

## 10.4 RCU 基本代码模式

读侧：

```c
rcu_read_lock();

p = rcu_dereference(global_ptr);
if (p)
    do_something(p);

rcu_read_unlock();
```

写侧：

```c
old = rcu_dereference_protected(global_ptr, lockdep_is_held(&lock));

new = kmalloc(sizeof(*new), GFP_KERNEL);
*new = *old;
new->field = new_value;

rcu_assign_pointer(global_ptr, new);

synchronize_rcu();

kfree(old);
```

---

## 10.5 RCU 的关键 API

| API                         | 作用                      |
| --------------------------- | ----------------------- |
| `rcu_read_lock()`           | 进入 RCU 读侧临界区            |
| `rcu_read_unlock()`         | 退出 RCU 读侧临界区            |
| `rcu_dereference()`         | 安全读取 RCU 保护的指针          |
| `rcu_assign_pointer()`      | 安全发布 RCU 新指针            |
| `synchronize_rcu()`         | 同步等待 grace period       |
| `call_rcu()`                | 异步等待 grace period 后调用回调 |
| `kfree_rcu()`               | grace period 后释放对象      |
| `list_for_each_entry_rcu()` | 遍历 RCU 链表               |
| `list_add_rcu()`            | 添加 RCU 链表节点             |
| `list_del_rcu()`            | 从 RCU 链表删除节点            |

---

## 10.6 grace period 是什么？

grace period 可以理解成：

> 从 writer 删除旧对象开始，到所有可能还在引用旧对象的 reader 都退出为止，这段时间就是 grace period。

例如：

```text
T0 reader A 进入 rcu_read_lock
T1 writer 删除 old object
T2 writer 调用 synchronize_rcu
T3 reader A 退出 rcu_read_unlock
T4 synchronize_rcu 返回
T5 writer 可以 kfree old object
```

重点：

```text
synchronize_rcu 等待的是在它开始前已经存在的 reader。
```

之后新进来的 reader，要么看到新对象，要么不再能通过全局指针拿到旧对象。

---

## 10.7 synchronize_rcu 和 call_rcu 的区别

### synchronize_rcu()

同步等待。

```c
list_del_rcu(&obj->list);
synchronize_rcu();
kfree(obj);
```

含义：

```text
当前线程阻塞等待 grace period 结束
等结束后继续执行 kfree
```

缺点：

```text
可能睡眠，不能在中断上下文使用
等待时间可能较长
```

---

### call_rcu()

异步回调。

```c
list_del_rcu(&obj->list);
call_rcu(&obj->rcu, obj_free_rcu);
```

含义：

```text
当前线程不等
把回调挂到 RCU callback 队列
grace period 结束后内核调用 obj_free_rcu
```

适合：

```text
不能阻塞
不想同步等待
释放动作可以延后
```

---

## 10.8 RCU 读侧为什么快？

在很多配置下，RCU 读侧近似只是：

```c
preempt_disable();
...
preempt_enable();
```

甚至在某些非抢占场景下几乎没有实际指令。

它不需要：

```text
修改共享读计数
抢 cacheline
做原子操作
阻塞 writer
```

这就是它比 rwlock 更适合极端读多写少场景的原因。

---

## 10.9 RCU 的典型使用场景

非常适合：

```text
路由表
文件描述符表
进程列表
网络协议栈查找表
只读查询多、更新少的数据结构
链表/哈希表中节点删除后的延迟释放
```

不适合：

```text
写特别频繁
读者必须长期持有并睡眠
对象更新无法 copy
需要强一致事务语义
```

---

## 10.10 RCU 最大的坑

### 坑 1：删除不等于释放

```c
list_del_rcu(&p->list);
kfree(p);   // 错
```

错误原因：

```text
reader 可能还在遍历 p
```

应该：

```c
list_del_rcu(&p->list);
synchronize_rcu();
kfree(p);
```

或者：

```c
list_del_rcu(&p->list);
kfree_rcu(p, rcu);
```

---

### 坑 2：读侧不能随便睡眠

普通 RCU 读侧：

```c
rcu_read_lock();
...
rcu_read_unlock();
```

里面不能随便睡眠，尤其在一些 RCU 类型和上下文下。

如果确实要在读侧睡眠，需要考虑 SRCU，也就是 sleepable RCU。

---

### 坑 3：RCU 不保护对象内容的并发修改

RCU 主要保护：

```text
指针替换
对象生命周期
遍历期间对象不被释放
```

但如果多个 writer 同时修改对象内容，还是需要锁。

常见模式是：

```text
reader 用 RCU
writer 用 mutex/spinlock 串行化
```

---

# 11. RCU synchronize 原理分析

wowotech 的 RCU synchronize 文章关注的是 RCU 最核心的问题：writer 如何知道所有旧 reader 都离开了。检索摘要里提到 RCU callback 通常在 synchronize_rcu 中添加，用于等待 GP 结束后唤醒进程，callbacks 处理和软中断有关。([Wowotech][15])

## 11.1 synchronize_rcu 要解决的问题

写侧想做：

```c
remove_pointer(old);
synchronize_rcu();
free(old);
```

问题是：

```text
怎么判断所有 CPU 上的旧 reader 都退出了？
```

---

## 11.2 quiescent state 静止状态

RCU 判断 reader 退出，不一定真的追踪每个 reader。

它更关心 CPU 是否经过了 quiescent state。

可以粗略理解成：

```text
如果某个 CPU 发生过上下文切换、用户态切换、idle 等，
说明它之前的 RCU 读侧临界区应该已经结束。
```

不同 RCU 类型判断方式不同，但直觉是：

```text
只要所有 CPU 都经历了某种“安全点”，旧 reader 就不可能还在。
```

---

## 11.3 grace period 的结束条件

一个 grace period 要结束，需要确认：

```text
所有可能持有旧引用的 CPU / task
都已经离开对应的 RCU read-side critical section
```

之后 callback 才能执行。

---

## 11.4 RCU callback

如果使用：

```c
call_rcu(&obj->rcu, callback);
```

RCU 会把 callback 挂起来。

等 grace period 结束后：

```text
执行 callback
释放对象
```

这避免 writer 同步等待。

---

# 12. RCU 历史实现：2.5.43、2.6.11、2.6.23

wowotech 有几篇 RCU 历史实现文章，包括 Linux 2.5.43 版本的 RCU 实现、Linux 2.6.11 classic RCU、Linux 2.6.23 sleepable RCU。检索结果显示这些文章分别围绕 RCU callback 数据结构、classic RCU 实现、sleepable RCU / synchronize_rcu 与 synchronize_sched 的拆分等展开。([Wowotech][16])

## 12.1 为什么要看旧版本 RCU？

因为现代 RCU 很复杂：

```text
Tree RCU
Tiny RCU
SRCU
Tasks RCU
Preemptible RCU
RCU nocb
RCU callback offloading
```

直接看现代内核会被数据结构淹没。

旧版本更适合理解：

```text
RCU 的最小核心逻辑是什么
callback 怎么挂
grace period 怎么推进
CPU 状态怎么记录
```

---

## 12.2 2.5.43 RCU 的学习价值

早期 RCU 的重点：

```text
1. 每个 CPU 有自己的 RCU 数据
2. callback 被挂到队列
3. grace period 结束后处理 callback
4. 通过 CPU 状态推进 RCU
```

它虽然废弃，但适合理解 RCU 的骨架。

---

## 12.3 2.6.11 classic RCU

classic RCU 更接近正式产品级实现。

学习重点：

```text
1. 每 CPU rcu_data
2. 全局 rcu_ctrlblk
3. grace period 编号
4. callback batch
5. CPU quiescent state 上报
```

你可以把它理解成：

```text
全局控制器负责推进 GP
每 CPU 负责记录本 CPU 状态和 callback
```

---

## 12.4 2.6.23 sleepable RCU

sleepable RCU / SRCU 解决的是：

```text
普通 RCU 读侧不能随便睡眠
但有些读侧确实需要睡眠
```

所以 SRCU 允许读侧睡眠，但代价是：

```text
读侧开销更大
需要显式 srcu_struct
grace period 检测更复杂
```

普通 RCU：

```c
rcu_read_lock();
...
rcu_read_unlock();
```

SRCU：

```c
idx = srcu_read_lock(&ss);
...
srcu_read_unlock(&ss, idx);
```

---

# 13. RCU 使用方法

wowotech 的 RCU（1）概述、RCU（2）使用方法，是更偏“怎么用”的文章。检索结果确认这两篇在 2019 年发布，分别讲 RCU 概述和使用方法。([Wowotech][17])

## 13.1 RCU 链表添加

```c
struct foo {
    int key;
    struct list_head list;
    struct rcu_head rcu;
};
```

添加节点：

```c
spin_lock(&foo_lock);
list_add_rcu(&new->list, &foo_list);
spin_unlock(&foo_lock);
```

为什么 writer 还要锁？

因为 RCU 不保证多个 writer 之间互斥。

RCU 保护 reader，writer 之间仍然要串行化。

---

## 13.2 RCU 链表遍历

```c
rcu_read_lock();

list_for_each_entry_rcu(p, &foo_list, list) {
    if (p->key == key) {
        do_something(p);
        break;
    }
}

rcu_read_unlock();
```

这里 reader 不拿传统锁。

但它依赖两个条件：

```text
1. 节点不会在读侧临界区内被释放
2. 指针访问有正确的 RCU 内存序语义
```

---

## 13.3 RCU 删除

```c
spin_lock(&foo_lock);
list_del_rcu(&p->list);
spin_unlock(&foo_lock);

synchronize_rcu();

kfree(p);
```

或者：

```c
spin_lock(&foo_lock);
list_del_rcu(&p->list);
spin_unlock(&foo_lock);

kfree_rcu(p, rcu);
```

---

## 13.4 RCU 替换指针

```c
old = rcu_dereference_protected(global, lockdep_is_held(&lock));

new = kmalloc(sizeof(*new), GFP_KERNEL);
*new = *old;
new->value = value;

rcu_assign_pointer(global, new);

synchronize_rcu();
kfree(old);
```

# 14. perfbook RCU 翻译

wowotech 的 perfbook RCU 翻译上、下篇，是对 RCU 理论部分的补充。检索结果提到 RCU 允许多个 reader thread 和一个 updater thread 并发，从而提升 scalability；下篇涉及 synchronize_rcu 等。([Wowotech][18])

你看这两篇时重点抓：

## 14.1 RCU 的三个核心能力

```text
1. publish-subscribe
2. wait for pre-existing readers
3. defer reclamation
```

也就是：

```text
发布新对象
等待旧读者
延迟释放旧对象
```

---

## 14.2 RCU 和引用计数的区别

引用计数：

```text
每个 reader 增加 refcnt
用完减少 refcnt
writer 等 refcnt 归零
```

问题：

```text
每次读都要 atomic inc/dec
读路径有 cacheline 竞争
```

RCU：

```text
reader 不改对象 refcnt
只进入 RCU 读侧临界区
writer 等 grace period
```

优点：

```text
读路径极快
```

缺点：

```text
释放延迟
写侧复杂
对象生命周期理解更难
```

---

## 14.3 RCU 和 seqlock 的区别

seqlock：

```text
reader 可能重试
writer 可以让 reader 读失败
适合小数据
```

RCU：

```text
reader 不重试
reader 看到旧版本或新版本
适合指针型数据结构
```

# 15. mutex：睡眠互斥锁

wowotech mutex 文章说明，Linux 内核中的 mutex 是一种保证串行化的睡眠锁机制，语义类似 spinlock，但拿不到锁时会阻塞；文章还提到 Linux mutex 有乐观自旋优化，即持锁失败时可能先在 mutex 状态上自旋，等待 owner 释放，也可能进入睡眠。([Wowotech][19])

## 15.1 mutex 解决什么问题？

和 spinlock 一样，mutex 也是互斥：

```text
同一时间只允许一个 task 进入临界区
```

不同点：

```text
spinlock 拿不到就自旋
mutex 拿不到就睡眠
```

---

## 15.2 mutex 的基本 API

```c
DEFINE_MUTEX(lock);

mutex_lock(&lock);
/* critical section */
mutex_unlock(&lock);
```

尝试获取：

```c
if (mutex_trylock(&lock)) {
    ...
    mutex_unlock(&lock);
}
```

可中断等待：

```c
ret = mutex_lock_interruptible(&lock);
if (ret)
    return ret;
```

---

## 15.3 mutex 不能用于中断上下文

因为 mutex 可能睡眠。

中断上下文不能睡眠，所以不能：

```c
irq_handler()
{
    mutex_lock(&lock);  // 错
}
```

---

## 15.4 mutex 适合什么场景？

适合：

```text
进程上下文
临界区可能较长
临界区可能睡眠
不要求中断上下文使用
```

例如：

```text
驱动 open/release
ioctl 配置路径
设备状态修改
文件系统较长路径
```

---

## 15.5 mutex 的 owner

mutex 通常记录 owner：

```text
谁持有锁
```

这样可以支持：

```text
debug
死锁检测
乐观自旋判断 owner 是否正在运行
priority inheritance 相关场景
```

---

## 15.6 mutex 乐观自旋

现代内核 mutex 不一定一拿不到就睡。

它可能判断：

```text
owner 正在 CPU 上运行
owner 很可能马上释放锁
```

于是当前 task 先自旋一小会儿。

这样避免：

```text
睡眠 -> 调度出去 -> 被唤醒 -> 再调度回来
```

的上下文切换开销。

但如果 owner 不在运行，或者等待太久，就睡眠。

---

# 16. Queued spinlock：排队自旋锁

wowotech queued spinlock 文章说明，spinlock 是 Linux 内核常用互斥锁，和 mutex 不同，拿不到锁时不会阻塞，而是自旋；queued spinlock 是在普通 spinlock 基础上解决高并发竞争问题。([Wowotech][20])

## 16.1 普通 spinlock 高竞争有什么问题？

假设 100 个 CPU 同时抢一把锁：

```text
CPU0 持锁
CPU1~CPU99 都在读同一个 lock cacheline
CPU0 解锁时，所有 CPU 同时争抢
```

问题：

```text
cacheline 在 CPU 间疯狂 bouncing
总线/cache 一致性压力大
公平性差
扩展性差
```

---

## 16.2 queued spinlock 的核心思想

不要让所有 CPU 都盯着同一个锁变量抢。

而是：

```text
等待者排队
每个 CPU 尽量只等自己的前驱
锁释放时按队列传递
```

这类似 MCS lock 的思想。

---

## 16.3 MCS lock 直觉

每个等待者有自己的节点：

```c
struct mcs_node {
    struct mcs_node *next;
    int locked;
};
```

等待时：

```text
CPU1 排在 CPU0 后面
CPU2 排在 CPU1 后面
CPU3 排在 CPU2 后面
```

CPU2 不需要一直读全局锁变量，它只需要等 CPU1 通知自己。

这样减少 cacheline 争用。

---

## 16.4 qspinlock 的状态

Linux qspinlock 通常把多个状态压缩到一个 32-bit 字段中，例如：

```text
locked
pending
tail
```

你之前分析 vmcore 里看到的：

```text
locked_pending
tail
```

就和 qspinlock 高竞争路径有关。

直觉：

```text
locked 表示锁当前被持有
pending 表示有人准备接锁
tail 表示等待队列尾部
```

---

## 16.5 queued spinlock 适合解决什么？

它解决的是：

```text
多核系统上 spinlock 高竞争时扩展性差
```

尤其在大 CPU 数机器上，如果仍使用简单 test-and-set spinlock，性能会非常差。

---

# 17. futex：用户态锁和内核等待队列

wowotech futex 文章说明，futex 是 Fast Userspace Mutex 的缩写，核心涉及用户空间的 futex word 和内核空间等待队列；Linux man page 也说明 futex wait/wake 通常围绕一个用户态 futex word，释放锁时先重置锁状态，再执行 futex wake 唤醒等待者。([Wowotech][21])

## 17.1 futex 解决什么问题？

用户态 mutex 如果每次 lock/unlock 都进内核，太慢。

所以 futex 的设计目标是：

```text
无竞争：完全用户态完成
有竞争：才进入内核睡眠/唤醒
```

---

## 17.2 futex 的核心结构

用户态有一个整数：

```c
int futex_word;
```

它可能表示：

```text
0 = unlocked
1 = locked no waiter
2 = locked with waiter
```

用户态先用 atomic CAS 抢锁。

成功：

```text
不用 syscall
```

失败：

```text
调用 futex syscall 进入内核睡眠
```

---

## 17.3 futex lock 伪代码

```c
if (atomic_cmpxchg(&futex_word, 0, 1) == 0)
    return;  // fast path

while (1) {
    old = atomic_xchg(&futex_word, 2);
    if (old == 0)
        return;

    futex_wait(&futex_word, 2);
}
```

---

## 17.4 futex unlock 伪代码

```c
old = atomic_xchg(&futex_word, 0);

if (old == 2)
    futex_wake(&futex_word, 1);
```

意思：

```text
如果没人等，直接用户态解锁
如果有人等，进入内核唤醒一个等待者
```

---

## 17.5 futex 为什么快？

因为绝大多数锁是无竞争的。

无竞争时：

```text
pthread_mutex_lock
pthread_mutex_unlock
```

只需要用户态原子操作，不需要 syscall。

只有竞争时才进入内核。

这就是 futex 的名字：

```text
Fast Userspace Mutex
```

---

## 17.6 futex 和 mutex 的关系

内核 mutex：

```text
内核内部使用
保护内核数据
```

futex：

```text
用户态锁的底层支持
内核只负责 wait/wake
锁状态主要在用户态维护
```

pthread mutex 通常基于 futex 实现，但 futex 本身不是完整 mutex，它只是一个机制。

---

# 18. rwsem：读写信号量 / 睡眠读写锁

wowotech rwsem 文章“Linux 读写锁逻辑解析”说明，除了 mutex，Linux 内核还有常用的睡眠锁 rwsem；它也有类似 mutex 的乐观自旋，需要先获取 osq 锁，才能进入 rwsem 乐观自旋，否则自旋在 per-cpu MCS lock 上。([Wowotech][22])

## 18.1 rwsem 解决什么问题？

mutex 是完全互斥：

```text
reader 和 reader 也互斥
```

rwsem 允许：

```text
多个 reader 并发
writer 独占
等待时可以睡眠
```

所以 rwsem 是：

```text
睡眠版读写锁
```

---

## 18.2 rwsem 和 rw spinlock 的区别

| 项目       | rw spinlock | rwsem                      |
| -------- | ----------- | -------------------------- |
| 等待方式     | 自旋          | 睡眠                         |
| 可用于中断上下文 | 通常可以按变体使用   | 不可以                        |
| 临界区长度    | 短           | 可以较长                       |
| 是否可睡眠    | 不可以         | 可以                         |
| 典型场景     | 短小读写保护      | mmap_sem / inode sem 等较长路径 |

---

## 18.3 rwsem API

读侧：

```c
down_read(&sem);
/* read critical section */
up_read(&sem);
```

写侧：

```c
down_write(&sem);
/* write critical section */
up_write(&sem);
```

---

## 18.4 rwsem 的状态

rwsem 内部通常要表达：

```text
当前 reader 数量
是否有 writer
是否有 waiters
owner
等待队列
乐观自旋队列
```

所以它比普通 mutex 更复杂。

---

## 18.5 rwsem 乐观自旋

如果 writer 持有 rwsem，但 owner 正在 CPU 上运行，等待者可能先自旋一会儿。

如果很快释放，就避免睡眠唤醒开销。

这和 mutex 的 optimistic spinning 很像。

---

## 18.6 rwsem 的适用场景

适合：

```text
读多写少
读侧临界区可能睡眠
临界区较长
需要 writer 独占修改
```

不适合：

```text
中断上下文
极短临界区
读侧极致性能场景
```

极致读性能通常考虑 RCU。

---

# 19. Linux 中常见同步机制设计原理

wowotech 的“Linux 中常见同步机制设计原理”是从设计层面比较各种同步机制，包括自旋锁、信号量等。([Wowotech][23])

这篇的学习重点不是某个 API，而是：

```text
为什么不同同步机制会这样设计
```

## 19.1 同步机制设计要考虑什么？

主要考虑：

```text
1. 等待时是自旋还是睡眠
2. 临界区能不能睡眠
3. 是否区分 reader/writer
4. 是否需要公平性
5. 是否需要优先级继承
6. 是否用于中断上下文
7. 是否有 cacheline 扩展性问题
8. 是否允许 reader 和 writer 并发
9. 是否保护对象生命周期
```

---

## 19.2 自旋锁设计权衡

自旋锁优点：

```text
简单
快速
不能睡眠上下文可用
临界区短时效率高
```

缺点：

```text
等待浪费 CPU
高竞争时 cacheline 抖动
临界区不能睡眠
```

---

## 19.3 信号量 semaphore

信号量和 mutex 类似，都是睡眠等待。

但 semaphore 更通用：

```text
count = 1 时类似 mutex
count > 1 时允许多个执行流同时进入
```

例如：

```text
最多允许 N 个用户访问某资源
```

但现代内核里，如果只是互斥，通常更推荐 mutex，因为语义更明确、debug 更好。

---

## 19.4 mutex 设计权衡

mutex 适合：

```text
互斥
睡眠
owner 明确
debug 友好
```

不适合：

```text
中断上下文
持锁者可能在不能睡眠上下文
```

---

## 19.5 RCU 设计权衡

RCU 把成本转移到 writer：

```text
reader 极快
writer 复杂
对象释放延迟
内存占用可能增加
```

它是典型的：

```text
空间换时间
写复杂换读性能
```

---

# 20. 编译乱序 Compiler Reordering

这篇单独拿出来很有价值，因为很多人学锁时只关注 CPU，不关注编译器。wowotech 这篇文章强调，编译器的工作就是优化代码以提高性能。([Wowotech][8])

## 20.1 编译器为什么会乱序？

因为单线程语义不变即可。

例如：

```c
a = 1;
b = 2;
```

如果编译器认为换成：

```c
b = 2;
a = 1;
```

不会影响当前线程结果，它就可能这么做。

但在并发里，其他 CPU 可能观察顺序。

---

## 20.2 编译器可能做哪些优化？

```text
1. 重排访问顺序
2. 合并多次读取
3. 消除看似无用的读取
4. 把变量缓存到寄存器
5. 拆分访问
```

例如：

```c
while (!flag)
    ;
```

编译器可能认为 `flag` 没被当前线程修改，于是优化成：

```c
if (!flag)
    while (1)
        ;
```

所以并发变量要用：

```c
READ_ONCE(flag)
```

---

## 20.3 barrier() 的作用

```c
barrier();
```

告诉编译器：

```text
不要把 barrier 前后的内存访问互相重排
```

但它不一定生成 CPU 屏障指令。

所以：

```text
barrier() 主要管编译器
smp_mb() 管 CPU/内存系统
```

---

# 21. Why Memory Barriers / perfbook memory barrier

这几篇翻译文章是为了让你真正理解：

```text
为什么多核下“我写了”不等于“别人按顺序看到了”
```

## 21.1 store buffer

CPU 写变量时，可能不是立刻写入共享 cache，而是先放到 store buffer。

所以 CPU0：

```c
x = 1;
```

对 CPU0 自己来说已经完成。

但 CPU1 可能还看不到。

---

## 21.2 invalidate queue

当 CPU0 要写某个 cacheline，可能需要让其他 CPU 的副本失效。

但其他 CPU 处理 invalidate 消息也可能有队列。

所以失效不是瞬间完成。

这就可能导致不同 CPU 看到的内存顺序不同。

---

## 21.3 cache coherence 不等于 sequential consistency

cache coherence 保证的是：

```text
对同一个地址的写，所有 CPU 最终看到一致顺序
```

但它不保证：

```text
对不同地址 x 和 y 的写，所有 CPU 都按同一顺序看到
```

所以：

```c
x = 1;
y = 1;
```

CPU1 可能先看到 y，再看到 x。

---

## 21.4 memory barrier 的本质

memory barrier 的本质是：

```text
强制某些内存访问在某个点之前完成或可见
再允许后续访问继续
```

但不同 barrier 强度不同。

---

# 22. 各机制之间的核心对比

## 22.1 atomic vs spinlock

| 对比      | atomic               | spinlock       |
| ------- | -------------------- | -------------- |
| 保护范围    | 单变量简单操作              | 任意临界区          |
| 是否能保护链表 | 不适合                  | 适合             |
| 是否睡眠    | 不睡眠                  | 不睡眠            |
| 典型用途    | refcnt、flags、counter | list、queue、状态机 |

口诀：

```text
一个变量一次改，用 atomic。
多个字段一起改，用锁。
```

---

## 22.2 spinlock vs mutex

| 对比     | spinlock | mutex |
| ------ | -------- | ----- |
| 等待方式   | 自旋       | 睡眠    |
| 中断上下文  | 可以按变体使用  | 不可以   |
| 临界区    | 必须短      | 可以较长  |
| 持锁期间睡眠 | 不允许      | 可以    |
| 等待开销   | 浪费 CPU   | 上下文切换 |

口诀：

```text
不能睡眠，用 spinlock。
可以睡眠，优先 mutex。
```

---

## 22.3 rwlock vs rwsem

| 对比        | rwlock | rwsem |
| --------- | ------ | ----- |
| 等待方式      | 自旋     | 睡眠    |
| reader 并发 | 可以     | 可以    |
| writer 独占 | 是      | 是     |
| 中断上下文     | 可以按变体  | 不可以   |
| 临界区长度     | 短      | 可长    |

口诀：

```text
短临界区读写锁：rwlock。
长临界区读写锁：rwsem。
```

---

## 22.4 rwlock vs seqlock vs RCU

| 机制      | reader 行为 | writer 行为 | 适合数据     |
| ------- | --------- | --------- | -------- |
| rwlock  | 拿读锁       | 拿写锁       | 普通读写共享数据 |
| seqlock | 不加锁，但可能重试 | 独占更新 seq  | 小数据、一致快照 |
| RCU     | 不加传统锁     | 替换指针，延迟释放 | 指针型数据结构  |

口诀：

```text
读者要稳定进入，用 rwlock/rwsem。
读者可重试，用 seqlock。
读者要极快且对象可延迟释放，用 RCU。
```

---

## 22.5 RCU vs 引用计数

| 对比   | RCU            | 引用计数           |
| ---- | -------------- | -------------- |
| 读侧开销 | 极低             | atomic inc/dec |
| 释放时机 | grace period 后 | refcnt 归零      |
| 适合   | 读多写少           | 对象被长期持有        |
| 缺点   | 释放延迟           | cacheline 竞争   |

---

## 22.6 seqlock vs RCU

| 对比            | seqlock | RCU        |
| ------------- | ------- | ---------- |
| reader 是否重试   | 会       | 通常不会       |
| writer 是否复制对象 | 不一定     | 通常复制/替换    |
| 是否保护指针生命周期    | 不适合     | 适合         |
| 适合            | 时间、小结构体 | 链表、哈希表、路由表 |

---

# 23. 你应该形成的选择框架

以后看到一个共享数据，按这个顺序问：

## 23.1 能不能避免共享？

能的话，用：

```text
Per-CPU
局部变量
分片锁
```

这是最佳方案。

---

## 23.2 是不是简单计数？

是的话，用：

```text
atomic_t
refcount_t
percpu counter
```

---

## 23.3 临界区能不能睡眠？

不能睡眠：

```text
spinlock
raw_spinlock
rwlock
seqlock
RCU
```

可以睡眠：

```text
mutex
semaphore
rwsem
SRCU
```

---

## 23.4 是不是读多写少？

读多写少：

```text
优先考虑 RCU
如果读者可重试，考虑 seqlock
如果读侧可能睡眠，考虑 rwsem/SRCU
```

---

## 23.5 是否涉及对象释放？

涉及对象释放，尤其是指针、链表、哈希表：

```text
优先考虑 RCU 或引用计数
不要只靠 seqlock
```

---

## 23.6 是否在中断上下文？

中断上下文：

```text
不能用 mutex/rwsem
一般用 spinlock_irqsave
或者无锁/atomic
```

# 25. 结合你做调度/内核分析时最相关的点

你后面看调度器、rq lock、EEVDF、CFS、load_balance，会大量遇到同步问题。

## 25.1 rq lock 为什么是 raw spinlock？

调度器 runqueue 是非常核心的数据结构。

它可能在：

```text
调度路径
唤醒路径
迁移路径
负载均衡路径
tick 路径
stopper 线程
```

中被访问。

这些路径要求：

```text
不能睡眠
延迟极低
必须严格保护 rq 内部一致性
```

所以不能用 mutex，只能用 raw spinlock。

---

## 25.2 你之前看到的 qspinlock slowpath

你之前分析过：

```text
__pv_queued_spin_lock_slowpath
_raw_spin_lock
raw_spin_rq_lock_nested
load_balance
newidle_balance
pick_next_task_fair
```

这正是 queued spinlock 的真实应用。

当 rq lock 高竞争时，就会进入 queued spinlock slowpath。

你要从 qspinlock 文章里重点学：

```text
locked / pending / tail
MCS queue
PV qspinlock
为什么不是所有 CPU 直接抢同一个变量
```

## 25.3 RCU 在调度器和进程管理里也很多

比如：

```text
task list
pid
cgroup
sched_domain
cpuset
net namespace
```

很多读多写少的数据结构都用 RCU。

你后面看到：

```c
rcu_read_lock();
task = find_task_by_vpid(pid);
...
rcu_read_unlock();
```

要立刻意识到：

```text
这里保护的是 task 指针查找期间不被释放
不是保护 task 内部所有字段不变
```

# 26. 最终压缩版：一口气记住这组文档

这 26 篇可以压缩成下面这张知识图：

```text
Linux 内核同步机制
├── 基础问题
│   ├── 竞态：多个执行流同时访问共享数据
│   ├── 原子性：读-改-写不能被打断
│   ├── 可见性：一个 CPU 写了，另一个 CPU 何时看见
│   └── 顺序性：代码顺序不等于执行/可见顺序
│
├── 基础工具
│   ├── atomic：保护单变量原子操作
│   ├── Per-CPU：每 CPU 一份，减少共享
│   ├── READ_ONCE/WRITE_ONCE：约束编译器访问
│   └── memory barrier：约束 CPU/内存访问顺序
│
├── 忙等锁
│   ├── spinlock：短临界区，不睡眠
│   ├── rw spinlock：读者并发，写者独占
│   ├── seqlock：读者无锁重试，写者优先
│   └── queued spinlock：高竞争下排队自旋
│
├── 睡眠锁
│   ├── mutex：普通互斥睡眠锁
│   ├── semaphore：计数型睡眠同步
│   └── rwsem：睡眠版读写锁
│
├── RCU
│   ├── 读者极快
│   ├── 写者复制/替换
│   ├── grace period 后释放旧对象
│   ├── synchronize_rcu 同步等待
│   ├── call_rcu 异步回调
│   └── SRCU 支持 sleepable 读侧
│
└── 用户态同步
    └── futex：无竞争用户态完成，有竞争进入内核
```

最重要的选择口诀：

```text
能不共享，就 Per-CPU。
只是计数，用 atomic。
不能睡眠，用 spinlock。
可以睡眠，用 mutex/rwsem。
读多写少，用 RCU。
读者可重试，用 seqlock。
用户态锁优化，用 futex。
高竞争自旋锁，看 queued spinlock。
涉及顺序可见性，必须考虑 memory barrier。
```

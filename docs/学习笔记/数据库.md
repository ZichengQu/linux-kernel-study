# 一、SQL 语法核心知识点

## 1. SQL 最核心的是增删改查

你需要优先掌握：

```sql
INSERT INTO table(col1, col2) VALUES(val1, val2);

UPDATE table
SET col = val
WHERE id = 1;

DELETE FROM table
WHERE id = 1;

SELECT col1, col2
FROM table
WHERE condition;
```

其中最重要的安全点是：

> `UPDATE` 和 `DELETE` 一定要带 `WHERE`，否则可能更新或删除整张表。文档也明确提醒，更新和删除前可以先用 `SELECT` 测试过滤条件。([GitHub][1])

---

## 2. SELECT 的基本结构

建议记成这个顺序：

```sql
SELECT 查询字段
FROM 表
WHERE 行过滤
GROUP BY 分组字段
HAVING 分组后过滤
ORDER BY 排序字段
LIMIT 返回行数;
```

你之前容易卡的是 `WHERE`、`GROUP BY`、`HAVING`。

核心区别是：

```text
WHERE：
    分组前过滤原始行。

GROUP BY：
    把相同值的行放到一组。

HAVING：
    分组统计之后，再过滤分组结果。
```

比如：

```sql
SELECT city, COUNT(*) AS num
FROM users
WHERE age > 18
GROUP BY city
HAVING num >= 10;
```

意思是：

```text
先找 age > 18 的用户；
再按 city 分组；
再只保留人数 >= 10 的城市。
```

文档中也强调：`WHERE` 过滤行，`HAVING` 过滤分组，行过滤应该先于分组过滤。([GitHub][1])

## 3. DISTINCT 和 GROUP BY 的区别

你之前问过：

```sql
SELECT col FROM mytable GROUP BY col;
```

和：

```sql
SELECT DISTINCT col FROM mytable;
```

在只查一个字段时，结果通常都是：

```text
返回 col 的不重复值。
```

但语义不同：

```text
DISTINCT：
    目的就是去重。

GROUP BY：
    目的是分组，通常配合 COUNT / SUM / AVG / MAX / MIN。
```

比如：

```sql
SELECT col, COUNT(*) AS num
FROM mytable
GROUP BY col;
```

这就不是简单去重，而是统计每组数量。

记忆：

```text
只想去重：
    用 DISTINCT。

想分组统计：
    用 GROUP BY。
```

## 4. 子查询和相关子查询

你之前问过这个 SQL：

```sql
SELECT cust_name,
       (SELECT COUNT(*)
        FROM Orders
        WHERE Orders.cust_id = Customers.cust_id) AS orders_num
FROM Customers
ORDER BY cust_name;
```

核心理解：

> 外层每拿到一个客户，内层子查询就根据这个客户的 `cust_id` 去订单表里数订单数量。

例如：

`Customers`：

| cust_id | cust_name |
| ------- | --------- |
| 1       | 张三        |
| 2       | 李四        |
| 3       | 王五        |

`Orders`：

| order_id | cust_id |
| -------- | ------- |
| 101      | 1       |
| 102      | 1       |
| 103      | 2       |

执行时相当于：

```text
处理张三时：
    SELECT COUNT(*) FROM Orders WHERE cust_id = 1;
    结果 2

处理李四时：
    SELECT COUNT(*) FROM Orders WHERE cust_id = 2;
    结果 1

处理王五时：
    SELECT COUNT(*) FROM Orders WHERE cust_id = 3;
    结果 0
```

这叫**相关子查询**，因为内层查询依赖外层当前行。文档中也说明，这类子查询会对第一个查询检索出的每个客户执行一次。([GitHub][1])

## 5. JOIN 连接

JOIN 用来把多张表按关系拼起来。文档中说，连接用于连接多个表，条件通常写在 `ON` 后面，而且连接可以替代子查询，效率一般更好。([GitHub][1])

常见类型：

```text
INNER JOIN：
    只保留两边都匹配的行。

LEFT JOIN：
    左表全部保留，右表没匹配则为 NULL。

RIGHT JOIN：
    右表全部保留，左表没匹配则为 NULL。
```

例如客户和订单：

```sql
SELECT c.cust_name, o.order_id
FROM Customers c
LEFT JOIN Orders o
ON c.cust_id = o.cust_id;
```

如果某个客户没有订单，`LEFT JOIN` 仍然会显示这个客户，只是订单字段为 `NULL`。

## 6. 函数部分重点

你之前问过 `TRIM()`、`CONCAT()`、`LEFT()`、`RIGHT()`、`LOWER()`、`UPPER()`、`SOUNDEX()`。

核心记忆：

| 函数              | 作用        |
| --------------- | --------- |
| `CONCAT()`      | 拼接字符串     |
| `TRIM()`        | 去掉首尾空格    |
| `LTRIM()`       | 去掉左边空格    |
| `RTRIM()`       | 去掉右边空格    |
| `LEFT(col, n)`  | 取左边 n 个字符 |
| `RIGHT(col, n)` | 取右边 n 个字符 |
| `LOWER()`       | 转小写       |
| `UPPER()`       | 转大写       |
| `LENGTH()`      | 求长度       |
| `SOUNDEX()`     | 转换成语音相似编码 |

比如：

```sql
SELECT CONCAT(TRIM(vend_name), ' (', TRIM(vend_country), ')') AS vend_title
FROM vendors;
```

意思是：

```text
去掉供应商名称和国家两边空格；
再拼成：
ACME (USA)
```

`SOUNDEX()` 不是精确匹配，而是用于发音相近的匹配。文档中也说，它会把字符串转换为描述语音表示的字母数字模式。([GitHub][1])

# 二、数据库系统原理核心知识点

## 1. 事务 ACID

事务就是一组操作，要么全部成功，要么全部失败。文档中定义事务是满足 ACID 特性的一组操作，可以通过 `COMMIT` 提交，也可以通过 `ROLLBACK` 回滚。([GitHub][2])

ACID：

| 特性              | 含义          | 例子            |
| --------------- | ----------- | ------------- |
| 原子性 Atomicity   | 要么全成功，要么全失败 | 转账不能只扣钱不加钱    |
| 一致性 Consistency | 事务前后数据合法    | 总金额不能凭空变化     |
| 隔离性 Isolation   | 并发事务互不干扰    | 别人未提交的数据我不能乱读 |
| 持久性 Durability  | 提交后数据不丢     | 宕机恢复后数据还在     |

你需要重点理解：

```text
原子性：
    靠 Undo Log 支持回滚。

持久性：
    靠 Redo Log 支持崩溃恢复。

隔离性：
    靠锁和 MVCC 实现。
```

---

## 2. 并发一致性问题

重点四个：

| 问题    | 通俗解释                |
| ----- | ------------------- |
| 丢失修改  | 两个事务都改同一数据，后一个覆盖前一个 |
| 脏读    | 读到了别人还没提交的数据        |
| 不可重复读 | 同一事务内，两次读同一行，值变了    |
| 幻读    | 同一事务内，两次查同一范围，行数变了  |

你之前问过不可重复读和幻读，最重要的是这句：

```text
不可重复读：
    重点是同一行数据变了。

幻读：
    重点是同一范围内多了/少了行。
```

例子：

```text
不可重复读：
    第一次查 id=1，balance=100；
    别人把 id=1 改成 200 并提交；
    第二次查 id=1，balance=200。

幻读：
    第一次查 age > 18，有 3 行；
    别人插入一条 age=20 的记录并提交；
    第二次查 age > 18，变成 4 行。
```

---

## 3. 隔离级别

四个隔离级别：

| 隔离级别             | 能解决什么      | 可能还有什么问题    |
| ---------------- | ---------- | ----------- |
| READ UNCOMMITTED | 基本不解决      | 脏读、不可重复读、幻读 |
| READ COMMITTED   | 解决脏读       | 不可重复读、幻读    |
| REPEATABLE READ  | 解决脏读、不可重复读 | 理论上可能幻读     |
| SERIALIZABLE     | 基本都解决      | 并发性能最低      |

MySQL InnoDB 默认是 `REPEATABLE READ`。

你之前最关键的疑问是：

> 普通 `SELECT` 不加锁，为什么还能可重复读？

答案是：

```text
InnoDB 默认 REPEATABLE READ 下，普通 SELECT 是快照读。
它不是靠读锁阻止别人修改，
而是靠 MVCC + Read View + Undo Log 读旧版本。
```

---

## 4. MVCC、快照读、当前读

文档中说，MVCC 是 InnoDB 实现提交读和可重复读的一种方式；它利用多版本思想，写操作更新最新版本，读操作读旧版本快照，从而减少读写互斥。([GitHub][2])

你要这样记：

```text
MVCC = 多版本并发控制。

核心思想：
    写生成新版本；
    普通读读取符合自己 Read View 的旧版本；
    所以读不阻塞写，写不阻塞普通读。
```

### 快照读

```sql
SELECT * FROM account WHERE id = 1;
```

普通 `SELECT` 通常是快照读。

特点：

```text
不加行锁；
读的是某个时间点的一致性版本；
在 REPEATABLE READ 下，同一事务内复用第一次快照读的 Read View。
```

### 当前读

```sql
SELECT * FROM account WHERE id = 1 FOR UPDATE;

UPDATE account SET balance = 200 WHERE id = 1;

DELETE FROM account WHERE id = 1;
```

当前读特点：

```text
读最新版本；
需要加锁；
用于修改或即将修改的数据。
```

文档中也明确区分：MVCC 的普通 `SELECT` 是快照中的数据，不需要加锁；`INSERT`、`UPDATE`、`DELETE` 这类修改操作需要加锁并读取最新数据；`SELECT ... LOCK IN SHARE MODE` 和 `SELECT ... FOR UPDATE` 可以强制加锁读。([GitHub][2])

---

## 5. Read View 怎么理解

你可以把 Read View 理解成：

> 当前事务第一次快照读时，拍下来的“活跃事务名单”。

判断版本是否可见，大概是：

```text
如果这个版本是我自己改的：
    可见。

如果修改这个版本的事务，在我创建 Read View 前已经提交：
    可见。

如果修改这个版本的事务，在我创建 Read View 时还没提交：
    不可见，沿 undo log 找旧版本。

如果修改这个版本的事务，是我创建 Read View 后才出现的：
    不可见，沿 undo log 找旧版本。
```

所以：

```text
READ COMMITTED：
    每次 SELECT 都创建新的 Read View。
    所以可能不可重复读。

REPEATABLE READ：
    一个事务内第一次普通 SELECT 创建 Read View，后续复用。
    所以普通 SELECT 可以可重复读。
```

## 6. 锁：S 锁、X 锁、行锁、表锁、间隙锁

重点记这些：

```text
S 锁，共享锁：
    我读，你也可以读，但不能改。

X 锁，排他锁：
    我改，你不能读加锁读，也不能改。

行锁：
    锁某些行，并发高。

表锁：
    锁整张表，并发低。

意向锁：
    用来协调表锁和行锁。
```

你之前问过显式锁：

```sql
SELECT * FROM account WHERE id = 1 FOR SHARE;
```

意思是加共享锁：

```text
别人可以普通读；
但不能修改这行。
```

```sql
SELECT * FROM account WHERE id = 1 FOR UPDATE;
```

意思是加排他锁：

```text
我后面要改这行；
别人不能修改，也不能拿冲突锁。
```

## 7. Next-Key Lock

文档中说，MVCC 不能解决幻读问题，InnoDB 在 `REPEATABLE READ` 下通过 `MVCC + Next-Key Locks` 解决幻读；Next-Key Lock 是 Record Lock 和 Gap Lock 的结合。([GitHub][2])

你可以这样理解：

```text
Record Lock：
    锁住已有记录。

Gap Lock：
    锁住两个索引记录之间的间隙。

Next-Key Lock：
    锁住记录 + 前面的间隙。
```

例子：

```sql
SELECT c
FROM t
WHERE c BETWEEN 10 AND 20
FOR UPDATE;
```

它不只是锁住已有的 `10`、`20`，还要锁住中间间隙，防止别人插入 `15`。

所以：

```text
普通 SELECT 防不可重复读：
    靠 MVCC 快照。

当前读 / UPDATE / DELETE 防幻读：
    靠 Next-Key Lock。
```

## 8. 范式和数据库设计

范式的目的不是为了背定义，而是为了减少：

```text
数据冗余；
更新异常；
删除异常；
插入异常。
```

核心记忆：

```text
1NF：
    字段不可再分。

2NF：
    非主属性必须完全依赖主键，不能只依赖联合主键的一部分。

3NF：
    非主属性不能传递依赖主键。
```

实际开发中不是范式越高越好，因为过度拆表会导致查询复杂、JOIN 增多。一般业务系统常以 3NF 为基础，再根据性能做适当反范式。

## 2. B+Tree 为什么适合数据库索引

文档中说，B+Tree 基于 B Tree 和叶子节点顺序访问指针实现，具有平衡性，并通过顺序访问指针提高区间查询性能；数据库和文件系统普遍使用 B+Tree，是因为它树高更低、磁盘 I/O 更友好。([GitHub][3])

你要这样理解：

```text
红黑树：
    二叉树，每个节点最多 2 个孩子。
    数据多时树高较高。

B+Tree：
    多叉树，一个节点可以有很多 key 和很多孩子。
    树更矮。
    一次磁盘 I/O 可以读一个页，一个页里放很多 key。
```

所以：

```text
B+Tree 适合磁盘：
    因为树矮，I/O 次数少。

B+Tree 适合范围查询：
    因为叶子节点有序并且通过链表相连。
```

## 3. InnoDB 主键索引和辅助索引

你之前重点问过这个。

InnoDB 里：

```text
主键索引 / 聚簇索引：
    叶子节点存完整数据行。

辅助索引 / 二级索引：
    叶子节点存索引列 + 主键值。
```

文档中也说明，InnoDB 的主索引叶子节点记录完整数据记录，辅助索引叶子节点记录主键值，因此用辅助索引查找时，要先找到主键值，再到主索引中查完整行。([GitHub][3])

例子：

```sql
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(20),
    age INT,
    INDEX idx_age(age)
);
```

数据：

| id | name | age |
| -- | ---- | --- |
| 1  | Tom  | 18  |
| 2  | Jack | 20  |
| 3  | Lily | 20  |

主键索引：

```text
id=1 -> 完整行 {1, Tom, 18}
id=2 -> 完整行 {2, Jack, 20}
id=3 -> 完整行 {3, Lily, 20}
```

辅助索引 `idx_age`：

```text
age=18 -> id=1
age=20 -> id=2
age=20 -> id=3
```

执行：

```sql
SELECT * FROM user WHERE age = 20;
```

流程：

```text
先查 idx_age：
    age=20 -> id=2, id=3

再查主键索引：
    id=2 -> 完整行
    id=3 -> 完整行
```

这叫**回表**。

---

## 4. 覆盖索引

如果查询字段已经在辅助索引里，就不需要回表。

例如：

```sql
SELECT id, age
FROM user
WHERE age = 20;
```

因为 `idx_age` 叶子节点里有：

```text
age + 主键 id
```

所以可以直接返回，不需要再查主键索引。

这叫：

```text
覆盖索引：
    查询需要的字段，索引里已经都有了。
```

---

## 5. 联合索引和最左前缀

联合索引：

```sql
INDEX idx_age_name(age, name)
```

不是两个独立索引，而是一棵按 `(age, name)` 排序的 B+Tree。

排序规则：

```text
先按 age 排；
age 相同，再按 name 排；
如果还相同，再按主键排。
```

所以可以用：

```sql
WHERE age = 20;
```

也可以用：

```sql
WHERE age = 20 AND name = 'Tom';
```

但不适合直接用：

```sql
WHERE name = 'Tom';
```

因为这棵树不是全局按 `name` 排的。

文档也说，B+Tree 索引适用于全键值、键值范围和键前缀查找，但键前缀查找只适用于最左前缀。([GitHub][3])

类比：

```text
一本通讯录按：
    省份 -> 城市 -> 姓名

你查广东省，很快；
查广东省深圳市，很快；
但直接查姓名 Tom，就不方便。
```

---

## 6. 索引失效和优化

重点记几个规则：

```text
1. 索引列不要参与表达式计算。
2. 索引列不要作为函数参数。
3. LIKE '%xxx' 前缀通配符通常无法有效用 B+Tree。
4. 联合索引要遵守最左前缀。
5. 尽量使用覆盖索引减少回表。
6. 高选择性的列适合放在联合索引前面。
```

文档中明确说，索引列不能是表达式的一部分，也不能是函数参数，否则无法使用索引；多列索引通常比多个单列索引更好；选择性越高的列区分度越高，查询效率也越高。([GitHub][3])

不好：

```sql
WHERE actor_id + 1 = 5;
```

好：

```sql
WHERE actor_id = 4;
```

## 7. 倒排索引和 B+Tree 的区别

你之前问过倒排索引。

核心区别：

```text
B+Tree：
    字段值 -> 行
    适合等值查询、范围查询、排序。

倒排索引：
    关键词 -> 文档列表
    适合全文搜索。
```

比如文章：

```text
doc1: 我喜欢 MySQL
doc2: MySQL 索引 很重要
doc3: Linux 文件系统 索引
```

倒排索引：

```text
MySQL -> doc1, doc2
索引  -> doc2, doc3
Linux -> doc3
```

搜索：

```text
MySQL 索引
```

就是：

```text
MySQL -> doc1, doc2
索引  -> doc2, doc3

取交集：
doc2
```

文档中说，全文索引用于查找文本中的关键词，使用 `MATCH AGAINST`，底层使用倒排索引，记录关键词到所在文档的映射。([GitHub][3])

## 8. InnoDB 和 MyISAM

重点对比：

| 对比   | InnoDB  | MyISAM     |
| ---- | ------- | ---------- |
| 事务   | 支持      | 不支持        |
| 行级锁  | 支持      | 不支持，只支持表锁  |
| 外键   | 支持      | 不支持        |
| 崩溃恢复 | 更好      | 较弱         |
| 适合   | 绝大多数业务表 | 只读、小表、特殊场景 |

文档也明确总结：InnoDB 是事务型的，支持 `COMMIT` 和 `ROLLBACK`；MyISAM 只支持表级锁，而 InnoDB 还支持行级锁；InnoDB 支持外键和在线热备份，崩溃恢复也更好。([GitHub][3])

现代 MySQL 基本优先选 InnoDB。

## 9. 查询优化

核心原则：

```text
少查列：
    避免 SELECT *。

少查行：
    WHERE 条件尽量精确。

少扫描：
    建合适索引。

少回表：
    用覆盖索引。

看执行计划：
    EXPLAIN。
```

重点关注 `EXPLAIN` 里的：

```text
type：
    访问类型，越接近 const/ref/range 越好，ALL 通常表示全表扫描。

key：
    实际使用的索引。

rows：
    预计扫描行数。

Extra：
    是否 Using index、Using filesort、Using temporary。
```

## 10. 主从复制和读写分离

你之前问过这个。

主从复制的本质：

```text
主库把修改写入 binlog；
从库 I/O 线程拉取 binlog；
写入 relay log；
从库 SQL 线程重放 relay log；
从库逐渐追上主库。
```

读写分离：

```text
写请求：
    走主库。

普通读请求：
    可以走从库。

强一致读 / 写后立刻读：
    走主库。
```

最大问题：

```text
主从延迟。
```

例如：

```text
刚注册用户写入主库；
马上查用户信息，如果请求被路由到从库；
从库还没同步到，就可能查不到。
```

所以你要记住：

```text
读写分离提高读性能，但牺牲了一部分实时一致性。
```

## 11. 分库分表

重点理解：

```text
垂直切分：
    按业务或字段拆。
    例如 user 表、order 表、product 表拆到不同库。

水平切分：
    同一张表的数据按行拆。
    例如 user_0、user_1、user_2。
```

Sharding 策略：

```text
按范围：
    id 1~1000 放库 A，1001~2000 放库 B。

按 hash：
    user_id % 4 决定放哪个库。
```

问题：

```text
跨库 JOIN 困难；
分布式事务困难；
全局唯一 ID；
跨库分页排序复杂；
扩容迁移麻烦。
```

# 四、Redis 核心知识点

## 1. Redis 是什么

Redis 是速度很快的 NoSQL 内存键值数据库；key 只能是字符串，value 支持 String、List、Set、Hash、ZSet 五种常见类型；它还支持持久化、复制和分片。([GitHub][4])

你现在应该这样理解：

```text
Redis = 独立运行的内存 key-value 数据库服务。

不是 MySQL 那种表、行、列；
而是：
    key -> value
```

Redis 和 Java `HashMap` 的区别：

```text
HashMap：
    进程内数据结构；
    单机、单进程使用；
    通常比 Redis 单次访问更快；
    不能天然跨进程共享。

Redis：
    独立服务；
    多个应用服务器都能访问；
    支持过期、持久化、复制、分片、原子命令、多种数据结构。
```

一句话：

```text
HashMap 是你自己进程里的临时内存表；
Redis 是整个系统共享的内存数据库。
```

---

## 2. Redis 五大数据类型

| 类型     | 通俗理解                   | 典型场景             |
| ------ | ---------------------- | ---------------- |
| String | 一个简单值                  | 缓存、计数器、token、验证码 |
| List   | 有序列表，可重复               | 简单队列、历史记录        |
| Set    | 无序不重复集合                | 点赞、关注、共同好友、去重    |
| Hash   | 一个 key 下多个 field-value | 用户对象、商品对象、购物车    |
| ZSet   | 带分数的有序集合               | 排行榜、热搜、延迟队列      |

文档中也列出了五种类型能存储的值和操作：String 可做字符串/数值操作，List 可两端压入弹出，Set 可做交并差，Hash 可操作键值对，ZSet 可按分值和排名获取元素。([GitHub][4])

---

## 3. Set 和 Hash 的区别

你之前特别问过这个。

### Set

```text
key -> 一堆不重复成员
```

例如文章点赞：

```redis
SADD article:100:likes user:1
SADD article:100:likes user:2
SADD article:100:likes user:1
```

结果：

```text
article:100:likes = {user:1, user:2}
```

适合：

```text
是否存在；
去重；
共同好友；
点赞用户集合。
```

### Hash

```text
key -> field -> value
```

例如用户对象：

```redis
HSET user:1 name Tom
HSET user:1 age 18
HSET user:1 city Beijing
```

结果：

```text
user:1:
    name -> Tom
    age  -> 18
    city -> Beijing
```

适合：

```text
用户信息；
商品信息；
购物车商品数量。
```

区别：

```text
Set：
    只有成员，没有 field-value。

Hash：
    一个对象，有多个字段和值。
```

---

## 4. ZSet 和跳跃表

ZSet 是：

```text
Set + score。
```

例如排行榜：

```redis
ZADD rank 100 Tom
ZADD rank 90 Jack
ZADD rank 120 Lily
```

可以理解为：

```text
Jack -> 90
Tom  -> 100
Lily -> 120
```

Redis 需要支持：

```text
查某个成员分数；
按分数排序；
查排名；
查某个分数范围。
```

所以底层常用：

```text
dict：
    member -> score，用于快速查分数。

skiplist：
    按 score 排序，用于排行榜和范围查询。
```

文档中说，跳跃表是有序集合底层实现之一，它是基于多指针有序链表实现的，可以看成多个有序链表；查找时从上层开始，找到区间后再到下一层。([GitHub][4])

你可以记成：

```text
跳跃表 = 带多层目录的有序链表。
```

---

## 5. Redis 字典 dict 和渐进式 rehash

Redis 内部的字典类似 HashMap。

它用于：

```text
根据 key 快速找到 value。
```

底层是散列表，文档中说 `dictht` 是散列表结构，用拉链法解决哈希冲突。([GitHub][4])

重点概念：

```text
table：
    桶数组。

size：
    桶数量。

sizemask：
    用于快速计算下标。

used：
    当前已有 key-value 数量。

dictEntry：
    一个 key-value 节点。

next：
    冲突时串成链表。
```

### 拉链法

如果两个 key hash 后落到同一个桶：

```text
table[2] -> keyA -> keyB -> keyC
```

这就是拉链法。

### 为什么要 rehash

key 越来越多，桶不够用，链表越来越长，查找变慢。

所以需要扩容：

```text
4 个桶 -> 8 个桶 -> 16 个桶
```

### 为什么渐进式 rehash

不能一次性搬所有 key，否则 Redis 会卡住。

所以 Redis 用两个哈希表：

```text
ht[0]：旧表
ht[1]：新表
```

每次搬一点：

```text
这次搬 table[0]；
下次搬 table[1]；
再下次搬 table[2]；
...
```

`rehashidx` 记录搬到哪里。

核心记忆：

```text
Redis dict = HashMap + 拉链法 + 渐进式 rehash。
```

---

## 6. Redis 使用场景

文档中列出了计数器、缓存、查找表、消息队列、会话缓存、分布式锁等场景；比如 String 可做计数器，List 可用 `lpush/rpop` 做简单消息队列，Set 可做共同好友，ZSet 可做排行榜。([GitHub][4])

你需要重点记这些：

### 计数器

```redis
INCR article:100:views
```

用于：

```text
浏览量；
点赞数；
接口调用次数。
```

### 缓存

```redis
SET product:100 "{商品详情JSON}" EX 300
```

用于：

```text
热点商品；
用户信息；
配置；
首页数据。
```

### 消息队列

```redis
LPUSH queue:order order1001
RPOP queue:order
```

Redis List 可以做简单队列，但文档也提醒，更推荐 Kafka、RabbitMQ 等专业消息中间件。([GitHub][4])

你之前问过 Kafka/RabbitMQ，可以这样记：

```text
Redis List：
    简单小篮子。

RabbitMQ：
    可靠任务分发中心。

Kafka：
    高吞吐、可回放事件日志。
```

### 会话缓存

```redis
SET session:abc123 "user_id=1" EX 1800
```

解决：

```text
多台应用服务器共享登录状态。
```

文档中说，Redis 可统一存储多台应用服务器的会话信息，使应用服务器不再保存状态，用户可以请求任意服务器，更容易实现高可用和可伸缩。([GitHub][4])

### 分布式锁

```redis
SET lock:product:100 uuid NX EX 10
```

解决：

```text
多台机器上的多个进程，如何互斥操作同一份资源。
```

文档中也说，分布式场景下不能用单机锁同步多个节点进程，可以使用 `SETNX` 或 RedLock 实现分布式锁。([GitHub][4])

注意：

```text
加锁要用 SET key value NX EX seconds；
value 要唯一；
释放锁要判断 value；
最好用 Lua 脚本保证判断和删除原子。
```

## 7. 过期时间和淘汰策略

Redis 可以给 key 设置过期时间，到期自动删除；但对于 Hash 这种容器，只能给整个 key 设置过期时间，不能给 Hash 里面单个 field 设置过期时间。([GitHub][4])

例如：

```redis
SET code:phone:13800000000 123456 EX 300
```

表示验证码 5 分钟后过期。

内存满了怎么办？

Redis 可以设置淘汰策略，例如：

```text
allkeys-lru：
    从所有 key 中淘汰最近最少使用的。

volatile-lru：
    从设置了过期时间的 key 中淘汰最近最少使用的。

noeviction：
    不淘汰，内存满了写入报错。
```

文档中也列出多种淘汰策略，并说明 Redis 的淘汰算法实际实现不是扫描所有 key，而是抽样一部分并从中选出要淘汰的 key。([GitHub][4])

---

## 8. 持久化：RDB 和 AOF

Redis 是内存数据库，但可以把数据保存到磁盘。

两种方式：

```text
RDB：
    定期生成快照。
    像拍照片。

AOF：
    记录每次写命令。
    像记流水账。
```

文档中也说，Redis 支持 RDB 快照和 AOF 日志两种持久化策略。([GitHub][4])

核心对比：

| 对比   | RDB           | AOF     |
| ---- | ------------- | ------- |
| 本质   | 某个时间点的数据快照    | 写命令日志   |
| 恢复速度 | 快             | 相对慢     |
| 数据安全 | 可能丢最近一次快照后的数据 | 通常更安全   |
| 文件体积 | 较小            | 可能较大    |
| 适合   | 备份、快速恢复       | 更关注数据少丢 |

---

## 9. Redis 复制、Sentinel、Cluster

### 复制

```text
主节点负责写；
从节点复制主节点数据；
从节点可以分担读请求。
```

用途：

```text
扩展读性能；
提高可用性；
做备份。
```

### Sentinel

解决：

```text
主节点挂了，谁来自动切换？
```

Sentinel 会监控主从节点，当主节点不可用时，自动选一个从节点提升为新主节点。

### Cluster

解决：

```text
单台 Redis 内存不够；
单台 Redis 写能力不够。
```

Cluster 通过分片把不同 key 分散到不同 Redis 节点。文档中也提到 Redis Cluster 实现了分布式支持。([GitHub][4])

记忆：

```text
复制：
    多份相似数据，主要扩展读能力和可用性。

分片 / Cluster：
    不同数据放不同节点，主要扩展容量和写能力。
```

# 五、你之前重点问题的统一总结

## 1. 普通 SELECT 不加锁，为什么还能可重复读？

答案：

```text
因为 InnoDB 的普通 SELECT 是快照读。
它读的是 Read View 可见的版本。
别人可以修改并提交，但你仍然可以通过 undo log 读到旧版本。
```

所以：

```text
不加锁 ≠ 不能保证可重复读。
```

---

## 2. 当前读和快照读的区别

```text
快照读：
    普通 SELECT。
    不加锁。
    读历史版本。

当前读：
    SELECT FOR UPDATE / UPDATE / DELETE / INSERT。
    加锁。
    读最新版本。
```

---

## 3. 不可重复读和幻读区别

```text
不可重复读：
    同一行变了。

幻读：
    同一范围内多了/少了行。
```

---

## 4. 主键索引、辅助索引、联合索引

```text
主键索引：
    叶子节点存完整行。

辅助索引：
    叶子节点存索引列 + 主键值。

联合索引：
    多列组成一个 key，按从左到右排序。
```

---

## 5. 回表和覆盖索引

```text
回表：
    先查辅助索引拿主键；
    再查主键索引拿完整行。

覆盖索引：
    查询字段索引里都有；
    不需要回表。
```

---

## 6. B+Tree 和倒排索引

```text
B+Tree：
    字段值 -> 行。
    适合等值、范围、排序。

倒排索引：
    关键词 -> 文档列表。
    适合全文搜索。
```

---

## 7. Redis 和 HashMap

```text
HashMap：
    本进程内用，通常更快，但不能跨进程共享。

Redis：
    独立内存数据库服务，支持共享、过期、持久化、复制、分布式能力。
```

---

## 8. Redis Set 和 Hash

```text
Set：
    一堆不重复成员。
    适合点赞、关注、共同好友。

Hash：
    一个对象的多个字段和值。
    适合用户信息、商品信息、购物车数量。
```

---

## 9. Kafka、RabbitMQ 和 Redis

```text
Redis：
    主业是缓存和内存数据结构，可以做简单队列。

RabbitMQ：
    主业是可靠业务消息投递。

Kafka：
    主业是高吞吐事件流和日志系统。
```

---

## 10. 会话缓存和分布式锁

```text
会话缓存：
    解决多台应用服务器共享用户登录状态。

分布式锁：
    解决多台服务器互斥操作同一资源。
```

# 六、最终复习版口诀

你可以把 4 篇文档压缩成下面这份：

```text
SQL：
    SELECT 是核心。
    WHERE 过滤行，HAVING 过滤组。
    DISTINCT 是去重，GROUP BY 是分组。
    子查询是查询里嵌查询，相关子查询依赖外层当前行。
    JOIN 是多表关联，LEFT JOIN 能保留左表无匹配行。

数据库原理：
    事务靠 ACID 保证正确性。
    并发问题重点是脏读、不可重复读、幻读。
    普通 SELECT 是快照读，靠 MVCC。
    UPDATE / DELETE / FOR UPDATE 是当前读，靠锁。
    MVCC 解决普通读一致性，Next-Key Lock 解决当前读范围幻读。

MySQL：
    InnoDB 是主流存储引擎。
    B+Tree 适合磁盘和范围查询。
    主键索引叶子节点存整行。
    辅助索引叶子节点存主键。
    查辅助索引再查主键叫回表。
    查询字段都在索引里叫覆盖索引。
    联合索引遵守最左前缀。
    索引列不要参与函数和表达式计算。
    主从复制基于 binlog、relay log、SQL 线程重放。
    读写分离能提升读性能，但要注意主从延迟。

Redis：
    Redis 是内存 key-value 数据库。
    key 是字符串，value 有 String/List/Set/Hash/ZSet。
    String 做缓存和计数。
    List 做简单队列。
    Set 做去重和集合运算。
    Hash 做对象。
    ZSet 做排行榜。
    dict 类似 HashMap，使用拉链法和渐进式 rehash。
    skiplist 是多层有序链表，用于 ZSet 排序。
    Redis 适合缓存、计数器、会话、分布式锁、排行榜。
    RDB 是快照，AOF 是写日志。
    复制扩展读，分片扩展容量和写。
```

# 一、基础概念类

## 1. 《Linux kernel 内存管理的基本概念》

这篇是整个分类的入口。它不是讲某一个具体函数，而是讲 **Linux 内存管理为什么复杂**。文章强调内存是 Linux 系统里牵涉面极广的资源，上至应用程序，下至 kernel 和 driver 都离不开它；复杂性主要来自两个方面：需求复杂，以及 CPU、Device、Memory 三者之间的关系复杂。([Wowotech][2])

你要抓住这几个核心点：

### 1.1 内存管理面对的对象不只是“内存条”

很多初学者以为内存管理就是：

```text
谁申请内存，内核给它一块；
谁释放内存，内核收回来。
```

但内核视角更复杂：

```text
用户进程需要虚拟地址空间
内核自己需要动态分配对象
文件系统需要 page cache
块设备需要缓存和回写
驱动需要 DMA buffer
设备可能需要连续物理内存
CPU 访问内存会经过 Cache / TLB / MMU
多核访问共享内存还要考虑一致性和顺序
```

所以内存管理不是一个“malloc/free”的问题，而是一个系统级问题。

### 1.2 Linux 要同时服务三类主体

可以把 Linux 内存管理看成给三类对象提供服务：

```text
用户进程：
  关心 malloc、mmap、brk、缺页异常、swap、OOM。

内核本身：
  关心 kmalloc、vmalloc、alloc_pages、slab/slub、page cache、页回收。

设备驱动：
  关心 DMA、一致性内存、streaming mapping、scatterlist、CMA、IOMMU。
```

这也是为什么同样叫“内存地址”，会有很多类型：

```text
用户虚拟地址
内核虚拟地址
物理地址
PFN
DMA 地址
总线地址
I/O 地址
```

### 1.3 内核内存管理的核心抽象

后续所有文章都会反复出现这些抽象：

```text
page：
  内核用 struct page 描述一个物理页框。

PFN：
  Page Frame Number，物理页号。

zone：
  按用途和硬件限制划分的物理内存区域，比如 DMA、Normal、Movable。

node：
  NUMA 系统中的内存节点。

mm_struct：
  描述一个进程的用户态地址空间。

vm_area_struct：
  描述进程地址空间中的一段连续虚拟内存区域。

pte/pmd/pud/pgd：
  多级页表里的不同层级。

address_space：
  文件 page cache 的核心抽象。

lruvec / LRU：
  页面回收相关结构，用于管理可回收页面。
```

## 2. 《Linux 内存模型》

这篇主要解释 Linux 如何描述物理内存。文章明确提到 Linux 支持三种 memory model：**flat memory model、discontiguous memory model、sparse memory model**。([Wowotech][3])

### 2.1 为什么需要内存模型？

CPU 看到的物理地址空间不一定是连续的。比如：

```text
0x00000000 - 0x3fffffff   RAM
0x40000000 - 0x4fffffff   MMIO
0x50000000 - 0x7fffffff   RAM
```

中间可能有洞，可能是设备寄存器，也可能是保留区域。

Linux 需要回答两个问题：

```text
1. 给定一个物理地址 / PFN，怎么找到对应的 struct page？
2. 给定一个 struct page，怎么反推出它代表哪个物理页？
```

这就是内存模型要解决的问题。

### 2.2 Flat Memory Model

Flat memory 假设物理内存基本连续。

它的核心思想是：

```text
所有物理页都有一个全局 mem_map 数组描述。

PFN -> struct page:
  page = mem_map + pfn

struct page -> PFN:
  pfn = page - mem_map
```

优点：

```text
实现简单
转换快
适合早期简单系统
```

缺点：

```text
如果物理地址空间有很多空洞，mem_map 也要为不存在的 PFN 准备 struct page，浪费内存。
```

### 2.3 Discontiguous Memory Model

Discontiguous memory 适合 NUMA 或物理内存不连续的系统。

它按 node 管理：

```text
node0:
  mem_map for node0

node1:
  mem_map for node1

node2:
  mem_map for node2
```

这样每个节点只描述自己拥有的物理页。

优点：

```text
适合 NUMA
比 flat memory 更适合不连续物理内存
```

缺点：

```text
结构复杂
PFN 到 page 的转换需要先定位 node
```

### 2.4 Sparse Memory Model

Sparse memory 是现代内核常见模型，尤其适合：

```text
内存热插拔
物理地址空间巨大
内存分布稀疏
NUMA
ARM64 / x86_64 大内存系统
```

它把物理地址空间按 section 切分：

```text
section0
section1
section2
...
```

只有真正存在内存的 section 才需要对应的 `struct page` 信息。

核心思想：

```text
PFN -> section -> mem_map -> struct page
```

优点：

```text
适合大地址空间
支持 memory hotplug
不需要为所有地址洞分配 struct page
```

### 2.5 node、zone、page 的关系

你可以这样理解：

```text
整个机器
  ↓
NUMA node
  ↓
zone
  ↓
pageblock
  ↓
page frame
  ↓
struct page
```

举例：

```text
node0
  ZONE_DMA
  ZONE_NORMAL
  ZONE_MOVABLE

node1
  ZONE_NORMAL
  ZONE_MOVABLE
```

`node` 更多体现硬件距离；`zone` 更多体现分配限制；`page` 是最小管理单位。

## 3. 《ARM64 架构下地址翻译相关的宏定义》

这篇是 ARM64 地址翻译基础。文章指出，x86 里的 Linear Address 说法不适合 ARM64，ARM64 中更常见的是 VA、IPA、PA 等地址类型。([Wowotech][4])

### 3.1 ARM64 下常见地址类型

你先记住这几个：

```text
VA：Virtual Address
  CPU 执行 load/store 指令时使用的虚拟地址。

PA：Physical Address
  物理内存总线上的地址。

IPA：Intermediate Physical Address
  虚拟化场景下，Guest 看到的“物理地址”。

OA：Output Address
  地址翻译后的输出地址。
```

普通非虚拟化场景：

```text
VA -> PA
```

虚拟机场景：

```text
Guest VA -> Guest IPA -> Host PA
```

第一阶段翻译由 Guest 页表完成，第二阶段翻译由 Hypervisor 管理。

### 3.2 ARM64 页表层级

ARM64 常见页表层级是：

```text
PGD
  ↓
PUD
  ↓
PMD
  ↓
PTE
  ↓
物理页
```

不过具体有几级，取决于：

```text
虚拟地址位数
页大小：4KB / 16KB / 64KB
配置项
```

比如 4KB page + 48-bit VA，通常需要 4 级页表。

### 3.3 为什么有那么多宏？

因为内核要在 C 代码和汇编代码里计算：

```text
一个虚拟地址属于 PGD 的哪个 entry？
属于 PUD 的哪个 entry？
属于 PMD 的哪个 entry？
属于 PTE 的哪个 entry？
页内偏移是多少？
```

所以会有类似：

```text
PAGE_SHIFT
PAGE_SIZE
PAGE_MASK
PTRS_PER_PTE
PMD_SHIFT
PUD_SHIFT
PGDIR_SHIFT
```

这些宏本质上都是在描述：

```text
地址的哪些 bit 用来索引哪一级页表。
```

### 3.4 页大小影响很大

比如 4KB 页：

```text
页内偏移占 12 bit
因为 2^12 = 4096
```

如果虚拟地址 48 bit，那么剩下：

```text
48 - 12 = 36 bit
```

这些 bit 要分给各级页表索引。

所以页大小不同，页表层级和每级索引位数都会不同。

## 4. 《浅谈 Cache Memory》

这篇讲 CPU cache。它和内存管理、DMA、memory barrier 都有关。([Wowotech][5])

### 4.1 为什么需要 Cache？

CPU 太快，内存太慢。

如果每次 CPU 读写都直接访问 DRAM，CPU 会大量等待。Cache 的作用就是：

```text
把最近访问过、可能马上还会访问的数据，放到 CPU 附近。
```

通俗类比：

```text
内存 = 仓库
Cache = 办公桌
CPU = 正在工作的人

你不会每拿一张纸都跑仓库，而是把常用文件先放桌上。
```

### 4.2 Cache Line

Cache 不是按字节缓存，而是按 cache line 缓存。

常见 cache line 大小：

```text
64 bytes
```

如果 CPU 读取地址 `0x1000`，可能会把：

```text
0x1000 - 0x103f
```

整个 cache line 拉进来。

这会影响：

```text
性能
false sharing
DMA cache coherency
内存屏障
```

### 4.3 Cache 命中与不命中

```text
cache hit：
  数据在 cache 里，CPU 很快拿到。

cache miss：
  数据不在 cache 里，需要从下一级 cache 或内存加载。
```

多级 cache：

```text
L1：最快，最小
L2：较快，较大
L3：更大，多个 core 共享
DRAM：最慢
```

### 4.4 写策略

常见写策略：

```text
write-through：
  写 cache 的同时也写内存。

write-back：
  先写 cache，等 cache line 被替换或显式 flush 时再写回内存。
```

write-back 性能好，但会带来一致性问题：

```text
CPU 以为数据已经写了；
设备直接读内存时，可能还读到旧数据。
```

这就是 DMA 需要 cache maintenance 的原因。

### 4.5 Cache 与 DMA

如果设备 DMA 写内存：

```text
设备把新数据写到 DRAM
CPU cache 里可能还有旧数据
CPU 再读时可能读到旧 cache
```

如果 CPU 写 DMA buffer：

```text
CPU 只写到了 cache
DRAM 里还是旧数据
设备 DMA 读取 DRAM 时读到旧数据
```

所以非 coherent 平台需要：

```text
DMA_TO_DEVICE 前 clean cache
DMA_FROM_DEVICE 后 invalidate cache
```

DMA mapping API 会帮驱动处理这些平台差异。

## 5. 《内存一致性模型》

这篇讲 memory consistency model，也就是多核系统里，不同 CPU 对共享内存访问顺序的可见性规则。文章说明内存一致性模型用于描述多线程对共享存储器的访问行为，不同模型下共享内存访问行为可能差异很大。([Wowotech][6])

### 5.1 为什么需要内存一致性模型？

你写代码时以为顺序是：

```c
x = 1;
flag = 1;
```

你以为其他 CPU 看到 `flag == 1` 时，一定也能看到 `x == 1`。

但硬件和编译器可能重排：

```text
编译器重排
CPU pipeline 重排
store buffer 延迟可见
cache coherency 延迟传播
```

所以另一个 CPU 可能先看到 `flag = 1`，却还没看到 `x = 1`。

### 5.2 强一致性与弱一致性

强一致性模型更接近程序员直觉：

```text
所有 CPU 看到的内存操作顺序接近一致。
```

弱一致性模型允许更多重排：

```text
只要单线程语义不被破坏，CPU 可以为了性能改变其他 CPU 观察到的顺序。
```

ARM/ARM64 通常比 x86 更弱，所以内核里 memory barrier 非常重要。

### 5.3 Memory Barrier 的作用

barrier 本质是告诉编译器/CPU：

```text
这个点前后的内存操作不能随便乱序。
```

常见屏障：

```text
mb()：
  full memory barrier，读写都约束。

rmb()：
  read memory barrier，约束读。

wmb()：
  write memory barrier，约束写。

smp_mb() / smp_rmb() / smp_wmb()：
  SMP 场景下使用。

READ_ONCE() / WRITE_ONCE()：
  防止编译器拆分、合并、重复读取。
```

### 5.4 和锁的关系

锁不仅提供互斥，也通常隐含内存顺序保证。

比如：

```c
spin_lock(&lock);
shared_data = 1;
spin_unlock(&lock);
```

另一个 CPU：

```c
spin_lock(&lock);
read shared_data;
spin_unlock(&lock);
```

锁保证临界区访问的顺序性和可见性。

但无锁代码、设备寄存器访问、DMA 描述符提交等场景，经常需要显式 barrier。

# 二、启动早期、页表、地址映射类

## 6. 《内存初始化（上）》

这篇讲启动阶段内存管理的总流程。文章属于 ARM64/Linux 4.4.6 语境，关注 kernel 启动早期如何从还没有完整内存管理，逐步进入正常内存管理。([Wowotech][7])

### 6.1 启动早期为什么不能直接用 buddy？

因为 buddy 系统本身需要很多前置条件：

```text
struct page 数组要初始化
zone 要初始化
free_area 要初始化
页表映射要基本可用
内核自身代码/data/bss 要保留
dtb/initrd/reserved-memory 要识别
```

这些东西没准备好之前，内核不能调用正常的 `alloc_pages()`。

所以早期需要一个简单内存管理器：

```text
memblock
```

### 6.2 memblock 的核心思想

memblock 维护两类区域：

```text
memblock.memory：
  系统有哪些物理内存。

memblock.reserved：
  哪些物理内存已经被占用或保留。
```

早期分配内存时，不是从 buddy free list 里拿，而是在 `memblock.memory` 中找一块没被 `reserved` 覆盖的区域，然后把它加入 `memblock.reserved`。

通俗理解：

```text
memblock.memory = 酒店所有房间清单
memblock.reserved = 已经被订走的房间清单
分配 = 找一个还没订的房间，然后登记为已订
```

### 6.3 启动阶段要保留哪些内存？

常见 reserved 区域：

```text
kernel image
initrd
device tree blob
页表内存
reserved-memory 节点
CMA 区域
crashkernel
firmware 保留区域
设备专用内存
```

### 6.4 从 memblock 到 buddy

启动早期：

```text
memblock 管理所有物理内存
```

初始化后期：

```text
把可用 page frame 释放给 buddy
```

之后正常运行：

```text
alloc_pages()
kmalloc()
vmalloc()
slub
```

才真正可用。

关键点是：

**memblock 不是运行期通用分配器，它主要服务 boot 阶段。**

## 7. 《内存初始化代码分析（一）：identity mapping 和 kernel image mapping》

这篇分析 ARM64 启动汇编里的 `__create_page_tables`，重点是 identity mapping 和 kernel image mapping。文章说明它是按照 `__create_page_tables` 代码执行顺序分析，代码来自 ARM64 Linux 4.4.6。([Wowotech][8])

### 7.1 为什么启动早期需要 identity mapping？

启动早期 CPU 刚开 MMU 时，存在一个危险点：

```text
开 MMU 前：
  CPU 按物理地址执行指令。

开 MMU 后：
  CPU 按虚拟地址执行指令。
```

如果打开 MMU 的瞬间，当前 PC 地址无法在页表中找到正确映射，CPU 就会跑飞。

identity mapping 的意思是：

```text
VA == PA
```

比如：

```text
虚拟地址 0x80000 映射到物理地址 0x80000
```

这样打开 MMU 前后，当前执行流可以平滑过渡。

### 7.2 kernel image mapping 是什么？

内核最终不是运行在物理地址上，而是运行在内核虚拟地址区间。

比如 ARM64 常见内核虚拟地址区间在高地址：

```text
ffff000000000000...
```

kernel image mapping 要建立：

```text
内核虚拟地址 -> 内核镜像所在物理地址
```

这样 MMU 开启后，内核可以跳到正式虚拟地址继续执行。

### 7.3 这篇文章最该掌握的点

你不需要一开始记住每行汇编，但要知道：

```text
__create_page_tables 做两件大事：

1. 建 identity mapping
   保证打开 MMU 时当前代码还能继续跑。

2. 建 kernel image mapping
   保证内核可以在高地址虚拟空间运行。
```

### 7.4 页表项的创建逻辑

ARM64 页表创建大概是：

```text
计算虚拟地址属于哪个 PGD/PUD/PMD/PTE entry
如果中间页表不存在，分配/清零页表页
填入 table descriptor 或 block/page descriptor
设置属性：读写、可执行、cacheable、shareable 等
```

在启动汇编里，这些逻辑由一堆宏完成，比如创建中间 level 页表项、创建 block mapping 等。

## 8. 《内存初始化代码分析（二）：内存布局》

这篇重点讲启动阶段如何从 device tree / boot 参数中识别内存布局，以及保留内核、initrd、fdt 等区域。搜索结果摘录显示文章讨论了通过 `linux,usable-memory` 或 `reg` 属性保存 memory node 的物理地址信息，并分析保留 fdt、内核、initrd 所占内存。([Wowotech][9])

### 8.1 内核从哪里知道机器有多少内存？

常见来源：

```text
device tree memory node
UEFI memory map
bootloader 传入信息
架构相关探测
```

ARM/ARM64 嵌入式系统里，device tree 很常见：

```dts
memory@80000000 {
    device_type = "memory";
    reg = <0x0 0x80000000 0x0 0x40000000>;
};
```

含义可能是：

```text
从物理地址 0x80000000 开始，有 1GB 内存。
```

### 8.2 为什么有些内存不能进入普通内存管理？

因为一些区域已经有特殊用途：

```text
内核镜像区域
initrd 区域
device tree blob 区域
firmware 保留区域
reserved-memory
CMA 预留区域
crashkernel
设备专用 buffer
secure world 区域
```

这些区域不能随便交给 buddy，否则普通内核分配可能覆盖它们。

### 8.3 memblock 初始化内存布局

大概流程：

```text
扫描 device tree / boot 参数
  ↓
memblock_add() 添加可用内存
  ↓
memblock_reserve() 保留特殊区域
  ↓
后续初始化时只把真正可用区域释放给 buddy
```

### 8.4 为什么 MemTotal 小于机器物理内存？

因为 `/proc/meminfo` 里的 MemTotal 不是“内存条容量”，而是 Linux 最终能管理的内存总量。内核自身、保留区域、firmware 区域、某些 reserved memory 不一定计入普通可管理内存。RSS 摘录里对 MemTotal 也明确说明，它对应内核能管理的 page frame 数，不等于机器配置的物理内存总数。([蜗窝科技][10])

## 9. 《内存初始化代码分析（三）：创建系统内存地址映射》

这篇进入 `paging_init` / `map_mem` 这类核心流程，讲系统内存的正式映射。搜索结果显示文章说，经过前两篇过渡后，终于来到内存初始化核心部分：`paging_init`；同时提到 memblock 已经收集了内存布局，也知道 free memory 信息。([Wowotech][11])

### 9.1 前面两篇解决了什么？

前两篇主要解决：

```text
1. 先能跑起来：
   identity mapping + kernel image mapping

2. 知道物理内存布局：
   哪些是 RAM，哪些 reserved
```

第三篇开始解决：

```text
给系统内存建立完整线性映射。
```

### 9.2 什么是 linear mapping？

linear mapping，也常叫 direct mapping，意思是：

```text
一大段内核虚拟地址连续映射到物理内存。
```

通常存在固定偏移关系：

```text
kernel_va = phys_addr + PAGE_OFFSET
```

有了 linear mapping，内核可以很方便地把物理页转成可访问的内核虚拟地址。

比如：

```text
page -> pfn -> phys -> virt
```

### 9.3 为什么要映射整个系统内存？

因为内核经常要直接操作物理页内容：

```text
清零 page
复制 page
page cache 读写
内存回收
fork COW
块 I/O buffer
网络 buffer
```

如果没有 linear mapping，每次访问物理页都要临时映射，成本太高。

### 9.4 map_mem 做什么？

大概逻辑：

```text
遍历 memblock.memory 中的 RAM 区域
  ↓
跳过或特殊处理 reserved / no-map 区域
  ↓
按照合适粒度建立页表映射
  ↓
设置内存属性：normal memory、cacheable、shareable、RW、NX 等
```

### 9.5 启动早期页表与正式页表的区别

启动早期页表：

```text
只够内核启动
映射范围较小
结构较临时
主要服务开 MMU 和 early boot
```

正式页表：

```text
覆盖系统内存
考虑权限和属性
支持后续内核正常访问物理页
```

## 10. 《Fix-Mapped Addresses》

这篇讲 fixmap。文章标题就是 Fix-Mapped Addresses。([Wowotech][12])

### 10.1 fixmap 是什么？

fixmap 是一段固定虚拟地址区域。

它的特点：

```text
虚拟地址固定
物理地址可以动态指定
常用于启动早期或特殊用途映射
```

普通映射通常是：

```text
给我一个物理地址，内核分配一个虚拟地址来映射它。
```

fixmap 是：

```text
我先规定好一组虚拟地址槽位，需要时把某个物理地址塞进去。
```

### 10.2 为什么需要 fixmap？

启动早期很多内存分配器还没准备好，但内核需要临时访问某些物理地址：

```text
device tree
early console MMIO
页表页
ACPI/UEFI 表
临时 ioremap
高端内存临时映射
```

这时不能依赖完整 vmalloc/ioremap 机制，就需要 fixmap。

### 10.3 fixmap 的典型使用方式

大致流程：

```text
选定一个 fixed index
  ↓
通过 fix_to_virt(index) 得到固定虚拟地址
  ↓
把目标物理地址填到对应 PTE
  ↓
flush TLB
  ↓
通过固定虚拟地址访问目标物理地址
```

### 10.4 fixmap 和普通 ioremap 的区别

```text
普通 ioremap：
  运行期使用，动态分配虚拟地址范围。

fixmap：
  虚拟地址预留固定，适合早期或特殊固定映射。
```

### 10.5 fixmap 的直观类比

你可以把 fixmap 想成酒店前台的“固定备用房间号”：

```text
房间号固定：比如 8888
今天可以安排 A 客人
明天可以安排 B 客人
房间号不变，但里面住谁可以变
```

fixmap：

```text
虚拟地址固定
映射的物理地址可变
```

## 11. 《fixmap addresses 原理》

这篇更偏 ARM64/aarch64 下 fixmap 代码实现。搜索结果显示，文章代码分析基于 Linux 4.15，架构基于 aarch64，并提到只需要设置 PTE 页表对应的页表项。([Wowotech][13])

### 11.1 fixmap 地址空间如何布局？

内核会预留一段虚拟地址范围：

```text
FIXADDR_START ... FIXADDR_TOP
```

每个 fixmap entry 对应一个 page 大小的虚拟地址槽。

概念上：

```text
FIX_HOLE
FIX_EARLYCON_MEM_BASE
FIX_TEXT_POKE0
FIX_BTMAP_BEGIN
...
```

每个枚举值都能转换为一个虚拟地址。

### 11.2 fix_to_virt / virt_to_fix

核心转换：

```text
fix_to_virt(index):
  固定 index -> 固定虚拟地址

virt_to_fix(addr):
  固定虚拟地址 -> index
```

这说明 fixmap 的虚拟地址不是动态分配出来的，而是编译期/启动时规划好的。

### 11.3 set_fixmap 做什么？

类似：

```text
set_fixmap(idx, phys_addr)
```

实际含义：

```text
把 fixmap 中 idx 对应的固定虚拟地址
映射到 phys_addr 对应的物理页。
```

底层会修改 PTE，然后做必要的 TLB 维护。

### 11.4 early_ioremap 与 fixmap

early_ioremap 常借助 fixmap，因为启动早期还没有完整 ioremap 机制。

流程大概是：

```text
early_ioremap(phys, size)
  ↓
找一段 fixmap slot
  ↓
建立 PTE
  ↓
返回固定虚拟地址
```

### 11.5 注意 no-map 区域

device tree 的 reserved-memory 里可能有：

```dts
no-map;
```

意思是：

```text
内核不要在线性映射中映射这段物理内存。
```

如果驱动需要访问，通常要通过专门 ioremap 或 DMA 机制处理。CMA 则通常不能带 no-map，因为 CMA 区域平时还要给普通可迁移页使用。RSS 中 CMA 摘录也提到 CMA reserved memory 节点需要 reusable，不能有 no-map。([蜗窝科技][14])

## 12. 《TLB flush 操作》

这篇讲 TLB flush。文章说明 Linux VM 子系统在很多场合都需要 TLB flush，并希望把相关知识点描述清楚。([Wowotech][15])

### 12.1 TLB 是什么？

TLB 是页表转换结果的缓存。

页表在内存里，查页表很慢，所以 CPU 缓存最近的地址翻译：

```text
虚拟地址 VA -> 物理地址 PA
```

TLB hit：

```text
不需要走完整页表，速度快。
```

TLB miss：

```text
需要硬件或软件 page table walk。
```

### 12.2 为什么修改页表后要 flush TLB？

因为页表变了，但 CPU 可能还缓存旧转换。

比如：

```text
原来：
  VA 0x400000 -> PA 0x100000

内核修改页表：
  VA 0x400000 -> PA 0x200000

但 TLB 里还有旧项：
  VA 0x400000 -> PA 0x100000
```

如果不 flush，CPU 继续用旧映射，后果可能是：

```text
访问错物理页
权限检查错误
use-after-free
数据损坏
安全问题
```

### 12.3 哪些场景需要 TLB flush？

常见场景：

```text
munmap
mprotect
exit 释放地址空间
page reclaim 解除 PTE 映射
COW 修改 PTE
内核修改 vmalloc 映射
ioremap/iounmap
页迁移
NUMA balancing
KASLR / 启动早期页表切换
```

### 12.4 本地 flush 与远程 flush

单核系统：

```text
只 flush 当前 CPU 的 TLB。
```

SMP 系统：

```text
一个 mm 可能正在多个 CPU 上运行；
修改页表后，要通知相关 CPU flush TLB。
```

这通常通过 IPI 完成：

```text
CPU0 修改页表
  ↓
发送 IPI 给 CPU1/CPU2
  ↓
远程 CPU 执行 TLB invalidate
```

### 12.5 为什么 TLB flush 很贵？

因为它可能导致：

```text
丢失地址翻译缓存
后续大量 TLB miss
跨 CPU IPI
等待远程 CPU 响应
影响性能
```

所以内核会尽量：

```text
batch flush
延迟 flush
只 flush 范围
只 flush 相关 mm
只通知真正使用该 mm 的 CPU
```

## 13. 《ARM64 Kernel Image Mapping 的变化》

这篇讲 ARM64 内核镜像映射机制变化。搜索结果显示文章围绕 `create_page_tables`、section mapping、PUD/PMD/PTE 等展开。([Wowotech][16])

### 13.1 Kernel Image Mapping 是什么？

就是把内核镜像所在物理内存映射到内核虚拟地址。

内核镜像包括：

```text
_text
_etext
_data
_bss
init text/data
rodata
```

不同段需要不同权限：

```text
代码段：可执行，只读
rodata：只读，不可执行
data/bss：可读写，不可执行
init 段：启动后可释放
```

### 13.2 早期为什么喜欢 section/block mapping？

因为启动早期简单高效。

block mapping 可以用一个较高级页表项映射大块内存，例如：

```text
2MB
1GB
```

优点：

```text
页表项少
TLB 压力小
启动代码简单
```

缺点：

```text
权限粒度粗
不能精细区分 text/rodata/data 权限
不利于安全加固
```

### 13.3 为什么后来要变化？

现代内核更重视安全：

```text
代码段不可写
只读数据不可写
数据段不可执行
模块区域权限控制
KASLR 随机化
ro_after_init
```

这些都要求更细粒度映射。

如果用 1GB block mapping，代码、只读数据、可写数据可能落在同一个大块里，权限不好细分。

### 13.4 变化的核心方向

从粗粒度映射走向细粒度映射：

```text
早期：
  大块 block mapping，简单快速。

后来：
  更细页表映射，便于设置精确权限。
```

### 13.5 和 KASLR 的关系

KASLR 会随机化内核映射位置，这对页表建立提出更多要求：

```text
kernel image 虚拟地址可能变化
物理加载地址也可能变化
页表创建逻辑要支持随机偏移
```

## 14. 《KASLR》

这篇讲 Kernel Address Space Layout Randomization。搜索结果显示，文章讨论 ARM64 kernel 映射范围、`KIMAGE_VADDR`、`__create_page_tables` 等。([Wowotech][17])

### 14.1 KASLR 解决什么问题？

如果内核地址固定，攻击者更容易构造攻击：

```text
知道内核函数地址
知道 gadget 地址
知道内核全局变量地址
```

KASLR 的目标是：

```text
让内核镜像加载/映射地址随机化。
```

这样攻击者不能轻易知道内核符号实际地址。

### 14.2 KASLR 随机化什么？

可能包括：

```text
物理加载地址随机化
虚拟映射地址随机化
模块加载地址随机化
vmalloc 区域布局变化
```

不同架构和内核版本支持程度不同。

### 14.3 ARM64 KASLR 的难点

ARM64 启动早期需要很早建立页表。

但 KASLR 又要求：

```text
随机偏移要尽早确定
页表要按随机后的位置建立
异常向量、内核镜像、线性映射等都要协调
```

### 14.4 KASLR 和符号地址

开启 KASLR 后：

```text
System.map 里的静态地址
```

不一定等于运行时真实地址。

调试 vmcore、ftrace、kprobe、crash 时要注意符号偏移。

### 14.5 KASLR 的局限

KASLR 不是万能防御：

```text
信息泄露可以绕过 KASLR
侧信道可能泄露地址
内核日志可能暴露地址
某些调试配置会降低效果
```

所以它是安全加固的一环，不是全部。

# 三、物理页分配、碎片、SLUB、CMA

## 15. 《linux kernel 内存碎片防治技术》

这篇讲内存碎片。文章指出 Linux 物理内存管理使用 buddy system，并讨论碎片导致高阶连续页分配失败的问题。([Wowotech][18])

### 15.1 什么是内存碎片？

内存总量够，但连续空间不够。

比如你有 16 个 page：

```text
F U F U F U F U F U F U F U F U
```

F 是 free，U 是 used。

空闲页有 8 个，但没有 2 个连续空闲页，因此 order-1 分配都会失败。

### 15.2 buddy system 为什么会碎片化？

buddy 分配按 2 的幂次管理：

```text
order-0: 1 page
order-1: 2 pages
order-2: 4 pages
order-3: 8 pages
...
```

分配小页时，会不断拆大块：

```text
8 pages -> 4 + 4
4 pages -> 2 + 2
2 pages -> 1 + 1
```

释放时，只有 buddy 伙伴也空闲，才能合并。

如果伙伴被占用，就不能合并，大块连续内存越来越少。

### 15.3 碎片类型

可以粗略分：

```text
外部碎片：
  空闲内存分散，无法形成大连续块。

内部碎片：
  分配粒度大于实际需求，分配到的空间内部浪费。

不可迁移页造成碎片：
  某些页不能移动，卡在大块内存中间，阻碍合并。
```

### 15.4 Linux 的反碎片思路

核心是按页面迁移类型分组。

常见 migratetype：

```text
MIGRATE_UNMOVABLE：
  不可迁移页，比如很多内核对象。

MIGRATE_RECLAIMABLE：
  可回收页，比如部分 slab、page cache。

MIGRATE_MOVABLE：
  可迁移页，比如用户匿名页、page cache。

MIGRATE_CMA：
  CMA 区域页。

MIGRATE_ISOLATE：
  内存隔离、迁移、热拔除时使用。
```

核心思想：

```text
把不能移动的页和能移动的页尽量分开，
避免一个不可移动页污染一大块连续内存。
```

### 15.5 pageblock

Linux 以 pageblock 为粒度管理迁移类型。

比如一个 pageblock 可能是：

```text
2MB
```

内核尽量让同一 pageblock 里的页具有相似迁移属性。

### 15.6 compaction

内存规整就是把可移动页搬走，腾出连续空间。

类似整理磁盘碎片：

```text
移动可迁移页
  ↓
把空闲页集中到一起
  ↓
形成高阶连续页
```

### 15.7 碎片问题常见表现

你会看到：

```text
内存看起来还有很多 free
但 order-9 / order-10 分配失败
DMA buffer 分配失败
hugepage 分配失败
CMA 分配失败
```

诊断时常看：

```bash
cat /proc/buddyinfo
cat /proc/pagetypeinfo
cat /proc/meminfo
dmesg | grep -i "page allocation failure"
```

## 16. 《slub 分配器》

这篇讲 SLUB。文章指出 buddy system 以页为单位管理物理内存，而 SLUB/SLAB 用于更细粒度的小对象分配。([Wowotech][19])

### 16.1 为什么需要 SLUB？

buddy 最小单位是 page，通常 4KB。

如果你只需要分配 64 字节：

```text
直接给一个 4KB page，浪费严重。
```

内核大量对象都很小：

```text
dentry
inode
task_struct
file
cred
sk_buff
kmalloc-32
kmalloc-64
kmalloc-128
```

所以需要小对象分配器。

### 16.2 SLUB 的基本思想

SLUB 从 buddy 申请若干 page，然后切成固定大小的小对象。

比如：

```text
一个 4KB page
  ↓
切成 64 个 64B object
```

然后对象分配释放都在这个 slab page 内进行。

### 16.3 kmem_cache

`kmem_cache` 表示一种对象缓存。

比如：

```text
dentry_cache
inode_cache
task_struct cache
kmalloc-64
kmalloc-128
```

每个 cache 管理固定大小、固定布局的对象。

创建对象 cache：

```c
kmem_cache_create()
```

分配对象：

```c
kmem_cache_alloc()
```

释放对象：

```c
kmem_cache_free()
```

### 16.4 kmalloc 和 SLUB 的关系

`kmalloc(100)` 并不是每次都从 buddy 直接拿 100 字节。

它通常会映射到一个通用 kmalloc cache：

```text
kmalloc(100) -> kmalloc-128
kmalloc(60)  -> kmalloc-64
kmalloc(1000)-> kmalloc-1024
```

所以 kmalloc 背后也是 SLUB。

### 16.5 per-CPU freelist

为了减少锁竞争，SLUB 给 CPU 准备本地 freelist。

```text
CPU0 有自己的当前 slab / freelist
CPU1 有自己的当前 slab / freelist
```

分配时优先从本 CPU freelist 拿对象：

```text
不需要全局锁
cache locality 好
速度快
```

### 16.6 partial slab

一个 slab page 有三种状态：

```text
full：
  对象都被分配了。

partial：
  有些对象被分配，有些空闲。

empty：
  对象都空闲。
```

SLUB 会维护 partial slab 列表，方便后续分配。

### 16.7 SLUB 相比 SLAB 的特点

SLAB 比较复杂，有更多队列和对象管理结构。

SLUB 设计目标是：

```text
结构更简单
减少元数据
提高可扩展性
更适合大规模 SMP
```

### 16.8 常见问题

SLUB 相关 bug 通常是：

```text
use-after-free
double free
out-of-bounds
freelist corruption
对象没有初始化
释放到错误 cache
```

这些就需要 SLUB DEBUG 或 KASAN 定位。

## 17. 《图解 SLUB》

这篇用图解释 SLUB。文章开头也强调 buddy 以页为单位管理和分配内存。([Wowotech][20])

### 17.1 一页 slab 的内部结构

概念上：

```text
struct page
  ↓
slab page
  ↓
object0 | object1 | object2 | object3 | ...
```

空闲对象通过 freelist 串起来：

```text
object2 -> object5 -> object7 -> NULL
```

分配时：

```text
从 freelist 头上摘一个
```

释放时：

```text
把对象插回 freelist
```

### 17.2 freelist 指针藏在哪里？

SLUB 常把 freelist 指针保存在空闲对象内部。

比如 object 空闲时：

```text
object memory 前几个字节保存 next 指针
```

一旦分配给用户，这块内存就变成对象内容。

这很节省元数据，但也有风险：

```text
如果越界写破坏了空闲 object 内的 next 指针，
SLUB freelist 就坏了。
```

### 17.3 CPU slab

每个 CPU 有当前正在使用的 slab：

```text
kmem_cache_cpu
  freelist
  page
  partial
```

快速路径：

```text
freelist 不空
  ↓
直接 pop 一个 object
```

慢路径：

```text
当前 freelist 空
  ↓
从 per-cpu partial / node partial 找 slab
  ↓
还没有就从 buddy 申请新 slab page
```

### 17.4 node partial

NUMA 系统下，每个 node 也有 partial 列表。

```text
kmem_cache_node
  partial list
  nr_partial
```

CPU 本地没有可用对象时，会从 node partial 获取。

### 17.5 SLUB 的性能来源

SLUB 快主要因为：

```text
固定大小对象
per-CPU freelist
少锁
对象复用
cache locality 好
元数据少
```

### 17.6 SLUB 的代价

```text
对象大小会向上取整
存在内部碎片
freelist corruption 可能造成严重后果
debug 功能会牺牲性能
```

## 18. 《SLUB DEBUG 原理》

这篇讲 SLUB DEBUG。搜索结果显示文章标签为 slub、内存管理。([Wowotech][21])

### 18.1 SLUB DEBUG 解决什么问题？

定位小对象分配相关 bug：

```text
越界写
use-after-free
double free
释放非法地址
释放到错误 cache
对象内容被踩
freelist 被破坏
```

### 18.2 red zone

red zone 是在对象前后加保护区。

比如原对象 64B：

```text
[redzone][64B object][redzone]
```

释放或检查时，如果 redzone 被改了，说明发生越界写。

### 18.3 poison

poison 是用特殊字节填充内存。

常见逻辑：

```text
分配时填充特定模式
释放时填充另一种模式
```

如果释放后对象被访问，可能读到明显异常的 poison 值。

### 18.4 store user

store user 会记录对象分配/释放调用栈。

当发现错误时，可以告诉你：

```text
这个对象在哪里分配的
在哪里释放的
谁可能二次释放
```

### 18.5 freelist 检查

SLUB freelist 一旦被破坏，后果可能非常严重。

DEBUG 会检查：

```text
freelist 指针是否落在合法 slab 内
是否对齐
是否形成环
是否指向非法对象
```

### 18.6 开启方式

常见启动参数类似：

```text
slub_debug
slub_debug=FZPU
```

大致含义：

```text
F：sanity checks
Z：red zoning
P：poisoning
U：store user
```

具体含义以目标内核文档为准。

### 18.7 代价

SLUB DEBUG 会：

```text
增加内存占用
降低分配释放速度
改变对象布局
可能影响复现时序
```

所以生产环境要谨慎开启，调试环境很有用。

## 19. 《CMA 模块学习笔记》

这篇讲 CMA，Contiguous Memory Allocator。文章明确说 CMA 是内存管理子系统中的一个模块，负责物理地址连续的内存分配；它主要位于需要连续内存块的模块和伙伴系统之间。([Wowotech][22])

### 19.1 为什么需要 CMA？

很多设备需要大块连续物理内存。

比如摄像头：

```text
1300 万像素
每像素 3B
一张图约 39MB
```

如果需要 DMA 直接写入 buffer，而设备/IOMMU 能力有限，就可能要求物理连续。

普通 buddy 长时间运行后会碎片化：

```text
总空闲内存够
但连续 32MB 不一定有
```

### 19.2 传统方案的问题

方案一：启动时永久预留大块内存。

优点：

```text
可靠
```

缺点：

```text
设备不用时内存浪费
```

方案二：使用时再从 buddy 分配。

优点：

```text
不浪费
```

缺点：

```text
运行久了可能因碎片化分配失败
```

CMA 试图兼顾两者。

### 19.3 CMA 的核心思想

启动时预留一块连续物理区域：

```text
CMA area
```

但这块区域平时可以给普通可迁移页面使用：

```text
page cache
用户匿名页
可迁移页
```

当设备需要连续内存时：

```text
把 CMA area 中的可迁移页迁走
  ↓
腾出连续物理页
  ↓
分配给设备
```

RSS 摘录也明确说，CMA 内存在驱动没有分配使用时可以被内核其他模块使用；当驱动分配 CMA 内存后，被其他模块使用的内存需要“吐出来”，形成物理地址连续的大块内存给驱动使用。([蜗窝科技][14])

### 19.4 CMA 区域如何配置？

文章提到两类方式：

```text
1. device tree reserved-memory
2. command line / kernel config
```

device tree 方式类似：

```dts
reserved-memory {
    linux,cma {
        compatible = "shared-dma-pool";
        reusable;
        size = <...>;
        linux,cma-default;
    };
};
```

RSS 摘录说明，compatible 为 `shared-dma-pool` 的 reserved-memory 节点可用于建立 global CMA area；CMA 对应 reserved memory 节点必须有 `reusable` 属性，不能有 `no-map` 属性。([蜗窝科技][14])

### 19.5 CMA、memblock、buddy 的关系

启动早期：

```text
memblock 识别并 reserve CMA 区域
```

buddy 初始化时：

```text
CMA 区域最终也进入伙伴系统
但标记为 MIGRATE_CMA
```

运行期：

```text
普通可迁移页可以从 CMA 区域分配
不可迁移页尽量不能污染 CMA 区域
```

设备申请 CMA：

```text
cma_alloc()
  ↓
隔离目标 pageblock
  ↓
迁移里面的 movable pages
  ↓
从 buddy 分配连续页
```

### 19.6 CMA 分配失败的原因

常见原因：

```text
CMA 区域太小
CMA 区域被不可迁移页污染
页面迁移失败
长期 pin 住的页阻碍迁移
设备要求过大连续块
内存压力太大
```

### 19.7 驱动视角

驱动通常不是直接裸用 `cma_alloc()`，而是通过 DMA API 间接使用：

```c
dma_alloc_coherent()
```

底层可能从 CMA 区域分配连续内存。

# 四、DMA、scatterlist、用户/内核访问

## 20. 《Dynamic DMA mapping Guide》

这篇是驱动工程师非常重要的文章。文章明确讲 DMA API 中涉及虚拟地址、物理地址、总线地址，并解释设备看到的地址和 CPU 看到的地址可能不同。([Wowotech][23])

### 20.1 CPU 地址和设备地址不同

CPU 用虚拟地址访问内存：

```text
CPU VA -> MMU -> PA
```

设备做 DMA 时不用 CPU 的 MMU。

设备看到的是：

```text
DMA address / bus address
```

有些系统中：

```text
bus address == physical address
```

但很多系统中：

```text
bus address -> IOMMU -> physical address
```

所以驱动不能把 `kmalloc()` 返回的虚拟地址直接写进设备 DMA 寄存器。

### 20.2 DMA mapping API 做什么？

比如：

```c
dma_addr_t dma = dma_map_single(dev, cpu_addr, size, DMA_TO_DEVICE);
```

它可能做几件事：

```text
把 CPU 虚拟地址转换到物理页
建立 IOMMU 映射
进行 cache clean/invalidate
返回设备可使用的 DMA 地址
```

设备 DMA 完成后：

```c
dma_unmap_single(dev, dma, size, DMA_TO_DEVICE);
```

释放映射并做必要同步。

### 20.3 两类 DMA mapping

文章明确区分两类：consistent/coherent DMA mapping 和 streaming DMA mapping。RSS 摘录中也详细描述了 consistent DMA mapping 的特点：持续使用、CPU 和 DMA controller 并行访问时不需要软件考虑 cache 影响；streaming DMA mapping 通常是一次性的，用完就 unmap。([蜗窝科技][14])

#### consistent / coherent DMA

特点：

```text
长期使用
CPU 和设备都能看到彼此更新
通常用于描述符 ring、控制结构
```

常见 API：

```c
cpu_addr = dma_alloc_coherent(dev, size, &dma_handle, gfp);
dma_free_coherent(dev, size, cpu_addr, dma_handle);
```

返回两个地址：

```text
cpu_addr：
  CPU 使用的内核虚拟地址。

dma_handle：
  设备使用的 DMA 地址。
```

#### streaming DMA

特点：

```text
一次 DMA 传输临时映射
传输前 map
传输后 unmap
```

常见 API：

```c
dma_map_single()
dma_unmap_single()
dma_map_sg()
dma_unmap_sg()
dma_sync_single_for_cpu()
dma_sync_single_for_device()
```

### 20.4 DMA 方向

方向非常重要：

```text
DMA_TO_DEVICE：
  数据从内存到设备。

DMA_FROM_DEVICE：
  数据从设备到内存。

DMA_BIDIRECTIONAL：
  双向。

DMA_NONE：
  非法/未指定。
```

RSS 摘录也强调，如果知道方向，应尽量准确指定，因为平台可能据此做 cache 操作和优化。([蜗窝科技][14])

### 20.5 coherent 也不等于不需要 barrier

即使 coherent DMA 保证 cache 一致，也不保证 CPU 指令顺序。

比如提交描述符：

```c
desc->addr = dma_addr;
wmb();
desc->valid = 1;
```

如果没有 `wmb()`，设备可能先看到 `valid = 1`，却还没看到正确的 `addr`。

RSS 摘录中也举了类似描述符 word0/word1 的例子，需要 `wmb()` 保证顺序。([蜗窝科技][14])

### 20.6 DMA mask

设备不一定能访问所有内存。

比如设备只能访问 32-bit 地址：

```text
最大 DMA 地址 0xffffffff
```

驱动需要设置：

```c
dma_set_mask()
dma_set_coherent_mask()
```

否则内核可能分配设备访问不到的内存。

## 21. 《Linux kernel scatterlist API 介绍》

这篇讲 scatterlist。RSS 摘录解释得很清楚：scatterlist 可以理解为“物理内存的散列表”，即把分散的物理内存以列表形式组织起来；常见于 MMC、Video、Audio 等需要和用户空间交互大量数据的子系统。([Wowotech][24])

### 21.1 为什么需要 scatterlist？

很多 buffer 在虚拟地址上连续，但物理地址上不连续。

比如用户空间：

```text
用户看到：
  0x400000 - 0x410000 连续

物理页可能是：
  PFN 100
  PFN 305
  PFN 88
  PFN 901
```

设备 DMA 可能需要知道每一段物理地址。

scatterlist 就是描述：

```text
第 1 段：物理地址 A，长度 len1
第 2 段：物理地址 B，长度 len2
第 3 段：物理地址 C，长度 len3
```

### 21.2 struct scatterlist 表示什么？

概念上一个 sg entry 包含：

```text
page 指针
offset
length
dma_address
dma_length
```

注意：

```text
page + offset + length
```

描述 CPU/内核视角的内存段。

DMA mapping 后：

```text
dma_address + dma_length
```

描述设备视角的 DMA 段。

### 21.3 scatterlist 和 DMA mapping

常见流程：

```c
sg_init_table(sg, nents);
sg_set_page(&sg[i], page, len, offset);

mapped_nents = dma_map_sg(dev, sg, nents, dir);

for_each_sg(sg, s, mapped_nents, i) {
    dma_addr = sg_dma_address(s);
    dma_len  = sg_dma_len(s);
}

dma_unmap_sg(dev, sg, nents, dir);
```

关键点：

```text
dma_map_sg() 返回的 mapped_nents 可能小于原始 nents。
```

因为 IOMMU 或 DMA 层可能把相邻段合并。

### 21.4 chain scatterlist

文章搜索结果提到可以把不同 scatterlist 数组链在一起，也叫 chain scatterlist。([Wowotech][24])

这用于 sg entry 很多、不想一次性分配巨大数组的场景。

### 21.5 常见用途

```text
块设备 I/O
MMC/SD/eMMC
网络
视频采集
音频
DMA engine
用户页 pin 住后给设备 DMA
```

## 22. 《copy_{to,from}_user() 的思考》

这篇讲内核和用户空间拷贝数据。搜索结果显示文章围绕 `copy_from_user()`、`copy_to_user()`，以及 ARM64 的 TTBR0_EL1/TTBR1_EL1 地址空间设计展开。([Wowotech][25])

### 22.1 为什么不能直接 memcpy 用户地址？

用户传给内核的地址可能是：

```text
非法地址
未映射地址
只读地址
跨页部分有效部分无效
恶意构造地址
并发被 munmap
触发缺页
指向内核地址
```

如果内核直接：

```c
memcpy(kernel_buf, user_ptr, len);
```

可能导致：

```text
kernel crash
安全漏洞
越权访问
page fault 无法恢复
```

所以必须使用：

```c
copy_from_user()
copy_to_user()
get_user()
put_user()
```

### 22.2 copy_from_user

```c
copy_from_user(kernel_buf, user_buf, len);
```

方向：

```text
用户空间 -> 内核空间
```

典型场景：

```text
write 系统调用
ioctl 参数
用户传配置给驱动
```

### 22.3 copy_to_user

```c
copy_to_user(user_buf, kernel_buf, len);
```

方向：

```text
内核空间 -> 用户空间
```

典型场景：

```text
read 系统调用
ioctl 返回数据
驱动把状态返回用户态
```

### 22.4 返回值容易误解

`copy_to_user()` / `copy_from_user()` 返回值通常是：

```text
未成功拷贝的字节数
```

不是成功返回 0/失败返回 -EFAULT 那种普通风格。

所以常见写法：

```c
if (copy_to_user(buf, kbuf, len))
    return -EFAULT;
```

### 22.5 access_ok

访问前需要判断用户地址范围是否合法：

```c
access_ok(user_ptr, len)
```

它不能保证后续拷贝一定成功，但可以先过滤明显非法的用户地址范围。

### 22.6 异常表机制

用户地址访问可能触发 page fault。

内核通过 exception table 机制，把某些用户访问异常修正到错误处理路径，而不是直接 panic。

粗略流程：

```text
执行用户地址 load/store
  ↓
触发异常
  ↓
异常处理查 exception table
  ↓
发现这是 copy_from_user 可恢复异常
  ↓
跳到 fixup 代码
  ↓
返回未拷贝字节数 / -EFAULT
```

### 22.7 ARM64 TTBR0/TTBR1 背景

ARM64 常见设计：

```text
TTBR0_EL1：
  用户地址空间页表

TTBR1_EL1：
  内核地址空间页表
```

内核态访问用户地址时，要保证用户地址页表可用，同时避免安全问题。这和 PAN、UAO、KPTI 等机制也有关。

# 五、/proc/meminfo、page cache、回写、回收、OOM

## 23. 《/proc/meminfo 分析（一）》

这篇逐项解释 `/proc/meminfo`。RSS 摘录里详细解释了 MemTotal、MemFree、MemAvailable、Buffers、Cached、SwapCached 等字段。([Wowotech][26])

### 23.1 MemTotal

`MemTotal` 不是物理内存条容量。

它表示：

```text
Linux 当前可以管理的物理内存总量
```

RSS 摘录说明，它对应 `totalram_pages`，表示 Linux 内核可以管理的 page frame 数量；内核自身、保留区域等会从机器配置内存里扣掉。([蜗窝科技][10])

### 23.2 MemFree

`MemFree` 表示当前完全空闲的页。

它们通常在 buddy 系统 free list 里。

但注意：

```text
MemFree 低不代表系统没内存。
```

因为 Linux 会尽量利用空闲内存做 cache。

### 23.3 MemAvailable

这比 MemFree 更重要。

它表示：

```text
不引起明显 swap / 高代价回收的情况下，系统还能给新程序用多少内存。
```

RSS 摘录说，MemAvailable 会考虑：

```text
MemFree
各 zone watermark
可回收 page cache
可回收 slab
```

并扣除不能轻易使用的部分。([蜗窝科技][10])

### 23.4 Buffers

现代内核里 buffer cache 已经和 page cache 体系统一。

`Buffers` 更多指块设备相关的 page cache，比如：

```text
块设备元数据
super block raw data
直接访问 block device 产生的缓存
```

RSS 摘录也提到，新的内核已没有独立 buffer cache，一切统一到 page cache 框架下，所谓 buffer cache 是块设备的 page cache。([蜗窝科技][10])

### 23.5 Cached

`Cached` 主要是普通文件 page cache。

读文件时：

```text
第一次读：从磁盘读到 page cache，再拷给用户
第二次读：直接从 page cache 读
```

所以 Cached 高通常是好事，不是“内存泄漏”。

### 23.6 SwapCached

SwapCached 不是普通文件缓存。

它解决的是 swap in / swap out 的同步问题。

RSS 摘录说明，swap cache 是为了解决 page frame 和 swap area 之间同步问题引入的，例如一个进程准备 swap in 时，内核回收模块可能同时试图 swap out。([蜗窝科技][10])

### 23.7 你以后看 meminfo 要这样判断

```text
只看 MemFree：容易误判
重点看 MemAvailable：更接近可用内存
看 Cached：理解 page cache 占用
看 SReclaimable/SUnreclaim：判断 slab 是否可回收
看 SwapTotal/SwapFree/SwapCached：判断 swap 压力
看 Active/Inactive：判断冷热页
看 Dirty/Writeback：判断回写压力
```

## 24. 《文件缓存回写简述》

这篇讲 page cache 的 dirty page 何时写回磁盘。搜索结果显示文章主题是“文件缓存回写”，也有搜索摘录提醒不要把 TLB flush 和 dirty page writeback 混淆；TLB 是 CPU 上的地址翻译缓存，dirty page writeback 是内存中的脏页写回磁盘。([Wowotech][27])

### 24.1 什么是 page cache？

文件读写通常不直接操作磁盘，而是经过 page cache。

写文件时：

```text
用户 write()
  ↓
数据拷到 page cache
  ↓
标记 dirty
  ↓
稍后由内核回写到磁盘
```

### 24.2 什么是 dirty page？

page cache 中的内容比磁盘新：

```text
内存中已经修改
磁盘上还没更新
```

这个 page 就是 dirty page。

### 24.3 为什么不每次 write 都立刻写磁盘？

因为磁盘 I/O 慢。

延迟回写可以：

```text
合并多次写
按更好的顺序写
减少随机 I/O
提高性能
```

但代价是：

```text
掉电可能丢失未落盘数据
回写压力可能突然变大
```

### 24.4 回写触发时机

常见触发：

```text
周期性后台回写
dirty page 超过阈值
用户调用 fsync/sync
内存回收需要回收 dirty page
文件关闭不一定立刻回写
系统 suspend / unmount
```

### 24.5 writeback 相关线程

常见角色：

```text
flush-* / wb_workfn
bdi writeback
balance_dirty_pages()
```

用户写太快时，`balance_dirty_pages()` 会让写进程自己慢下来，避免 dirty page 无限增长。

### 24.6 和 page reclaim 的关系

如果内存紧张，回收遇到 dirty page：

```text
干净 page cache：
  可以直接丢弃

脏 page cache：
  必须先写回磁盘，再回收
```

所以 dirty page 多会让回收变慢，甚至导致 direct reclaim 卡顿。

## 25. 《页面回收的基本概念》

这篇是 page reclaim 的入门。RSS 摘录明确说，page frame reclaiming 是保证系统空闲内存在指定水位之上；不能等内存完全耗尽再回收，因为回收本身也可能需要内存。([Wowotech][28])

### 25.1 为什么需要页面回收？

Linux 会尽量使用内存：

```text
空闲内存不用也是浪费
```

所以它会把很多内存用于：

```text
page cache
slab cache
匿名页
buffer
```

当空闲内存低于水位时，需要回收一些 page frame 给 buddy。

### 25.2 watermark

每个 zone 有水位：

```text
WMARK_MIN
WMARK_LOW
WMARK_HIGH
```

大概理解：

```text
高于 HIGH：
  很安全

低于 LOW：
  唤醒 kswapd 后台回收

低于 MIN：
  分配路径可能 direct reclaim，甚至失败
```

### 25.3 为什么不能等 free=0 再回收？

RSS 摘录说得很关键：页面回收过程中也会消耗内存，比如把页面写到磁盘可能需要分配 buffer head 等结构。([蜗窝科技][10])

所以必须提前维持一部分空闲内存。

### 25.4 哪些页面可以回收？

大致分：

```text
1. 不可回收页
   内核关键数据、mlock 页、正在使用的页。

2. 可交换页 swappable
   用户匿名页，可以写到 swap。

3. 可同步页 syncable
   文件 page cache，有后备文件。

4. 可直接释放页
   某些 cache/slab，或干净 page cache。
```

RSS 摘录也区分了 swappable 和 syncable：swappable 的最终地点是内存，磁盘只是扩展内存；syncable 的最终地点是磁盘，内存只是缓存。([蜗窝科技][10])

### 25.5 回收优先级原则

RSS 摘录给了几个重要规则：

```text
尽量不要修改页表
所有未 mlock 的用户页理论上可回收
共享页要清除所有 PTE 才能回收
不要回收最近使用过的页
尽量先回收不需要磁盘 I/O 的页
```

这些原则非常重要。([蜗窝科技][10])

### 25.6 active/inactive LRU

Linux 用 LRU 思想区分冷热页：

```text
active list：
  最近访问过，先不回收。

inactive list：
  较冷，优先回收。
```

文件页和匿名页又分开：

```text
active_file
inactive_file
active_anon
inactive_anon
```

### 25.7 kswapd 和 direct reclaim

```text
kswapd：
  后台回收线程，尽量提前回收。

direct reclaim：
  分配内存的进程自己进入回收路径。
```

direct reclaim 对性能影响更明显，因为业务进程会卡在内存分配路径。

## 26. 《linux kernel 内存回收机制》

这篇更深入讲 Linux 内存回收机制。搜索结果显示文章主题是“无论计算机上有多少内存都是不够的，因而 linux kernel 需要回收一些很少使用的内存”。([Wowotech][29])

### 26.1 内存回收主线

典型路径：

```text
alloc_pages()
  ↓
发现 zone free pages 低于 watermark
  ↓
尝试快速分配失败
  ↓
唤醒 kswapd 或进入 direct reclaim
  ↓
扫描 LRU
  ↓
回收 file page / anon page / slab
  ↓
释放 page 回 buddy
```

### 26.2 文件页回收

文件页有后备文件。

如果是 clean：

```text
直接丢弃 page cache
以后再访问从磁盘读回来
```

如果是 dirty：

```text
先 writeback
写回完成后再回收
```

### 26.3 匿名页回收

匿名页没有文件后备。

比如：

```text
堆
栈
匿名 mmap
```

如果要回收，需要：

```text
写到 swap
  ↓
PTE 改成 swap entry
  ↓
释放物理页
```

下次访问触发缺页：

```text
从 swap 读回内存
```

### 26.4 slab 回收

部分 slab 是 reclaimable 的，比如：

```text
dentry cache
inode cache
```

内核可以通过 shrinker 回收。

每个子系统可以注册 shrinker：

```text
告诉 VM：我这里有多少对象可回收
VM：请你释放一部分
```

### 26.5 页面隔离与回收

回收时不是看到一个页就直接释放。

通常要：

```text
从 LRU 隔离一批页
  ↓
检查引用、映射、dirty、writeback、mlock 等状态
  ↓
能释放就释放
  ↓
不能释放就放回 LRU
```

### 26.6 回收困难的页面

以下页面难回收：

```text
正在 writeback 的 dirty page
被 mlock 的页
被 GUP pin 住的页
频繁访问的 hot page
共享映射很多的 page
不可迁移内核页
```

### 26.7 回收与系统卡顿

direct reclaim 可能导致：

```text
进程分配内存变慢
系统抖动
I/O 激增
kswapd CPU 占用高
延迟毛刺
```

如果内存长期紧张，可能出现：

```text
thrashing
```

也就是大量时间都在换入换出，真正业务执行很少。

## 27. 《逆向映射的演进》

这篇讲 reverse mapping，搜索结果和 RSS 摘录都说明文章通过历史演进理解 Linux 逆向映射机制。([Wowotech][30])

### 27.1 正向映射是什么？

正常页表是：

```text
进程虚拟地址
  ↓
PTE
  ↓
物理页
```

也就是：

```text
VA -> PA/page
```

### 27.2 逆向映射是什么？

逆向映射要回答：

```text
给定一个物理 page，
哪些进程的哪些虚拟地址映射了它？
```

也就是：

```text
page -> 所有映射它的 VMA/PTE
```

### 27.3 为什么需要 rmap？

页面回收时，内核想释放一个物理页。

但这个页可能被多个进程映射：

```text
进程 A 的 VA 0x400000
进程 B 的 VA 0x500000
进程 C 的 VA 0x600000
```

如果要回收这个 page，必须找到并修改所有 PTE：

```text
清除 present
设置 swap entry
或者解除映射
```

没有 rmap，就可能要扫描所有进程的所有页表，代价极高。

### 27.4 匿名页和文件页的 rmap 不同

匿名页：

```text
没有文件后备
常通过 anon_vma 建立反向关系
```

文件页：

```text
有 address_space
可以通过文件映射关系找到相关 VMA
```

### 27.5 rmap 在哪些场景用？

```text
page reclaim
swap out
page migration
NUMA balancing
KSM
COW
mprotect
mlock 判断
内存热拔除
```

### 27.6 rmap 的代价

rmap 提升回收效率，但也引入成本：

```text
每次建立/解除映射要维护 rmap
fork/COW 复杂度增加
VMA、anon_vma 关系更复杂
锁竞争可能增加
```

### 27.7 直观类比

正向映射像：

```text
人 -> 房间号
```

逆向映射像：

```text
房间号 -> 住了哪些人
```

如果酒店要维修某个房间，只知道“人到房间”不够，还要能查“这个房间有哪些住户”。

## 28. 《Linux vm 运行参数之一：overcommit 相关参数》

这篇讲 overcommit。搜索结果显示文章讨论 overcommit 相关参数，并提到系统可能“拆东墙补西墙”，还可能涉及 dirty page flush 等磁盘 I/O。([Wowotech][31])

### 28.1 什么是 overcommit？

用户申请虚拟内存时，Linux 不一定立刻分配真实物理内存。

比如：

```c
p = malloc(10GB);
```

可能成功，即使机器只有 8GB 内存。

因为：

```text
申请虚拟地址 ≠ 立即占用物理内存
```

只有真正写入页面时，才通过缺页分配物理页。

### 28.2 为什么允许 overcommit？

因为很多程序申请了很多内存，但实际不用那么多。

比如：

```text
JVM/数据库预留大地址空间
进程栈预留很大但实际只用少量
mmap 大文件但只访问部分
fork 后 COW 共享父进程页面
```

如果申请时就严格按最大值扣物理内存，会浪费。

### 28.3 overcommit 的风险

如果大家后来真的都用起来：

```text
实际物理内存 + swap 不够
```

就会触发：

```text
direct reclaim
swap
OOM killer
```

### 28.4 vm.overcommit_memory

常见模式：

```text
0：启发式 overcommit
1：总是允许 overcommit
2：严格限制 commit
```

模式 0：

```text
内核根据启发式判断是否允许申请。
```

模式 1：

```text
尽量都允许，适合某些高性能/科学计算场景，但风险高。
```

模式 2：

```text
严格限制已承诺内存不能超过 CommitLimit。
```

### 28.5 CommitLimit 和 Committed_AS

常见字段：

```text
CommitLimit：
  系统允许承诺的最大虚拟内存量。

Committed_AS：
  当前已经承诺给进程的虚拟内存量。
```

粗略公式：

```text
CommitLimit ≈ SwapTotal + RAM * overcommit_ratio / 100
```

具体和内核版本、配置有关。

### 28.6 overcommit 和 fork

fork 时子进程理论上复制父进程地址空间。

但实际使用 COW：

```text
父子共享物理页
写时才复制
```

overcommit 需要估计未来 COW 可能消耗的内存。

### 28.7 工作中怎么理解

如果你看到：

```text
malloc 成功
但运行一段时间后被 OOM kill
```

可能就是 overcommit 允许申请，但后续实际触碰页面时物理内存不够。

## 29. 《Linux vm 运行参数之二：OOM 相关参数》

这篇讲 OOM killer 和相关参数。搜索结果显示文章主要讲 OOM 相关参数，并先解释什么是 OOM。([Wowotech][32])

### 29.1 什么是 OOM？

OOM = Out Of Memory。

当内核已经尝试：

```text
回收 page cache
回收 slab
swap out
direct reclaim
compact
```

仍然无法满足内存分配，就可能触发 OOM。

### 29.2 为什么要杀进程？

如果不杀进程：

```text
内核无法分配必要内存
系统可能死锁或崩溃
```

杀掉一个或多个进程可以释放内存，让系统继续运行。

### 29.3 OOM killer 选择谁？

内核会给进程计算 badness 分数。

影响因素包括：

```text
占用内存大小
oom_score_adj
是否特权进程
是否共享大量内存
是否不可杀
```

用户可看：

```bash
cat /proc/<pid>/oom_score
cat /proc/<pid>/oom_score_adj
```

`oom_score_adj` 范围通常是：

```text
-1000 到 1000
```

```text
-1000：
  尽量不被 OOM kill。

1000：
  更容易被 kill。
```

### 29.4 panic_on_oom

参数：

```text
vm.panic_on_oom
```

可能控制 OOM 时是否 panic。

嵌入式/高可靠系统有时希望：

```text
OOM 后直接 panic 重启
```

服务器通常更希望：

```text
杀掉坏进程，系统继续运行
```

### 29.5 oom_kill_allocating_task

这个参数控制是否优先杀正在分配失败的任务。

直觉上：

```text
谁申请失败杀谁
```

但实际不一定合理，因为申请失败者可能不是内存消耗大户。

### 29.6 cgroup OOM

如果使用 memory cgroup：

```text
整个系统还有内存
但某个 cgroup 超过限制
```

会触发 cgroup 内部 OOM，只在该 cgroup 里选择进程杀。

这对容器很常见。

### 29.7 OOM 日志怎么看？

重点看：

```text
触发 OOM 的进程
gfp_mask
order
内存节点/zone 状态
Mem-Info
各进程内存占用表
被 kill 的进程
oom_score_adj
```

## 30. 《zRAM 内存压缩技术原理与应用》

这篇讲 zRAM。搜索结果显示文章讲 zRAM 内存压缩技术原理与应用，并提到可在 `/proc/meminfo` 中查看相关信息。([Wowotech][33])

### 30.1 zRAM 是什么？

zRAM 是一种压缩块设备，通常用作 swap。

普通 swap：

```text
内存页 -> 写到磁盘/SSD
```

zRAM：

```text
内存页 -> 压缩 -> 存在内存里的 zram 设备
```

它用 CPU 换内存空间。

### 30.2 为什么 zRAM 有用？

比如一个 4KB 页面，压缩后只有 1KB。

那么：

```text
原本 4KB 内存
压缩后 1KB zRAM 存储
节省约 3KB
```

适合：

```text
手机
嵌入式设备
内存较小系统
不希望频繁写 flash 的系统
```

### 30.3 zRAM 和 swap 的关系

zRAM 通常注册成 block device：

```text
/dev/zram0
```

然后：

```bash
mkswap /dev/zram0
swapon /dev/zram0
```

内核把它当 swap 设备用。

### 30.4 zRAM 优点

```text
比磁盘 swap 快
减少 flash 写入
提升小内存系统可用性
压缩比好时相当于变相增加内存
```

### 30.5 zRAM 缺点

```text
消耗 CPU
压缩比不稳定
仍然占用内存
内存极度紧张时也可能无力
不适合替代真正内存扩容
```

### 30.6 zRAM 关键指标

常看：

```bash
cat /sys/block/zram0/mm_stat
cat /proc/meminfo
zramctl
```

关注：

```text
原始数据大小
压缩后大小
内存占用
压缩算法
swap 使用量
```

# 六、KASAN、调试、安全访问

## 31. 《KASAN 实现原理》

这篇讲 Kernel Address Sanitizer。搜索结果显示它是 Linux kernel 的快速内存错误检测工具。([Wowotech][34])

### 31.1 KASAN 能发现什么？

常见错误：

```text
out-of-bounds
use-after-free
use-after-scope
double free
invalid free
slab object 越界
全局变量越界
栈越界，取决于模式和配置
```

### 31.2 KASAN 的核心思想：shadow memory

KASAN 给内核内存配一块 shadow memory。

常见比例：

```text
1 byte shadow 描述 8 bytes 真实内存
```

概念：

```text
真实内存：
  addr

shadow 内存：
  shadow = addr >> 3 + offset
```

shadow byte 表示对应真实内存是否可访问。

### 31.3 编译器插桩

KASAN 需要编译器配合。

每次内核代码访问内存：

```c
*p = value;
x = *p;
```

编译器会插入检查：

```text
访问前检查 shadow memory
如果非法，报告 KASAN bug
```

### 31.4 slab 场景

对象分配时：

```text
object 标记为可访问
redzone 标记为不可访问
```

对象释放时：

```text
整个 object 标记为不可访问
```

如果释放后继续访问：

```text
shadow 检查失败
报告 use-after-free
```

### 31.5 KASAN 报告怎么看？

一般包含：

```text
错误类型
访问地址
读/写
访问大小
当前进程
调用栈
分配调用栈
释放调用栈
对象所属 cache
对象大小
redzone 信息
shadow bytes
```

### 31.6 KASAN 的代价

```text
内存占用明显增加
性能下降
需要特殊编译内核
不一定适合生产环境长期打开
```

但对内核内存 bug 定位非常有效。

## 32. 《ARM64 Kernel Image Mapping 的变化》与《KASLR》的组合理解

这两篇最好一起看。

它们共同回答：

```text
内核镜像到底映射在哪里？
为什么有时是大块映射，有时是细粒度映射？
为什么随机化后调试地址会变？
```

核心关系：

```text
Kernel Image Mapping：
  解决“内核怎么被映射和执行”。

KASLR：
  解决“内核映射地址不要固定，提升攻击难度”。
```

```text
修改映射后：TLB flush
```

## 33. 《文件缓存回写简述》与《页面回收》的组合理解

这两块必须连起来学。

文件页有三种状态：

```text
不在内存：
  访问时从磁盘读。

在 page cache 且 clean：
  可以直接回收。

在 page cache 且 dirty：
  必须先 writeback，再回收。
```

所以内存压力下：

```text
clean file page 最好回收
dirty file page 回收成本高
anonymous page 需要 swap
mlock/pinned page 很难回收
```

这也是为什么系统中：

```text
Dirty 很高
Writeback 很高
kswapd 很忙
direct reclaim 很多
```

时，应用延迟可能明显变差。

## 34. 《overcommit》《OOM》《zRAM》的组合理解

这三篇是一条链：

```text
overcommit：
  申请虚拟内存时先允许你承诺。

实际触碰页面：
  需要真实物理页。

内存不够：
  reclaim / swap / zRAM。

还不够：
  OOM killer。
```

zRAM 的位置：

```text
它不是避免 OOM 的绝对方案，
而是让一部分匿名页压缩后留在内存中，
推迟或减少真正磁盘 swap / OOM。
```

# 九、这组文章最终要形成的理解

如果你没时间看原文，至少要把下面这段理解透：

Linux 内核把物理内存切成一个个 page frame，并用 `struct page` 描述。启动早期，内核还没有 buddy 和 SLUB，只能用 memblock 记录哪些物理内存可用、哪些被保留；之后建立 identity mapping、kernel image mapping、linear mapping、fixmap 等页表映射，打开 MMU 并进入正式内核虚拟地址空间。正常运行后，整页分配主要靠 buddy，小对象分配靠 SLUB，需要连续物理内存时靠 CMA 辅助。用户进程看到的是虚拟地址，真实物理页通过页表映射；CPU 为了加速有 TLB 和 Cache，因此修改页表要 TLB flush，DMA 访问内存要处理设备地址、IOMMU 和 cache 一致性。内存紧张时，内核通过 page reclaim 回收 page cache、swap out 匿名页、调用 shrinker 回收 slab；如果仍无法满足分配，就触发 OOM killer。调试上，`/proc/meminfo` 帮你看系统内存状态，SLUB DEBUG 和 KASAN 帮你抓越界、释放后使用、重复释放等内核内存 bug。

# 0. 总体主线：这批文档到底在讲什么？

这批文档可以归纳成一个核心问题：

> **Linux 如何把“程序”抽象成“可调度任务”，如何创建、识别、管理这些任务，然后在单核/多核系统中公平、高效地选择任务运行？**

你可以把内核进程管理和调度器想成一个“大型工厂排班系统”：

| 内核概念           | 类比                 |
| -------------- | ------------------ |
| task_struct    | 员工档案               |
| pid / tgid     | 员工编号 / 团队编号        |
| fork / clone   | 招一个新员工 / 复制一个工位执行流 |
| exec           | 员工换成另一套工作内容        |
| scheduler      | 排班主管               |
| runqueue       | 每个 CPU 的待运行队列      |
| sched_class    | 不同等级的排班规则          |
| CFS/EEVDF      | 普通员工的公平排班算法        |
| RT             | 紧急任务排班             |
| Deadline       | 有明确截止时间的任务排班       |
| PELT           | 统计员工最近忙不忙          |
| load balance   | 多个车间之间调人           |
| context switch | 当前员工下班，另一个员工上工位    |
| TLB switch     | 换员工时，换他自己的“地址空间缓存” |

这批文档不是孤立的，它们大概分成这些层次：

```text
用户态接口
  nice / sched_setscheduler / sched_setaffinity / fork / clone / exec
        ↓
进程表示
  task_struct / pid / tgid / cred / signal_struct / files_struct / mm_struct
        ↓
调度核心
  schedule / pick_next_task / context_switch
        ↓
调度类
  stop / deadline / rt / fair / idle
        ↓
CFS/EEVDF 普通任务调度
  sched_entity / cfs_rq / vruntime / lag / deadline
        ↓
负载统计
  PELT / load_avg / util_avg / runnable_avg
        ↓
多核均衡
  sched_domain / sched_group / load_balance / select_task_rq
        ↓
频率调节
  schedutil / cpufreq / util -> frequency
```

# 1. 进程、线程和 task_struct

Linux 里，用户常说的“进程”和“线程”，在内核里本质上都是 `task_struct`。也就是说，调度器真正调度的是 **task**，不是用户态抽象上的“进程”。

## 1.1 task_struct 是什么？

`task_struct` 是 Linux 中描述一个执行实体的核心结构。它里面保存了这个任务几乎所有重要信息：

```c
struct task_struct {
    volatile long state;
    void *stack;
    struct mm_struct *mm;
    struct mm_struct *active_mm;

    pid_t pid;
    pid_t tgid;

    struct task_struct *parent;
    struct list_head children;
    struct list_head sibling;

    const struct cred *cred;

    struct files_struct *files;
    struct fs_struct *fs;
    struct signal_struct *signal;
    struct sighand_struct *sighand;

    struct sched_entity se;
    struct sched_rt_entity rt;
    struct sched_dl_entity dl;

    int prio;
    int static_prio;
    int normal_prio;
    unsigned int policy;
};
```

你不需要一开始记住每个字段，但要理解这些分组：

| 字段类型 | 作用                          |
| ---- | --------------------------- |
| 身份信息 | pid、tgid、parent、real_parent |
| 地址空间 | mm、active_mm                |
| 文件资源 | files、fs                    |
| 信号资源 | signal、sighand              |
| 权限资源 | cred                        |
| 调度资源 | se、rt、dl、prio、policy        |
| 状态资源 | state、exit_state、flags      |

## 1.2 进程和线程的本质区别

在传统操作系统教材里：

* 进程：资源分配单位。
* 线程：CPU 调度单位。

在 Linux 里更准确的说法是：

> **task 是调度单位；多个 task 可以通过 clone 共享地址空间、文件表、信号处理等资源，于是它们表现为线程。**

也就是说，Linux 并不是单独搞一套“线程对象”，而是用 `clone()` 的不同 flag 决定两个 task 共享什么资源。

例如：

```c
clone(CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD, ...)
```

这些 flag 大致表示：

| flag          | 含义                    |
| ------------- | --------------------- |
| CLONE_VM      | 共享地址空间                |
| CLONE_FS      | 共享当前工作目录、root 等 fs 信息 |
| CLONE_FILES   | 共享文件描述符表              |
| CLONE_SIGHAND | 共享信号处理函数              |
| CLONE_THREAD  | 加入同一个线程组              |

所以：

```text
fork 创建的子进程：
  多数资源复制或 copy-on-write，不共享线程组

pthread_create 创建的线程：
  共享 mm/files/sighand/signal 等资源，属于同一线程组
```

# 2. PID、TGID、PGID、SID：进程标识体系

wowotech 的 process identification 页面明确提到，多线程进程中，第一个线程，也就是 group leader，它的 `pid` 等于 `tgid`；后续线程都有自己的 `pid`，但共享同一个 `tgid`，这个 `tgid` 更接近用户态传统意义上的“进程 ID”。([Wowotech][2])

## 2.1 PID：内核调度实体 ID

每个 task 都有自己的 `pid`。

比如一个进程有 3 个线程：

```text
主线程 pid = 1000, tgid = 1000
线程1 pid = 1001, tgid = 1000
线程2 pid = 1002, tgid = 1000
```

从内核视角看，这是 3 个 task。

从用户态 `ps` 默认视角看，它们可能表现为 1 个进程。

## 2.2 TGID：线程组 ID

TGID 可以理解为“用户态进程 ID”。

为什么这么设计？

因为 POSIX 线程模型里，一个多线程程序通常被视为一个进程。比如：

```bash
kill 1000
```

通常你希望杀掉整个进程，而不是只杀某一个线程。

## 2.3 PGID：进程组 ID

PGID 主要服务于 shell 作业控制。

例如：

```bash
sleep 100 | cat
```

`sleep` 和 `cat` 是两个进程，但 shell 会把它们放到同一个进程组里。这样你按 `Ctrl+C` 时，终端可以把 `SIGINT` 发送给整个前台进程组，而不是只发给其中一个进程。

## 2.4 SID：会话 ID

SID 是 session ID，和登录会话、控制终端有关。

典型关系：

```text
Session
  ├── foreground process group
  │       ├── process A
  │       └── process B
  └── background process group
          ├── process C
          └── process D
```

shell 登录后会创建 session。一个 session 可以有一个 controlling terminal。前台进程组可以读终端，后台进程组如果读终端，可能收到 `SIGTTIN`。

# 3. process credentials：进程权限身份

`process credentials` 文档讲的是：**内核如何判断一个进程有没有权限访问某个资源**。wowotech 该页摘要指出，权限检查主要依赖 effective user ID 和 effective group ID。

Linux 里不是只有一个 UID/GID，而是有多套身份：

| 身份                   | 含义                  |
| -------------------- | ------------------- |
| real uid             | 真实用户 ID：这个进程是谁启动的   |
| effective uid        | 有效用户 ID：权限检查时主要看谁   |
| saved uid            | 保存的 UID：用于临时切换权限后恢复 |
| fs uid               | 文件系统权限检查用的 UID      |
| real gid             | 真实组 ID              |
| effective gid        | 有效组 ID              |
| supplementary groups | 附加组                 |

## 3.1 为什么需要 effective uid？

比如 `/usr/bin/passwd` 允许普通用户修改自己的密码，但 `/etc/shadow` 只有 root 能写。

那普通用户执行 `passwd` 怎么修改 `/etc/shadow`？

因为 `passwd` 是 setuid root 程序：

```bash
-rwsr-xr-x root root /usr/bin/passwd
```

普通用户启动它时：

```text
real uid      = 普通用户
effective uid = root
```

所以内核检查权限时看到 effective uid 是 root，就允许它写某些需要 root 权限的文件。

## 3.2 cred 在内核里的意义

`task_struct` 里通常有：

```c
const struct cred *real_cred;
const struct cred *cred;
```

可以简单理解：

* `real_cred`：真实身份。
* `cred`：当前权限检查使用的身份。

内核中很多安全检查会走 `current_cred()`、`current_fsuid()`、capability 检查等路径。

# 4. 进程创建：fork、clone、exec

Process Creation 两篇文章主要围绕 Linux 如何创建进程展开。搜索摘要显示，这些文章涉及 fork/exec、进程描述符、credentials 等初始化过程。

## 4.1 fork 的本质

用户态调用：

```c
pid_t pid = fork();
```

大致路径：

```text
用户态 fork()
  -> glibc wrapper
  -> clone/fork syscall
  -> kernel_clone()
  -> copy_process()
  -> wake_up_new_task()
```

核心函数通常是：

```c
copy_process()
```

它负责创建新的 `task_struct`，并根据 flags 决定复制或共享哪些资源。

## 4.2 fork 后父子进程有什么关系？

fork 后：

```text
父进程：
  返回子进程 pid

子进程：
  返回 0
```

这也是为什么经典代码这样写：

```c
pid_t pid = fork();

if (pid == 0) {
    // child
} else {
    // parent
}
```

内核角度：

```text
父子进程有不同 task_struct
父子进程有不同 pid
父子进程最初看到的地址空间内容相同
但实际物理页通常通过 copy-on-write 延迟复制
```

## 4.3 copy-on-write：为什么 fork 不会立即复制全部内存？

假设父进程有 1GB 内存，如果 fork 马上复制 1GB，会非常慢。

所以 Linux 使用 COW：

```text
fork 后：
  父子进程页表都指向同一批物理页
  页表权限标记为只读

如果父或子写某一页：
  触发 page fault
  内核复制该页
  写入方获得自己的私有页
```

好处：

```text
fork + exec 场景非常快
```

因为很多时候 fork 后马上 exec，新进程会替换地址空间，如果 fork 时已经复制了全部内存，那就浪费了。

## 4.4 exec 的本质

`execve()` 不是创建新进程，而是：

> **在当前进程壳子里换一个新程序。**

也就是说：

```text
pid 通常不变
task_struct 仍然是当前 task
但是 mm_struct 会被替换
用户态代码、数据、堆、栈都会变成新程序的
```

比如 shell 执行：

```bash
ls
```

大致是：

```text
shell fork 出子进程
  子进程 execve("/bin/ls")
  父进程 wait 子进程退出
```

所以 shell 自己不会变成 `ls`，因为 exec 是在子进程里做的。

## 4.5 fork、vfork、clone 的区别

| 接口             | 含义                                      |
| -------------- | --------------------------------------- |
| fork           | 创建新进程，资源大多复制/COW                        |
| vfork          | 子进程 exec/exit 前，父进程阻塞，子进程共享父进程地址空间，风险更高 |
| clone          | 更底层，可精细控制共享哪些资源                         |
| pthread_create | 底层通常基于 clone 创建线程                       |

你可以这样记：

```text
fork  偏“创建进程”
clone 偏“创建 task，并指定共享关系”
exec  偏“换程序”
```

# 5. TTY、Session、进程组、作业控制

`process-tty-basic.html` 讲的是进程和终端的基础关系。这部分容易被忽视，但理解 shell、daemon、Ctrl+C、后台任务很重要。

## 5.1 控制终端 controlling terminal

当你登录一个 shell 时，大致形成：

```text
terminal
  -> session
      -> shell
      -> foreground process group
      -> background process groups
```

控制终端负责把输入事件、特殊按键转成信号。

例如：

| 按键     | 信号      |
| ------ | ------- |
| Ctrl+C | SIGINT  |
| Ctrl+\ | SIGQUIT |
| Ctrl+Z | SIGTSTP |

这些信号通常发给 **前台进程组**。

## 5.2 前台进程组和后台进程组

例如：

```bash
sleep 100
```

`sleep` 在前台进程组，终端输入控制权给它。

如果：

```bash
sleep 100 &
```

它在后台进程组。

后台进程如果试图读终端，可能被内核发送 `SIGTTIN` 暂停。

## 5.3 daemon 为什么要 setsid？

守护进程通常不应该依赖登录终端，所以会：

```c
setsid();
```

效果：

```text
创建新 session
脱离原控制终端
成为 session leader
成为 process group leader
```

经典 daemon 化步骤里常见：

```text
fork
父进程退出
子进程 setsid
再 fork
chdir("/")
关闭 stdin/stdout/stderr
```

# 6. 调度器用户态 API

`scheduler-API` 文档讲 Linux 调度器暴露给用户态的接口。搜索结果显示该文档主题就是调度器用户空间接口。

常见 API：

| API                     | 作用                           |
| ----------------------- | ---------------------------- |
| nice()                  | 调整当前进程 nice 值                |
| getpriority/setpriority | 获取/设置进程优先级                   |
| sched_setscheduler      | 设置调度策略和实时优先级                 |
| sched_getscheduler      | 获取调度策略                       |
| sched_setparam          | 设置调度参数                       |
| sched_setaffinity       | 设置 CPU 亲和性                   |
| sched_getaffinity       | 获取 CPU 亲和性                   |
| sched_yield             | 主动让出 CPU                     |
| sched_setattr           | 设置更复杂的调度属性，比如 SCHED_DEADLINE |

## 6.1 nice

```bash
nice -n 10 command
```

nice 值范围通常是：

```text
-20 ~ +19
```

含义：

```text
nice 越小，优先级越高
nice 越大，越“谦让”
```

但注意：

> nice 主要影响普通 CFS/fair 任务，不会让普通任务压过 RT/DL 任务。

## 6.2 sched_setscheduler

可以设置：

```c
SCHED_NORMAL
SCHED_BATCH
SCHED_IDLE
SCHED_FIFO
SCHED_RR
SCHED_DEADLINE
```

大致区别：

| 策略             | 说明              |
| -------------- | --------------- |
| SCHED_NORMAL   | 普通 CFS/EEVDF 任务 |
| SCHED_BATCH    | 批处理任务，交互性要求低    |
| SCHED_IDLE     | 极低优先级普通任务       |
| SCHED_FIFO     | 实时 FIFO         |
| SCHED_RR       | 实时轮转            |
| SCHED_DEADLINE | deadline 调度     |

## 6.3 sched_setaffinity

这个你工作中经常会用：

```bash
taskset -c 1-3 ./a.out
```

内核里会影响 task 的 CPU allowed mask：

```text
p->cpus_ptr
p->nr_cpus_allowed
```

调度器选 CPU 时必须尊重 affinity。你之前分析 sched_ext migration disabled 任务不能迁移，本质上也和 “任务能不能在目标 CPU 上运行” 这类约束有关。

# 7. 优先级体系：nice、prio、weight、调度类

`process-priority` 文档讲 nice value、scheduling priority，以及内核里的 static priority、实时优先级、归一化优先级等概念。

Linux 优先级最容易混乱，因为有好几个不同字段。

## 7.1 用户态 nice 和内核优先级

用户看到：

```text
nice: -20 ~ +19
```

内核普通任务优先级通常映射到：

```text
static_prio: 100 ~ 139
```

其中：

```text
nice -20 -> static_prio 100
nice 0   -> static_prio 120
nice +19 -> static_prio 139
```

公式大致是：

```c
static_prio = NICE_TO_PRIO(nice);
```

## 7.2 为什么 nice 会变成 weight？

CFS 不直接用“优先级数字”决定谁先跑，而是用 weight 决定 CPU 份额。

比如：

```text
nice 0   weight = 1024
nice 5   weight 更小
nice -5  weight 更大
```

如果两个任务：

```text
A weight = 1024
B weight = 1024
```

它们大约各拿 50% CPU。

如果：

```text
A weight = 1024
B weight = 512
```

A 大约拿 2/3，B 大约拿 1/3。

## 7.3 prio、static_prio、normal_prio、rt_priority

| 字段          | 含义              |
| ----------- | --------------- |
| static_prio | nice 映射来的静态优先级  |
| normal_prio | 根据调度策略计算出的普通优先级 |
| prio        | 调度器当前实际使用的优先级   |
| rt_priority | RT 任务实时优先级      |
| policy      | 调度策略            |

为什么有 `prio` 和 `normal_prio`？

因为有些情况下任务可能临时提升优先级，比如 priority inheritance。`normal_prio` 是“正常情况下的优先级”，`prio` 是“当前实际优先级”。

# 8. 调度类 sched_class：Linux 不是只有 CFS

Linux 调度器是分层的，不同策略由不同调度类处理。

大致顺序：

```text
stop_sched_class
  > dl_sched_class
  > rt_sched_class
  > fair_sched_class
  > idle_sched_class
```

调度器选任务时，大致是：

```text
从最高优先级调度类开始问：
  你有没有可运行任务？
  有就选你的
  没有就问下一个调度类
```

## 8.1 stop class

最高优先级，用于 CPU stop 任务，比如 CPU hotplug、migration 等特殊场景。

## 8.2 deadline class

处理 `SCHED_DEADLINE` 任务。

它根据：

```text
runtime
deadline
period
```

来调度。

## 8.3 rt class

处理实时任务：

```text
SCHED_FIFO
SCHED_RR
```

RT 任务优先级高于普通任务。只要有可运行 RT 任务，普通 CFS 任务通常没机会运行，除非 RT throttling 等机制介入。

## 8.4 fair class

处理普通任务：

```text
SCHED_NORMAL
SCHED_BATCH
SCHED_IDLE
```

传统上叫 CFS，现在新内核中普通调度核心已经逐步转向 EEVDF，但仍在 fair class 里。

## 8.5 idle class

每个 CPU 的 idle task 在没有其他任务可运行时运行。

# 9. 调度器历史：为什么会从 O(n) 到 O(1) 到 CFS/EEVDF？

`scheduler-history` 文档讲 Linux 调度器演进。理解历史很重要，因为很多设计都是为了解决前一代的问题。

## 9.1 早期 O(n) 调度器

早期调度器选择任务时可能扫描所有 runnable task：

```text
任务越多，选择成本越高
```

复杂度 O(n)。

缺点：

```text
大量任务时调度开销大
可扩展性差
```

## 9.2 O(1) 调度器

O(1) 调度器通过 active/expired array 和 bitmap 实现常数时间选任务。

优点：

```text
选任务快
适合大量任务
```

缺点：

```text
交互性 heuristics 复杂
公平性不够优雅
调参数困难
```

## 9.3 CFS

CFS 抛弃固定时间片和复杂交互性 heuristic，转向：

```text
虚拟运行时间 vruntime
红黑树
理想公平 CPU 模型
```

官方 CFS 文档说，CFS 的 80% 设计可以概括为：在真实硬件上模拟“理想、精确的多任务 CPU”。([内核官网][1])

## 9.4 EEVDF

EEVDF 引入：

```text
lag
virtual deadline
eligible entity
```

它选择：

```text
lag >= 0 的任务中 virtual deadline 最早的任务
```

这对延迟敏感任务更友好。官方 EEVDF 文档明确说，它会选择 lag 大于等于 0 的任务，并为每个任务计算 virtual deadline，然后选择 virtual deadline 最早的任务运行。([内核文档][3])

# 10. CFS 核心：vruntime、公平性、红黑树

wowotech CFS 系列包括基本原理、源码解析、组调度、PELT、带宽控制和总结，其中源码解析文章提到 `cfs_rq` 是 CFS 就绪队列，`curr` 指向当前 CPU 上运行 task 的调度实体，`update_curr()` 是重要函数，会在很多地方更新当前任务运行时间。([Wowotech][4])

## 10.1 CFS 要解决什么问题？

普通任务调度要解决：

```text
多个普通任务都想运行，CPU 只有一个，怎么公平分配？
```

最直观的公平是：

```text
有 N 个同权重任务，每个任务拿 1/N CPU
```

但真实系统里任务有不同 nice，不同权重，所以公平变成：

```text
按权重比例分配 CPU
```

例如：

```text
A weight = 1024
B weight = 1024
C weight = 2048
```

理想上：

```text
A 拿 25%
B 拿 25%
C 拿 50%
```

## 10.2 vruntime 是什么？

`vruntime` 是“虚拟运行时间”。

实际运行时间是：

```text
任务真的在 CPU 上跑了多久
```

虚拟运行时间是：

```text
根据任务权重折算后的运行时间
```

核心思想：

```text
权重大的人，实际跑同样时间，vruntime 增长慢
权重小的人，实际跑同样时间，vruntime 增长快
```

也就是说，高优先级任务虽然跑得多，但它的 `vruntime` 增长慢，所以不会很快被认为“你已经跑太多了”。

## 10.3 简化公式

可以粗略理解为：

```text
delta_vruntime = delta_exec * NICE_0_LOAD / weight
```

其中：

```text
delta_exec：实际运行时间
NICE_0_LOAD：nice 0 的基准权重，通常 1024
weight：当前任务权重
```

例如：

```text
nice 0 任务 weight = 1024
delta_exec = 10ms
delta_vruntime = 10ms
```

如果高优先级任务：

```text
weight = 2048
delta_exec = 10ms
delta_vruntime = 5ms
```

如果低优先级任务：

```text
weight = 512
delta_exec = 10ms
delta_vruntime = 20ms
```

所以：

```text
高优先级任务跑 10ms，只算虚拟跑了 5ms
低优先级任务跑 10ms，虚拟上算跑了 20ms
```

这就是 nice 影响 CPU 份额的核心。

## 10.4 红黑树为什么按 vruntime 排序？

CFS 需要快速找到：

```text
谁的 vruntime 最小？
谁最“欠 CPU 时间”？
```

所以把 runnable 的 `sched_entity` 按 `vruntime` 放进红黑树。

传统 CFS 里：

```text
红黑树最左边节点 = vruntime 最小的实体
pick_next_entity() 选择它
```

数据结构大致是：

```c
struct cfs_rq {
    struct rb_root_cached tasks_timeline;
    struct sched_entity *curr;
    struct sched_entity *next;
    struct sched_entity *last;
    struct sched_entity *skip;
    unsigned int nr_running;
    u64 min_vruntime;
};
```

## 10.5 min_vruntime 是什么？

`min_vruntime` 是当前 `cfs_rq` 上的最小虚拟时间基准。

它的作用：

```text
作为本 cfs_rq 的公平基准线
防止 vruntime 无限膨胀影响比较
新任务入队时避免占太大便宜
跨 CPU 迁移时进行 vruntime 归一化
```

你可以把它想成：

```text
这个 CPU 上大家平均已经跑到哪个进度了
```

新任务不能简单地把 `vruntime` 设为 0，否则它永远最小，会疯狂抢 CPU。

所以新任务入队时通常会参考：

```text
cfs_rq->min_vruntime
```

## 10.6 update_curr()

`update_curr()` 是 CFS 里非常关键的函数。

它做的事情大致是：

```text
计算当前任务从上次更新到现在实际运行了多久 delta_exec
更新 curr->sum_exec_runtime
按权重折算 delta_vruntime
更新 curr->vruntime
更新 cfs_rq->min_vruntime
更新 PELT 等统计
```

简化路径：

```text
scheduler_tick()
  -> task_tick_fair()
      -> entity_tick()
          -> update_curr()
```

以及很多 enqueue/dequeue、pick next、put prev 的路径也会调用它。

## 10.7 enqueue_entity / dequeue_entity

任务变成 runnable 时，需要入队：

```text
enqueue_task_fair()
  -> enqueue_entity()
      -> update_curr()
      -> place_entity()
      -> __enqueue_entity()
```

任务睡眠、迁移、退出时，需要出队：

```text
dequeue_task_fair()
  -> dequeue_entity()
      -> update_curr()
      -> __dequeue_entity()
```

入队/出队不只是操作红黑树，还会更新：

```text
nr_running
load
PELT
group scheduling 层级
runnable weight
```

## 10.8 pick_next_entity()

传统 CFS：

```text
pick_next_entity()
  -> 选 rb tree 最左边实体
```

但不只是简单最左，还会考虑：

```text
next
last
skip
wakeup preempt
buddy 机制
```

这些机制是为了兼顾：

```text
公平性
cache locality
交互响应
避免过度切换
```

# 11. CFS 抢占：什么时候当前任务会被换下？

CFS 抢占主要有两类：

```text
周期性 tick 检查
wakeup 抢占
```

## 11.1 tick 抢占

当前任务运行一段时间后，tick 到来：

```text
scheduler_tick()
  -> task_tick_fair()
      -> entity_tick()
          -> check_preempt_tick()
```

大致逻辑：

```text
当前任务已经运行太久？
当前任务的 vruntime 是否明显大于最左实体？
是否超过调度粒度？
```

如果需要抢占：

```c
resched_curr(rq);
```

设置：

```text
TIF_NEED_RESCHED
```

真正切换可能发生在：

```text
中断返回用户态
内核抢占点
显式 schedule()
```

## 11.2 wakeup 抢占

一个任务被唤醒时：

```text
try_to_wake_up()
  -> ttwu_do_wakeup()
  -> check_preempt_curr()
      -> check_preempt_wakeup()
```

如果被唤醒任务更应该运行，可能抢占当前任务。

典型场景：

```text
交互任务睡眠等待 IO
IO 完成后被唤醒
它很久没运行，vruntime 小
调度器倾向让它快速运行，提高响应性
```

## 11.3 为什么不能频繁抢占？

因为上下文切换有成本：

```text
保存/恢复寄存器
切换栈
可能切换地址空间
TLB/cache 影响
rq lock 操作
调度器计算开销
```

所以内核有最小粒度，避免任务刚跑一点点就被换掉。

# 12. CFS 组调度：cgroup 如何参与公平？

wowotech CFS 组调度文章摘要说明，CFS 管理的是调度实体；每个进程由 `task_struct` 描述，`task_struct` 包含 `sched_entity` 参与调度。组调度的关键是：调度实体不一定只是一个 task，也可以是一个 task group。([Wowotech][5])

## 12.1 为什么需要组调度？

假设：

```text
用户 A 启动 100 个进程
用户 B 启动 1 个进程
```

如果按 task 公平：

```text
A 拿 100/101 CPU
B 拿 1/101 CPU
```

这对用户 B 不公平。

组调度希望实现：

```text
先在组之间公平
再在组内部公平
```

例如：

```text
用户 A 组：50%
  A 的 100 个进程平分 50%

用户 B 组：50%
  B 的 1 个进程拿 50%
```

## 12.2 调度实体可以是 task，也可以是 group

CFS 统一抽象为：

```c
struct sched_entity
```

对于普通 task：

```text
sched_entity 嵌在 task_struct 里
```

对于 group：

```text
sched_entity 代表一个 cfs_rq 对应的调度组
```

层级大致是：

```text
root cfs_rq
  ├── group A 的 sched_entity
  │       └── group A 的 cfs_rq
  │             ├── task1 se
  │             └── task2 se
  └── group B 的 sched_entity
          └── group B 的 cfs_rq
                └── task3 se
```

## 12.3 cgroup CPU weight

在 cgroup v2 里可以通过：

```bash
cpu.weight
```

控制组权重。

在 cgroup v1 里常见：

```bash
cpu.shares
```

本质都是：

```text
影响 task group 这个 sched_entity 的 weight
```

# 13. CFS 带宽控制：quota / period / throttle

wowotech CFS 带宽控制文章摘要指出，该功能用于限制一个组最大 CPU 带宽；通过设置 `quota` 和 `period`，`period` 表示一个周期，`quota` 表示这个周期内组可以使用的 CPU 时间。([Wowotech][6])

## 13.1 quota / period 是什么？

例如：

```bash
cpu.cfs_period_us = 100000
cpu.cfs_quota_us  = 50000
```

含义：

```text
每 100ms 周期内，这个 cgroup 最多使用 50ms CPU
```

所以这个组大约被限制为：

```text
50%
```

如果：

```text
period = 100ms
quota  = 200ms
```

在多核系统中，它最多可同时用多个 CPU，总 CPU 带宽约为：

```text
200ms / 100ms = 2 CPUs
```

## 13.2 throttle

如果一个组在当前 period 内把 quota 用完了：

```text
该组被 throttle
里面的任务即使 runnable 也不能继续运行
```

等下一个 period 到来：

```text
quota 补充
组 unthrottle
任务重新可运行
```

## 13.3 关键结构

常见结构：

```c
struct cfs_bandwidth {
    raw_spinlock_t lock;
    ktime_t period;
    u64 quota;
    u64 runtime;
    struct hrtimer period_timer;
};

struct cfs_rq {
    s64 runtime_remaining;
    int throttled;
    struct list_head throttled_list;
};
```

大致流程：

```text
任务运行
  -> 消耗 cfs_rq runtime
  -> runtime_remaining <= 0
  -> throttle_cfs_rq()
  -> 从调度队列中移除
  -> period timer 到期
  -> distribute_cfs_runtime()
  -> unthrottle_cfs_rq()
```

## 13.4 你之前的 burst 现象如何理解？

你之前做过：

```text
quota=50ms period=100ms burst=50ms
```

burst 允许短时间借用未来额度，但如果任务持续满载：

```text
burst 很快耗尽
后续仍然 throttle
甚至可能由于借债/还债导致观察到 wall time 更长
```

所以 burst 不等于长期提升 CPU 份额，它更像“短期信用卡额度”。

# 14. PELT：调度器如何统计负载？

PELT 是 **Per-Entity Load Tracking**。wowotech 的 PELT 页面摘要说明，Linux 3.8 引入 PELT，用于分析负载跟踪的设计概念和实现；另一个摘要强调 PELT 不但能知道 CPU 负载，还能知道负载来自哪个调度实体，从而更精准地做负载均衡。([Wowotech][7])

## 14.1 为什么需要 PELT？

调度器要回答很多问题：

```text
这个 CPU 忙不忙？
这个任务最近是不是很重？
这个 cgroup 最近用了多少 CPU？
这个任务放到哪个 CPU 更合适？
CPU 频率要不要拉高？
```

仅靠瞬时状态不够。

例如：

```text
任务 A 此刻 runnable，但它过去 1 秒只运行了 1ms
任务 B 此刻 runnable，也许过去一直满载
```

这两个任务不能简单看成一样重。

## 14.2 PELT 的核心思想：指数衰减

PELT 不是简单累计总运行时间，而是更重视“最近”。

可以理解为：

```text
刚刚运行过，权重大
很久以前运行过，影响逐渐衰减
```

类似你统计一个人的“最近工作强度”：

```text
昨天加班 10 小时，很重要
一年前加班 10 小时，基本不影响今天判断
```

PELT 中常说 half-life 约 32ms，即每过一段时间，旧贡献衰减一半。

## 14.3 PELT 统计哪些指标？

常见指标：

```c
struct sched_avg {
    u64 last_update_time;
    u64 load_sum;
    u64 runnable_sum;
    u32 util_sum;

    unsigned long load_avg;
    unsigned long runnable_avg;
    unsigned long util_avg;
};
```

大致含义：

| 字段               | 含义               |
| ---------------- | ---------------- |
| load_avg         | 按权重计算的负载         |
| runnable_avg     | runnable 状态的历史比例 |
| util_avg         | 实际运行消耗的 CPU 算力   |
| last_update_time | 上次更新时间           |

## 14.4 load_avg 和 util_avg 区别

这是调度里非常重要的区别：

| 指标           | 关注点                | 用途                             |
| ------------ | ------------------ | ------------------------------ |
| load_avg     | 任务权重 + runnable 历史 | CFS 负载均衡                       |
| util_avg     | 实际 CPU 使用程度        | CPU capacity / schedutil / EAS |
| runnable_avg | runnable 但不一定运行    | 反映等待压力                         |

举例：

```text
一个低 nice 的高权重任务 runnable：
  load_avg 可能很高

一个任务实际只运行一点点：
  util_avg 可能不高
```

## 14.5 PELT 的层级传播

PELT 不是只统计 task，还会统计：

```text
task sched_entity
cfs_rq
task_group
rq
CPU
```

这和组调度结合起来非常重要：

```text
task 的负载汇总到 cfs_rq
cfs_rq 的负载影响 group se
group se 再参与上层 cfs_rq 调度
```

# 15. schedutil governor：调度器如何影响 CPU 频率？

`schedutil_governor` 文档摘要指出，PELT 的 dl utility、CFS/RT/DL 使用情况会参与判断 CPU 算力是否耗尽，从而影响返回最大 CPU utility 以及频率选择。([Wowotech][8])

## 15.1 cpufreq governor 是什么？

CPU 频率不是固定的。系统会根据负载调频：

```text
负载高 -> 提频
负载低 -> 降频
```

`schedutil` 的特点是：

> 直接使用调度器维护的 util 信息来决定频率。

## 15.2 schedutil 大致流程

```text
任务运行/唤醒/迁移
  -> PELT 更新 util_avg
  -> 调度器触发 cpufreq_update_util()
  -> schedutil 计算目标频率
  -> cpufreq driver 设置频率
```

简化公式可以理解为：

```text
freq = max_freq * util / capacity
```

如果：

```text
util 接近 capacity
```

就应该接近最高频。

## 15.3 为什么调度器能决定频率？

因为调度器最清楚：

```text
当前 CPU 上有哪些任务
它们最近运行了多久
它们是不是 RT/DL
它们是否正在等待 CPU
```

传统 governor 可能只看 CPU idle/busy 比例，而 schedutil 可以利用调度器内部更细粒度的信息。

## 15.4 排障意义

如果你遇到：

```text
延迟突然升高
CPU 使用率看起来不高
但任务响应慢
```

可能要看：

```text
util_avg 是否偏低
CPU 是否提频慢
uclamp 是否限制
RT/DL util 是否影响频率
```

# 16. 多核调度：load balance 和 task placement

wowotech load balance 概述文章摘要强调，PELT 能知道 CPU 负载，也知道负载来自哪个调度实体，从而更精准负载均衡；`load_balance_function` 摘要提到，除了 PELT 的 load avg，还会统计 sched group 中的 CFS 任务和总任务数量，只要 group 上有 CPU 存在多个任务，就会标记相关状态。([Wowotech][9])

## 16.1 多核调度要解决的问题

单核调度只需要：

```text
这个 CPU 下一个跑谁？
```

多核调度还要：

```text
这个任务应该放哪个 CPU？
哪个 CPU 忙？
哪个 CPU 闲？
任务迁移是否值得？
迁移会不会破坏 cache locality？
NUMA 访问会不会变差？
```

## 16.2 两类核心场景

| 场景             | 问题                  |
| -------------- | ------------------- |
| task placement | 新任务/唤醒任务放到哪个 CPU    |
| load balance   | CPU 之间如何周期性迁移任务保持均衡 |

## 16.3 task placement

典型路径：

```text
wake_up_new_task()
  -> select_task_rq()

try_to_wake_up()
  -> select_task_rq()
```

对于 fair class：

```text
select_task_rq_fair()
```

它会考虑：

```text
CPU affinity
当前 CPU 是否 idle
任务之前在哪个 CPU 运行
cache locality
NUMA locality
CPU capacity
energy aware scheduling
wake_affine
```

### 16.3.1 为什么唤醒任务不一定回原 CPU？

假设任务 A 之前在 CPU3 跑，后来睡眠等待 IO。

IO 完成时 CPU7 正好在执行唤醒者，且 CPU7 idle 或更适合运行 A。

调度器可能选择：

```text
把 A 放到 CPU7
```

原因可能是：

```text
减少唤醒延迟
避免 CPU3 忙导致等待
利用当前 cache 热度
```

但如果 A 的数据在 CPU3 cache 里很热，也可能倾向放回 CPU3。

这就是：

```text
latency vs locality
```

之间的取舍。

## 16.4 load balance

负载均衡常见触发：

| 类型                | 场景                     |
| ----------------- | ---------------------- |
| periodic balance  | 周期性 tick 检查            |
| idle balance      | CPU 空闲时拉任务             |
| newidle balance   | CPU 刚进入 idle 前尝试拉任务    |
| active balance    | 普通拉任务失败，主动推动任务         |
| nohz idle balance | tickless idle CPU 负载均衡 |

你之前 crash 里见过：

```text
pick_next_task_fair()
  -> newidle_balance()
      -> load_balance()
          -> raw_spin_rq_lock_nested()
```

这就是 CPU 没有本地合适任务时，尝试从别的 CPU 拉任务。

## 16.5 sched_domain / sched_group

调度器不会在所有 CPU 中乱选，而是按拓扑层级组织：

```text
SMT sibling
  -> core
  -> cluster
  -> LLC
  -> NUMA node
  -> system
```

对应结构：

```c
struct sched_domain
struct sched_group
```

负载均衡按 domain 逐层做。

例如：

```text
先在同 core/SMT 范围均衡
再在同 LLC 范围均衡
最后跨 NUMA 节点均衡
```

原因：

```text
越近迁移成本越低
越远迁移成本越高
```

## 16.6 load_balance 大致逻辑

简化：

```text
load_balance(this_cpu, sd)
  -> update_sd_lb_stats()
      -> 统计各 sched_group 负载
      -> 找 busiest group
      -> 找 local group
  -> calculate_imbalance()
      -> 算需要迁多少负载
  -> find_busiest_queue()
      -> 在 busiest group 里找 busiest rq
  -> detach_tasks()
      -> 从 busiest rq 拿任务
  -> attach_tasks()
      -> 放到 this_cpu rq
```

关键判断包括：

```text
是否有 imbalance
任务能不能迁移
是否 cache hot
是否符合 cpus_allowed
是否 throttled
是否 active_balance
```

## 16.7 负载均衡不是简单“任务数一样”

两个 CPU：

```text
CPU0: 1 个大任务，util=900
CPU1: 4 个小任务，util=100 each
```

任务数量：

```text
CPU0 1个
CPU1 4个
```

但负载：

```text
CPU0 更忙
```

所以现代调度更依赖 PELT/util/capacity，而不是只看 `nr_running`。

# 17. 上下文切换：context_switch / switch_mm / switch_to

wowotech context switch 基本框架文章摘要说明，它以 `context_switch` 为起点，分析进程切换中的基本操作和代码框架；TLB 文章则聚焦进程切换中的 TLB 处理。([Wowotech][10])

## 17.1 调度和切换的区别

```text
调度 scheduling:
  决定下一个运行谁

上下文切换 context switch:
  真正从当前任务切到下一个任务
```

路径大致是：

```text
schedule()
  -> __schedule()
      -> pick_next_task()
      -> context_switch()
          -> switch_mm_irqs_off()
          -> switch_to()
```

## 17.2 context_switch 做什么？

主要两件事：

```text
1. 切地址空间
2. 切 CPU 执行上下文
```

即：

```c
context_switch(rq, prev, next)
{
    switch_mm(prev->mm, next->mm, next);
    switch_to(prev, next, prev);
}
```

简化理解：

| 操作        | 内容                     |
| --------- | ---------------------- |
| switch_mm | 切换页表、ASID/PCID、TLB 上下文 |
| switch_to | 保存/恢复寄存器、栈指针、CPU 上下文   |

## 17.3 为什么线程切换比进程切换便宜？

如果两个线程共享同一个 `mm_struct`：

```text
prev->mm == next->mm
```

那么不需要完整切换地址空间。

而两个不同进程：

```text
prev->mm != next->mm
```

可能需要切换页表基址，处理 TLB 上下文。

这就是为什么：

```text
线程上下文切换通常比进程上下文切换更便宜
```

但注意，只是“通常”。如果锁竞争、cache 抖动严重，多线程也可能更慢。

## 17.4 active_mm

内核线程没有用户地址空间：

```text
kthread->mm == NULL
```

但 CPU 执行时仍需要一个有效的地址空间上下文，所以有：

```text
active_mm
```

内核线程可能借用前一个用户进程的 `active_mm`。

简化：

```text
普通用户进程：
  mm != NULL
  active_mm = mm

内核线程：
  mm = NULL
  active_mm = 借来的地址空间
```

# 18. TLB 处理：为什么进程切换会影响性能？

TLB 是页表缓存。

虚拟地址转换为物理地址需要查页表，页表多级访问很慢，所以 CPU 用 TLB 缓存最近的地址翻译。

## 18.1 问题：不同进程虚拟地址可以相同

进程 A：

```text
虚拟地址 0x400000 -> 物理页 X
```

进程 B：

```text
虚拟地址 0x400000 -> 物理页 Y
```

同一个虚拟地址在不同进程中含义不同。

所以从 A 切到 B 时，如果 TLB 里还缓存着 A 的翻译：

```text
0x400000 -> X
```

B 使用它就错了。

## 18.2 解决方法

常见方式：

```text
切换地址空间时刷新 TLB
或使用 ASID/PCID 标记 TLB 项属于哪个地址空间
```

## 18.3 ASID/PCID 的意义

如果硬件支持地址空间标识：

```text
TLB entry = ASID + virtual address -> physical address
```

那么不同进程可以共存 TLB 项。

切换进程时不一定要全部 flush TLB，只要换当前 ASID/PCID。

好处：

```text
减少 TLB flush
降低上下文切换成本
```

## 18.4 多核 TLB shootdown

如果一个进程在多个 CPU 上运行过，它的页表被修改时，其他 CPU 可能也有旧 TLB。

所以需要：

```text
IPI 通知其他 CPU flush 对应 TLB
```

这叫：

```text
TLB shootdown
```

性能排障中，如果频繁 munmap/mprotect/page migration，可能引起大量 TLB shootdown。

# 19. schedule 中的同步：preempt、rq lock、task state

`schedule sync` 文档摘要提到，它以 X 进程切换到 Y 进程为例，描述 `schedule()` 函数中同步机制的使用；在 X 进程上下文中，会先关闭抢占，X task 的 preempt counter 加 1。([Wowotech][11])

## 19.1 为什么 schedule 里同步复杂？

因为调度切换涉及多个共享状态：

```text
当前 CPU 的 rq
当前任务 current
任务 state
任务是否在 runqueue
任务是否正在被唤醒
任务是否正在迁移
```

如果同步不对，会出现：

```text
任务丢失
任务重复入队
任务明明被唤醒却不运行
任务在两个 CPU 上同时运行
rq 状态损坏
```

## 19.2 preempt_count

`preempt_count` 表示当前上下文是否允许抢占。

常见情况：

```text
持有 spinlock 时 preempt disabled
中断上下文 preempt disabled
softirq 上下文 preempt disabled
显式 preempt_disable()
```

如果 `preempt_count > 0`：

```text
不能被普通内核抢占
```

## 19.3 rq lock

每个 CPU 有自己的 runqueue：

```c
struct rq
```

调度器操作 rq 时需要加锁：

```c
rq_lock(rq, &rf);
```

因为可能有其他 CPU 同时：

```text
唤醒任务到这个 rq
从这个 rq 拉任务
迁移任务到/出这个 rq
修改任务状态
```

你之前 crash 看到的：

```text
raw_spin_rq_lock_nested()
```

就是和 rq 锁有关。

## 19.4 task state 和 schedule 的关系

典型睡眠模式：

```c
set_current_state(TASK_INTERRUPTIBLE);
if (!condition)
    schedule();
__set_current_state(TASK_RUNNING);
```

关键点：

```text
先设置 state，再检查 condition，再 schedule
```

否则可能发生 wakeup race。

错误模式：

```text
检查 condition
准备睡眠
这时事件发生并 wakeup
然后当前任务才 set state + schedule
结果错过 wakeup
```

这就是内核里为什么有大量 memory barrier、锁和状态顺序控制。

# 20. 中断上下文中为什么不能睡眠？

`schedule-in-interrupt` 文档摘要提出经典问题：每个 Linux 驱动工程师都知道中断上下文不能睡眠，但为什么 interrupt context 中不能调用导致睡眠的 kernel API？([Wowotech][12])

## 20.1 睡眠意味着什么？

睡眠通常意味着：

```text
当前执行流主动放弃 CPU
状态改成 TASK_INTERRUPTIBLE / TASK_UNINTERRUPTIBLE
调用 schedule()
等待别人 wake up
```

这要求当前执行流是一个正常 task。

## 20.2 中断上下文不是独立 task

硬中断发生时：

```text
打断当前正在运行的 task
借用当前 task 的内核栈或中断栈
执行中断 handler
处理完返回被打断现场
```

中断 handler 本身不是一个可以被调度器管理的 task。

所以它不能说：

```text
我睡了，等别人唤醒我
```

因为没有一个正常的“中断 task”挂到等待队列里。

## 20.3 中断上下文睡眠会破坏语义

如果中断里睡眠：

```text
当前被打断的 task 怎么办？
中断现场怎么保存？
中断什么时候返回？
被中断代码可能持有什么锁？
其他 CPU 是否等这个中断完成？
```

可能导致：

```text
死锁
上下文错乱
中断延迟不可控
系统卡死
```

## 20.4 驱动里怎么处理？

硬中断里只做紧急事情：

```text
读取设备状态
ack 中断
保存必要信息
唤醒 bottom half/thread
```

需要睡眠的工作放到：

```text
threaded irq
workqueue
kernel thread
tasklet/softirq 但 softirq/tasklet 也不能睡眠
```

注意：

| 上下文             | 能否睡眠        |
| --------------- | ----------- |
| hardirq         | 不能          |
| softirq         | 不能          |
| tasklet         | 不能          |
| workqueue       | 可以          |
| threaded irq    | 可以          |
| process context | 可以，视锁和上下文而定 |

# 21. Deadline 调度器

Deadline 文档讲 `SCHED_DEADLINE`。wowotech deadline 第二篇摘要说明，Linux DL 调度器面向有 deadline 限制的 sporadic task，是全局 EDF 调度器。

## 21.1 Deadline 三个核心参数

```text
runtime
deadline
period
```

含义：

```text
每 period 时间内
任务最多需要 runtime 的 CPU 时间
并希望在相对 deadline 前完成
```

例如：

```text
runtime  = 10ms
deadline = 30ms
period   = 50ms
```

意思是：

```text
每 50ms 来一份工作
每份工作需要最多 10ms CPU
希望 30ms 内完成
```

## 21.2 EDF：Earliest Deadline First

Deadline 调度基本思想：

```text
谁的 deadline 最近，谁先运行
```

例如：

```text
任务 A deadline = 20ms 后
任务 B deadline = 50ms 后
```

先跑 A。

## 21.3 CBS：Constant Bandwidth Server

CBS 用于限制 deadline 任务不会无限占 CPU。

如果 deadline 任务超过 runtime：

```text
它会被推迟/限流
```

避免一个 DL 任务把整个系统吃死。

## 21.4 admission control

不是所有 deadline 参数都允许设置。

内核要检查：

```text
总利用率是否超过 CPU capacity
```

比如单 CPU 上：

```text
任务 A runtime/period = 0.6
任务 B runtime/period = 0.6
```

总利用率 1.2，超过 1 个 CPU，理论不可调度。

所以设置 `SCHED_DEADLINE` 可能失败。

## 21.5 和 RT 的区别

| 调度类       | 关注点     |
| --------- | ------- |
| RT        | 固定实时优先级 |
| DL        | 截止时间和带宽 |
| CFS/EEVDF | 公平性和响应性 |

RT 是：

```text
优先级高的永远压过低的
```

DL 是：

```text
deadline 近的更紧急
```

# 22. ACCESS_ONCE / READ_ONCE / WRITE_ONCE

wowotech access-once 文档摘要说明，`ACCESS_ONCE` 本质是增加 `volatile`，确保编译器每次访问变量 x 都从内存获取。([Wowotech][13])

## 22.1 它解决什么问题？

内核并发代码中，编译器可能优化：

```c
while (!flag) {
}
```

如果编译器认为 `flag` 没被当前线程修改，可能把它优化成：

```c
if (!flag)
    while (1) {}
```

这在并发场景就错了。

所以需要：

```c
READ_ONCE(flag)
WRITE_ONCE(flag, 1)
```

告诉编译器：

```text
这次读/写必须真实发生
不要合并
不要乱优化
```

## 22.2 READ_ONCE 不是锁

非常重要：

```text
READ_ONCE/WRITE_ONCE 不能保证复合操作原子性
不能替代 spinlock/mutex
不能保证完整内存顺序
```

例如：

```c
x = READ_ONCE(a);
y = READ_ONCE(b);
```

它只能保证每个变量按一次显式访问，不保证 `a` 和 `b` 之间一定有你想要的同步关系。

## 22.3 和 barrier 的区别

| 工具             | 作用             |
| -------------- | -------------- |
| READ_ONCE      | 防止编译器合并/撕裂/重复读 |
| WRITE_ONCE     | 防止编译器合并/撕裂写    |
| barrier()      | 编译器屏障          |
| smp_rmb/wmb/mb | CPU 内存屏障       |
| spin_lock      | 锁 + 内存顺序       |
| mutex          | 睡眠锁 + 内存顺序     |

## 22.4 常见场景

```c
if (READ_ONCE(p->on_rq)) {
    ...
}
```

调度器、RCU、lockless fast path 中非常常见。

# 23. ARM 系统调用路径

wowotech ARM syscall 文档摘要指出，在旧规范中，需要从 SWI 指令获取系统调用号，并通过 `lr_svc` 找到 SWI 指令位置。([Wowotech][14])

## 23.1 系统调用是什么？

用户态不能直接执行内核特权操作。

例如：

```c
open()
read()
write()
clone()
sched_setscheduler()
```

这些最终都要进入内核。

系统调用路径大致是：

```text
用户态 libc wrapper
  -> 设置系统调用号和参数
  -> 执行 svc/swi 指令
  -> CPU 进入异常模式/EL1
  -> 内核异常入口
  -> syscall table
  -> sys_xxx()
  -> 返回用户态
```

## 23.2 ARM 上的参数传递

不同 ARM ABI 有差异，但通常是：

```text
系统调用号放在指定寄存器
参数放在 r0-r6 或 x0-x5 等寄存器
svc 触发异常
```

ARM64 常见：

```text
x8 = syscall number
x0-x5 = args
svc #0
```

## 23.3 和进程管理的关系

你前面看到的很多用户态 API：

```text
fork
clone
execve
setpriority
sched_setscheduler
sched_setaffinity
```

最终都通过 syscall 进入内核。

所以 syscall 是：

```text
用户态调度/进程管理接口进入内核的门
```

# 27. 对你最重要的学习提醒

这批文档你不需要逐字背，但有几个点必须真正理解：

第一，**Linux 调度的是 task，不是传统教材里的进程**。进程、线程、内核线程都是不同资源共享方式下的 task。

第二，**CFS 的核心是按权重公平**。`vruntime` 不是实际运行时间，而是经过权重折算后的虚拟时间。

第三，**PELT 是现代调度器的基础传感器**。调度器做负载均衡、任务放置、CPU 选频，都离不开它。

第四，**多核调度不是简单地把任务平均分配**。它要考虑 CPU 拓扑、cache locality、NUMA、capacity、affinity、migration cost。

第五，**上下文切换不是简单函数调用**。它涉及寄存器、栈、地址空间、TLB、rq lock、preempt count、内存屏障。

第六，**wowotech CFS 系列非常适合打基础，但你看 6.6 调度问题时要补 EEVDF**。传统 CFS 关注 `vruntime` 最小；EEVDF 更关注 eligible、lag 和 virtual deadline。([内核文档][3])

# 0. 先用一张总图建立全局

学习中断子系统，最重要的是先建立这条链路：

```text
外设产生中断
    ↓
外设的 interrupt pin / interrupt line 变化
    ↓
中断控制器接收，例如 ARM GIC、GPIO irqchip、PMIC irqchip
    ↓
中断控制器根据 enable、mask、priority、target CPU 等配置决定是否上报 CPU
    ↓
CPU 收到 IRQ 异常
    ↓
CPU 跳到异常向量表
    ↓
arch 入口代码保存现场
    ↓
从中断控制器读取硬件中断号 hwirq
    ↓
irq_domain 把 hwirq 映射成 Linux IRQ number，也叫 virq
    ↓
根据 virq 找到 irq_desc
    ↓
irq_desc->handle_irq，也就是 high level irq flow handler，负责流控
    ↓
调用 irq_desc->action 链表里的驱动 handler
    ↓
驱动 handler 快速处理，必要时唤醒 threaded irq / softirq / tasklet / workqueue
    ↓
中断控制器 ack/eoi/unmask
    ↓
恢复现场，返回被打断的代码
```

这条链路里，最核心的对象是：

```text
hwirq        硬件中断号，中断控制器认识
virq         Linux IRQ number，Linux 内核和驱动认识
irq_domain   负责 hwirq → virq 映射
irq_desc     一个 Linux IRQ 的总管理结构
irq_data     irq_desc 里保存硬件相关信息
irq_chip     操作具体中断控制器硬件的方法表
irqaction    驱动通过 request_irq() 注册进来的 handler
handle_irq   Linux 通用中断流控函数
```

你可以先记住一句话：

```text
硬件产生的是 hwirq；
Linux 驱动使用的是 virq；
irq_domain 负责翻译；
irq_desc 负责管理；
irq_chip 负责操作硬件；
irqaction 负责保存驱动 handler。
```

# 1. 中断为什么存在？

CPU 正在执行普通代码，例如用户进程、内核线程、系统调用。外设突然有事情需要 CPU 处理，比如：

```text
网卡收到包
磁盘完成一次 IO
键盘按下
定时器到期
GPIO 电平变化
触摸屏产生事件
UART 收到数据
```

如果没有中断，CPU 只能不停轮询：

```c
while (1) {
    check_netcard();
    check_disk();
    check_keyboard();
    check_timer();
}
```

这会浪费大量 CPU，而且响应也不及时。

中断的思想是：

```text
平时 CPU 正常执行任务；
外设有事时，主动打断 CPU；
CPU 暂停当前代码，执行中断处理函数；
处理完后，继续执行原来的代码。
```

所以中断本质上是一种**异步通知机制**。

# 2. 中断、异常、软中断不要混在一起

学习时容易把这些词混淆。

## 2.1 interrupt，中断

一般指**外部设备异步触发**的事件。

例如：

```text
网卡收包
GPIO 变化
磁盘 IO 完成
定时器中断
```

特点是：

```text
异步发生
和当前 CPU 正在执行的指令没有直接因果关系
通常来自外设或中断控制器
```

## 2.2 exception，异常

异常通常是 CPU 执行当前指令过程中产生的。

例如：

```text
缺页异常 page fault
除零异常
非法指令异常
系统调用异常
数据访问异常
指令预取异常
```

特点是：

```text
同步发生
和当前执行的指令直接相关
CPU 自己检测到问题或特殊事件
```

## 2.3 software interrupt，软件中断

软件中断这个词在不同上下文里含义不同。

有时指：

```text
通过软件指令触发异常，例如早期 x86 int 0x80 系统调用
```

有时又容易和 Linux 的：

```text
softirq
```

混在一起。

**Linux softirq 不是 CPU 异常意义上的软件中断，而是 Linux 内核里的 bottom half 机制。**

## 2.4 hardirq 和 softirq

Linux 里常说：

```text
hardirq：真正硬件中断上半部
softirq：中断下半部机制之一
```

hardirq 是外设中断进来以后马上执行的部分；softirq 是 Linux 为了延后处理部分工作而设计的机制。

# 3. 为什么中断处理要分上半部和下半部？

中断处理有一个核心原则：

```text
hardirq 里越快越好。
```

原因是 hardirq 上下文有很多限制：

```text
不能睡眠
不能拿可能睡眠的 mutex
不能做耗时操作
不能长时间关中断
不能访问可能阻塞的慢设备
不能执行复杂业务逻辑
```

例如网卡收到一个包，如果 hardirq 里完成所有协议栈处理，会导致：

```text
中断处理太久
其他中断响应延迟
用户进程长时间得不到运行
系统实时性变差
```

所以 Linux 把中断处理拆成两部分：

```text
上半部 top half / hardirq：
    立即执行
    做最紧急、最少量的事情
    例如确认中断、读取状态、屏蔽中断、搬运少量数据、唤醒下半部

下半部 bottom half：
    延后执行
    做相对耗时的事情
    例如协议栈处理、复杂状态机、可能睡眠的操作
```

Linux 常见下半部机制有：

```text
softirq
tasklet
workqueue
threaded irq handler
NAPI
timer
```

wowotech 的 softirq 和 tasklet 文章都围绕“中断分为 top half 和 bottom half”展开；softirq/tasklet 属于“越快越好型”的下半部，而 workqueue/threaded irq 属于运行在进程上下文、限制更少的后处理机制。([Wowotech][2])

# 4. 中断子系统总体架构

wowotech 的中断子系统综述文章把 Linux 中断子系统分成几个层次：硬件外设、中断控制器、CPU、体系结构相关代码、通用中断处理层和驱动 API。Linux generic IRQ 文档也强调，generic IRQ layer 的目标是为设备驱动提供对中断控制器硬件细节的抽象，使驱动可以通过通用 API request/enable/disable/free interrupt，而不需要直接了解具体中断硬件。([Wowotech][1])

可以抽象成：

```text
设备驱动层
    request_irq()
    request_threaded_irq()
    free_irq()
    enable_irq()
    disable_irq()

通用中断处理层 generic IRQ layer
    irq_desc
    irq_data
    irqaction
    high level irq flow handler
    irq_domain
    irq_chip glue

中断控制器驱动层 irqchip driver
    GIC driver
    GPIO irqchip driver
    PMIC irqchip driver
    interrupt controller cascade driver

体系结构相关层 arch layer
    异常向量表
    IRQ 入口汇编
    保存/恢复寄存器
    调用 generic IRQ layer

硬件层
    外设
    interrupt controller
    CPU IRQ line
```

最关键的设计思想是：

```text
驱动不直接操作中断控制器；
中断控制器驱动不直接知道所有外设业务；
generic IRQ layer 把两者连接起来。
```

# 5. 从硬件看中断：外设、中断控制器、CPU

一个 SoC 里通常不是每个外设都直接连到 CPU。

更常见的是：

```text
外设1  ─┐
外设2  ─┤
外设3  ─┤
GPIO   ─┼──> 中断控制器 ───> CPU IRQ line
PMIC   ─┤
Timer  ─┘
```

中断控制器的作用是：

```text
收集多个外设中断
给中断编号
控制 mask/unmask
控制 enable/disable
控制优先级
控制触发类型
控制发给哪个 CPU
向 CPU 报告当前最高优先级中断
接收 CPU 的 ack/eoi
```

多个中断控制器还可以级联：

```text
GPIO controller
    ↓
GIC
    ↓
CPU
```

这时 GPIO controller 自己也是一个 irqchip。GPIO 下挂的每个 pin 可以是一个 hwirq，但 GPIO controller 本身通过一个 parent IRQ 接到 GIC 上。

所以真实系统中经常有这种层级：

```text
设备 GPIO 中断
    ↓
GPIO irq_domain
    ↓
GPIO irqchip
    ↓
GIC irq_domain
    ↓
GIC irqchip
    ↓
CPU
```

这也是为什么后面会出现 `irq_domain` 和层级中断域。

# 6. hwirq、virq、IRQ number

这是中断子系统最关键的概念之一。

## 6.1 hwirq 是什么？

`hwirq` 是硬件中断号。

比如 GIC 里：

```text
SGI: 0~15
PPI: 16~31
SPI: 32 以后
```

某个网卡可能接在 GIC SPI 45 上，那么从 GIC 角度看，它的硬件中断号就是某个 hwirq。

设备树中通常描述的是硬件中断信息，例如：

```dts
interrupts = <0 45 4>;
```

这里面包含的是中断控制器能够理解的信息，不是 Linux 驱动最终拿到的 virq。

## 6.2 virq / IRQ number 是什么？

Linux 给每个中断分配一个逻辑编号，叫：

```text
IRQ number
virq
Linux IRQ
```

驱动调用：

```c
irq = platform_get_irq(pdev, 0);
request_irq(irq, handler, flags, name, dev);
```

这里的 `irq` 通常就是 Linux IRQ number，不是硬件 hwirq。

## 6.3 为什么不能直接用 hwirq？

因为 hwirq 只在某个中断控制器内部有意义。

例如：

```text
GIC 的 hwirq 5
GPIO controller 的 hwirq 5
PMIC irqchip 的 hwirq 5
```

它们可能完全不是同一个中断。

所以 Linux 用：

```text
irq_domain + hwirq
```

来唯一确定一个硬件中断源。

映射关系是：

```text
(domain, hwirq) → virq
```

irq_domain 文档也明确说明，irq_domain 库是在 irq_alloc_desc* 分配 IRQ number 的基础上，增加 hwirq 到 Linux IRQ number 的映射能力；中断控制器驱动通常创建并注册自己的 irq_domain，然后通过 irq_create_mapping() 建立 hwirq 到 irq_desc/IRQ number 的关联。([Infradead][3])

# 7. IRQ Domain 详细总结

## 7.1 IRQ Domain 解决什么问题？

它解决的是：

```text
硬件中断号 hwirq 如何映射成 Linux IRQ number？
```

可以把 `irq_domain` 理解成一本“翻译字典”。

```text
GIC domain:
    hwirq 32 → virq 45
    hwirq 33 → virq 46
    hwirq 34 → virq 47

GPIO domain:
    hwirq 0  → virq 120
    hwirq 1  → virq 121
```

同一个 hwirq，在不同 domain 下可以映射成不同 virq。

## 7.2 irq_domain 是谁创建的？

一般是**中断控制器驱动**创建的，不是普通设备驱动创建的。

例如：

```text
GIC driver 创建 GIC irq_domain
GPIO irqchip driver 创建 GPIO irq_domain
PMIC irqchip driver 创建 PMIC irq_domain
```

普通设备驱动只是通过：

```c
platform_get_irq()
of_irq_get()
irq_of_parse_and_map()
```

拿到已经映射好的 Linux IRQ number。

## 7.3 irq_domain 的典型数据结构

概念上可以看成：

```c
struct irq_domain {
    const char *name;
    const struct irq_domain_ops *ops;
    struct irq_domain_chip_generic *gc;
    void *host_data;
    ...
};
```

更关键的是 `irq_domain_ops`：

```c
struct irq_domain_ops {
    int (*match)(...);
    int (*map)(struct irq_domain *d, unsigned int virq, irq_hw_number_t hwirq);
    void (*unmap)(struct irq_domain *d, unsigned int virq);
    int (*xlate)(struct irq_domain *d, struct device_node *node,
                 const u32 *intspec, unsigned int intsize,
                 unsigned long *out_hwirq, unsigned int *out_type);
};
```

核心回调有：

```text
xlate：
    把设备树 interrupt specifier 翻译成 hwirq 和触发类型

map：
    建立 virq 和 hwirq 的关系
    设置 irq_chip
    设置 high level irq handler
    设置 irq_data
```

## 7.4 设备树如何参与 IRQ Domain？

设备树里，一个设备可能这样写：

```dts
uart0: serial@xxx {
    interrupts = <0 33 4>;
    interrupt-parent = <&gic>;
};
```

含义是：

```text
这个 UART 的中断由 gic 这个 interrupt-parent 管；
interrupts 里的数据要交给 gic 的 irq_domain 解析。
```

解析过程大致是：

```text
platform_get_irq()
    ↓
of_irq_get()
    ↓
of_irq_parse_one()
    ↓
找到 interrupt-parent
    ↓
找到 parent 对应的 irq_domain
    ↓
irq_create_of_mapping()
    ↓
irq_domain_ops->xlate()
    ↓
得到 hwirq + type
    ↓
irq_create_mapping()
    ↓
如果还没有 virq，则分配 irq_desc
    ↓
调用 irq_domain_ops->map()
    ↓
返回 Linux IRQ number
```

## 7.5 map 阶段做什么？

以 GIC 为例，`map()` 里一般会设置：

```c
irq_set_chip_and_handler(virq, &gic_chip, handle_fasteoi_irq);
irq_set_chip_data(virq, gic_data);
irq_set_status_flags(virq, IRQ_LEVEL);
```

也就是说，IRQ Domain 不只是建一个数字映射，它还负责把这个 virq 的处理方式配置好：

```text
这个 IRQ 用哪个 irq_chip？
这个 IRQ 用哪个 high level flow handler？
这个 IRQ 的触发类型是什么？
这个 IRQ 的 chip_data 是什么？
```

## 7.6 常见映射类型

旧内核里常见的 irq_domain 映射方式包括：

```text
linear map：
    hwirq 范围较小，直接数组映射
    查询快

tree/radix tree map：
    hwirq 范围稀疏或很大
    节省空间

legacy map：
    兼容老式静态 IRQ 编号

no-map：
    硬件号和 Linux IRQ 号直接对应的特殊情况
```

现代内核里还大量使用 hierarchical irqdomain，用于描述中断控制器层级关系，例如 GPIO irqchip 下面接 GIC。

# 8. irq_desc：一个 IRQ 的“档案袋”

wowotech 的 IRQ number 和中断描述符文章主要讲 `irq_desc`：系统里每一个外设中断线都会对应一个中断描述符，IRQ number 是 CPU/Linux 用来标识外设中断的虚拟 interrupt ID，而不是硬件无关的真实电气编号。([Wowotech][4])

可以把 `irq_desc` 理解成一个 IRQ 的总管理对象：

```text
irq_desc
 ├── irq_data
 │    ├── irq       Linux IRQ number
 │    ├── hwirq     硬件中断号
 │    ├── domain    所属 irq_domain
 │    └── chip      irq_chip
 │
 ├── handle_irq     high level irq flow handler
 ├── action         驱动注册的 irqaction 链表
 ├── status/state   IRQ 状态
 ├── depth          disable_irq 嵌套深度
 ├── lock           保护 desc 的自旋锁
 ├── kstat_irqs     /proc/interrupts 统计
 └── name/percpu 等信息
```

## 8.1 irq_desc 的核心作用

它负责管理一个 Linux IRQ 的全部状态：

```text
这个 IRQ 当前是否 enable？
是否 masked？
是否 pending？
是否 disabled？
是否 in progress？
是否 wakeup source？
绑定到哪些 CPU？
用哪个 irq_chip？
用哪个 flow handler？
有哪些驱动 handler？
统计次数是多少？
```

所以中断来了以后，通用层不是直接找驱动，而是先找到：

```text
virq → irq_desc
```

再通过 `irq_desc` 继续处理。

## 8.2 irq_desc 如何分配？

老内核里，可能有一个静态数组：

```c
struct irq_desc irq_desc[NR_IRQS];
```

这种方式简单，但问题是：

```text
如果 NR_IRQS 很大，会浪费内存；
如果硬件中断号很稀疏，数组也浪费；
热插拔和动态中断控制器不好处理。
```

所以后来引入 `CONFIG_SPARSE_IRQ`，irq_desc 可以动态分配。

动态方式下，只有真正用到某个 IRQ 时才分配 `irq_desc`，这和你之前学内存里的“页表按需分配”思想很像。

## 8.3 irq_data 是什么？

`irq_data` 是 `irq_desc` 里偏硬件的一部分。

它描述：

```text
这个 Linux IRQ 对应哪个 hwirq
属于哪个 irq_domain
对应哪个 irq_chip
chip 私有数据是什么
affinity 信息是什么
```

概念上：

```text
irq_desc 是总档案袋；
irq_data 是档案袋里记录硬件映射关系的那一页。
```

## 8.4 irq_chip 是什么？

`irq_chip` 是具体中断控制器的操作函数表。

典型函数有：

```c
struct irq_chip {
    void (*irq_mask)(struct irq_data *data);
    void (*irq_unmask)(struct irq_data *data);
    void (*irq_ack)(struct irq_data *data);
    void (*irq_eoi)(struct irq_data *data);
    int  (*irq_set_type)(struct irq_data *data, unsigned int type);
    int  (*irq_set_affinity)(struct irq_data *data,
                             const struct cpumask *dest,
                             bool force);
    int  (*irq_set_wake)(struct irq_data *data, unsigned int on);
};
```

它的作用是：

```text
generic IRQ layer 想屏蔽中断
    ↓
调用 desc->irq_data.chip->irq_mask()
    ↓
具体 GIC/GPIO/PMIC 驱动写硬件寄存器
```

所以：

```text
irq_desc 负责通用状态管理；
irq_chip 负责具体硬件操作。
```

## 8.5 irqaction 是什么？

驱动调用 `request_irq()` 后，内核会创建一个 `irqaction`：

```c
struct irqaction {
    irq_handler_t handler;
    void *dev_id;
    void __percpu *percpu_dev_id;
    struct irqaction *next;
    irq_handler_t thread_fn;
    struct task_struct *thread;
    unsigned int flags;
    const char *name;
};
```

一个 IRQ 可能有多个 action，因为中断可以共享：

```text
irq_desc
 └── action 链表
      ├── 网卡 handler
      ├── 声卡 handler
      └── USB handler
```

共享中断里，每个 handler 都会被调用，每个驱动要自己判断：

```text
这个中断是不是我设备产生的？
```

如果是，返回：

```c
IRQ_HANDLED
```

如果不是，返回：

```c
IRQ_NONE
```

# 9. High level irq event handler：中断流控层

wowotech 的 High level irq event handler 文章讲的是中断发生后 generic IRQ layer 如何根据中断类型执行不同的流控逻辑。Linux generic IRQ 文档也强调，早期 `__do_IRQ()` 大一统处理方式后来被拆成 level、edge、simple、fast EOI、per-CPU 等不同 high-level handlers，以便把“IRQ flow”和“chip hardware details”分离。([Wowotech][5])

## 9.1 它不是驱动 handler

注意区分：

```text
high level irq event handler：
    Linux 中断子系统内部的流控函数
    例如 handle_level_irq、handle_edge_irq

driver irq handler：
    驱动通过 request_irq 注册的函数
    例如 xxx_irq_handler
```

调用关系是：

```text
CPU IRQ entry
    ↓
generic_handle_irq(virq)
    ↓
desc->handle_irq(desc)
    ↓
handle_level_irq(desc) / handle_edge_irq(desc) / handle_fasteoi_irq(desc)
    ↓
handle_irq_event(desc)
    ↓
驱动 irqaction->handler()
```

也就是说：

```text
先走 flow handler；
flow handler 再调用驱动 handler。
```

## 9.2 为什么需要 flow handler？

因为不同类型中断，处理顺序不同。

例如：

```text
电平触发中断：
    如果设备中断源没有清掉，中断线会一直保持有效
    所以常常需要 mask/ack → handler → unmask

边沿触发中断：
    中断是瞬间边沿
    如果处理期间又来一个边沿，要记录 pending，不能丢事件

fast EOI 中断：
    硬件模型要求尽快 EOI
    典型如 GIC 的一些模式
```

所以 Linux 把“中断流控”抽象成多种 high level handler。

## 9.3 常见 flow handler

### handle_simple_irq()

适合简单中断控制器。

大概逻辑：

```text
不做复杂 mask/ack/eoi
直接调用驱动 handler
```

常用于某些级联或简单场景。

### handle_level_irq()

适合电平触发类型。

典型流程：

```text
加 desc lock
    ↓
mask + ack
    ↓
检查 IRQ 状态
    ↓
调用驱动 handler
    ↓
如果没有被 disable，则 unmask
```

为什么电平触发要小心？

因为电平触发表示：

```text
只要设备还在拉着中断线，中断就一直存在。
```

所以驱动 handler 里必须清设备中断源，例如：

```c
status = readl(dev->status_reg);
writel(status, dev->clear_reg);
```

否则：

```text
unmask 后立刻再次中断
    ↓
再次 handler
    ↓
再次 unmask
    ↓
中断风暴
```

### handle_edge_irq()

适合边沿触发。

边沿触发是事件型：

```text
低到高的一瞬间
或者高到低的一瞬间
```

问题是：

```text
handler 正在执行时，又来了一个边沿怎么办？
```

Linux 需要通过 pending 状态避免丢边沿。

简化逻辑：

```text
ack 当前边沿
    ↓
如果正在处理，则标记 pending
    ↓
如果没在处理，则调用 handler
    ↓
处理完后检查 pending
    ↓
如果 pending，再处理一轮
```

### handle_fasteoi_irq()

适合需要 EOI 的中断控制器，例如 GIC 风格。

典型流程：

```text
调用驱动 handler
    ↓
irq_eoi()
```

GIC 中 CPU interface 需要知道：

```text
CPU 已经处理完这个 interrupt ID 了
```

这个动作就是 EOI。

### handle_percpu_irq()

适合 per-CPU 中断。

例如：

```text
local timer
IPI
某些 PPI
```

特点是：

```text
每个 CPU 自己处理自己的中断；
不需要像全局共享 SPI 那样做复杂 affinity 分发。
```

### handle_nested_irq()

用于嵌套/级联中断场景。

例如：

```text
GPIO controller 挂在 GIC 一个中断上
GPIO controller 内部又管理很多 GPIO pin 中断
```

流程类似：

```text
GIC parent IRQ 触发
    ↓
GPIO irqchip 的 chained handler 执行
    ↓
读取 GPIO pending 寄存器
    ↓
找出具体哪个 GPIO pin 触发
    ↓
调用 nested IRQ 的 handler
```

# 10. mask、ack、eoi、unmask 的区别

这是非常容易混淆的点。

## 10.1 mask

```text
mask = 暂时屏蔽这个中断源，不让它继续上报 CPU。
```

它不一定清除已经发生的中断，只是禁止继续递送。

## 10.2 unmask

```text
unmask = 解除屏蔽，允许这个中断源继续上报 CPU。
```

## 10.3 ack

```text
ack = acknowledge，确认我已经看到这个中断。
```

对边沿中断尤其重要，因为硬件可能需要 ack 来清 pending latch。

## 10.4 eoi

```text
eoi = end of interrupt，告诉中断控制器 CPU 已经处理完这个中断。
```

在 GIC 这类控制器里，EOI 会影响 active 状态和后续中断分发。

## 10.5 清设备中断源

这又是另一个动作：

```text
清外设内部的 interrupt status bit。
```

例如：

```c
status = readl(DEV_INT_STATUS);
writel(status, DEV_INT_CLEAR);
```

注意它和 `ack/eoi` 不同：

```text
ack/eoi：
    通常是对中断控制器说的

清设备中断源：
    是对产生中断的外设说的
```

电平触发中断风暴经常是因为：

```text
GIC EOI 了
但是设备 status 没清
中断线仍然有效
于是马上又进中断
```

# 11. 驱动申请中断 API

wowotech 的驱动申请中断 API 文章围绕普通驱动如何向 Linux 中断子系统注册中断处理函数展开，重点涉及 `request_irq()`、`request_threaded_irq()` 以及共享中断、threaded handler、IRQF_ONESHOT 等机制。([Wowotech][6])

## 11.1 request_irq()

典型用法：

```c
ret = request_irq(irq, xxx_irq_handler,
                  IRQF_TRIGGER_HIGH,
                  "xxx-device", dev);
```

参数含义：

```text
irq：
    Linux IRQ number，不是 hwirq

handler：
    hardirq 上下文执行的处理函数

flags：
    中断触发类型、共享属性、suspend 行为等

name：
    /proc/interrupts 里显示的名字

dev_id：
    驱动私有指针
    共享中断时用于区分设备
    free_irq() 时也要传同一个 dev_id
```

handler 原型：

```c
static irqreturn_t xxx_irq_handler(int irq, void *dev_id)
{
    struct xxx_dev *dev = dev_id;

    /* 读取设备状态 */
    /* 判断是不是自己的中断 */
    /* 清中断源 */

    return IRQ_HANDLED;
}
```

返回值：

```text
IRQ_NONE：
    这个中断不是我设备产生的

IRQ_HANDLED：
    我处理了这个中断

IRQ_WAKE_THREAD：
    唤醒 threaded irq handler
```

## 11.2 request_irq() 内部大致做了什么？

简化路径：

```text
request_irq()
    ↓
request_threaded_irq(..., thread_fn = NULL)
    ↓
创建 irqaction
    ↓
保存 handler/name/dev_id/flags
    ↓
__setup_irq()
    ↓
找到 irq_desc
    ↓
检查 flags 是否兼容
    ↓
如果是共享中断，挂到 desc->action 链表
    ↓
如果是第一个 action，初始化 IRQ 硬件
    ↓
enable/unmask IRQ
```

核心结果是：

```text
irq_desc->action 链表上多了一个驱动 handler。
```

## 11.3 IRQF_SHARED

多个设备共享一个 IRQ 时使用：

```c
request_irq(irq, handler, IRQF_SHARED, name, dev_id);
```

共享中断要求：

```text
每个共享者都必须设置 IRQF_SHARED
dev_id 不能为 NULL
handler 必须能判断是不是自己的中断
```

为什么 `dev_id` 不能为 NULL？

因为释放中断时：

```c
free_irq(irq, dev_id);
```

内核要靠 `dev_id` 找到 action 链表里对应的那个节点。

## 11.4 request_threaded_irq()

典型用法：

```c
ret = request_threaded_irq(irq,
                           xxx_irq_handler,
                           xxx_irq_thread,
                           IRQF_ONESHOT,
                           "xxx-device",
                           dev);
```

分成两段：

```text
primary handler：
    hardirq 上下文执行
    要快
    不能睡眠

thread_fn：
    内核线程上下文执行
    可以睡眠
    可以拿 mutex
    可以访问 I2C/SPI 这类可能睡眠的接口
```

典型代码：

```c
static irqreturn_t xxx_irq_handler(int irq, void *data)
{
    struct xxx_dev *dev = data;

    /*
     * 快速判断、mask、读取少量状态。
     * 如果需要线程处理，返回 IRQ_WAKE_THREAD。
     */
    return IRQ_WAKE_THREAD;
}

static irqreturn_t xxx_irq_thread(int irq, void *data)
{
    struct xxx_dev *dev = data;

    /*
     * 可以睡眠。
     * 可以使用 mutex。
     * 可以做较慢的业务处理。
     */
    mutex_lock(&dev->lock);
    xxx_process_event(dev);
    mutex_unlock(&dev->lock);

    return IRQ_HANDLED;
}
```

## 11.5 handler 可以为 NULL 吗？

可以。

如果：

```c
request_threaded_irq(irq, NULL, thread_fn, flags, name, dev);
```

内核会使用默认 primary handler。这个默认 handler 的作用基本就是：

```text
返回 IRQ_WAKE_THREAD
```

也就是中断一来就唤醒 irq thread。

## 11.6 IRQF_ONESHOT

threaded irq 经常配合：

```c
IRQF_ONESHOT
```

作用是：

```text
在线程 handler 运行期间，保持该 IRQ masked；
等 thread_fn 执行完，再 unmask。
```

为什么需要？

假设没有 `IRQF_ONESHOT`：

```text
中断来了
    ↓
primary handler 返回 IRQ_WAKE_THREAD
    ↓
thread_fn 还没执行完
    ↓
同一个中断又来了
    ↓
可能重复唤醒、重入、状态混乱
```

`IRQF_ONESHOT` 可以保证：

```text
线程处理没结束之前，不再递送同一个 IRQ。
```

这对 I2C、GPIO 按键、触摸屏等慢速外设很常见。

## 11.7 devm_request_irq()

`devm_` 系列是设备资源管理版本：

```c
devm_request_irq(dev, irq, handler, flags, name, data);
```

好处是：

```text
设备 detach/remove 时自动释放资源
减少驱动 remove 路径里的 free_irq()
避免泄漏
```

但你仍然要理解底层本质：

```text
它最终还是注册 irqaction。
```

# 12. ARM 中断处理过程

wowotech 的 ARM 中断处理文章以 ARM 体系结构为例，讲中断处理前的准备、ARM 硬件发生中断时的行为、中断入口过程和中断退出过程。文章摘要里也明确提到 IRQ mode stack、SVC mode stack、异常向量表和中断控制器初始化等内容。([Wowotech][7])

## 12.1 中断发生前要准备什么？

内核启动时需要准备：

```text
异常向量表
各异常模式的栈
中断控制器初始化
generic IRQ layer 初始化
irq_desc 初始化
irq_domain 初始化
```

对 ARM 来说，CPU 有多种模式，例如：

```text
User mode
SVC mode
IRQ mode
FIQ mode
Abort mode
Undefined mode
System mode
```

中断来了以后，硬件会切到 IRQ mode。但 Linux 不一定在 IRQ mode 下完成所有处理，通常会快速切到更合适的内核栈和 SVC 上下文里处理。

## 12.2 CPU 收到 IRQ 后硬件自动做什么？

简化理解：

```text
当前代码正在运行
    ↓
外设中断到达 CPU
    ↓
CPU 保存 CPSR 到 SPSR_irq
    ↓
保存返回地址到 LR_irq
    ↓
切换到 IRQ mode
    ↓
关闭普通 IRQ
    ↓
跳转到异常向量表 IRQ 入口
```

异常向量表类似：

```text
0x00 reset
0x04 undefined instruction
0x08 software interrupt / SVC
0x0c prefetch abort
0x10 data abort
0x18 IRQ
0x1c FIQ
```

IRQ 发生后，CPU 跳到 IRQ vector。

## 12.3 中断入口汇编做什么？

中断入口代码主要做：

```text
保存通用寄存器
保存返回地址
保存状态寄存器
切换到内核期望的栈
建立 pt_regs
调用 C 语言中断处理函数
```

简化路径：

```text
vector_irq
    ↓
保存现场
    ↓
切换栈
    ↓
asm_do_IRQ / handle_IRQ
    ↓
generic IRQ layer
```

## 12.4 为什么要保存现场？

因为中断是“打断”当前代码。

例如当前进程执行到：

```c
x = a + b;
```

中断来了，CPU 去处理中断。处理完之后，必须回到原来的地方继续执行，而且寄存器状态不能被破坏。

所以必须保存：

```text
通用寄存器
栈指针
链接寄存器
程序状态寄存器
异常返回地址
```

## 12.5 irq_enter() 和 irq_exit()

进入 hardirq 时，内核会标记当前处于中断上下文：

```c
irq_enter();
...
irq_exit();
```

这些函数会影响：

```text
preempt_count
in_interrupt()
softirq 是否在中断返回路径执行
RCU/context tracking
调度行为
```

`irq_exit()` 时，如果发现有 pending softirq，可能会执行 softirq 或唤醒 ksoftirqd。

# 13. GIC 代码分析

GIC，即 Generic Interrupt Controller，是 ARM 平台常见中断控制器。wowotech 的 GIC 文章围绕 GIC 驱动代码展开，涉及 Distributor、CPU interface、pending 状态等；GIC 驱动会注册 irq_chip 和 irq_domain，把 ARM GIC 硬件接入 Linux generic IRQ layer。([Wowotech][8])

## 13.1 GIC 的基本结构

简化结构：

```text
外设
  ↓
GIC Distributor
  ↓
GIC CPU Interface
  ↓
CPU
```

## 13.2 Distributor 做什么？

Distributor 是全局分发器，负责：

```text
管理所有 interrupt source
设置每个中断 enable/disable
设置每个中断 priority
设置每个中断 target CPU
设置 edge/level
维护 pending 状态
维护 active 状态
```

常见寄存器概念：

```text
GICD_CTLR       Distributor 总开关
GICD_ISENABLER  enable interrupt
GICD_ICENABLER  disable interrupt
GICD_ISPENDR    set pending
GICD_ICPENDR    clear pending
GICD_ISACTIVER  active 状态
GICD_ICACTIVER  clear active
GICD_IPRIORITYR priority
GICD_ITARGETSR  target CPU
GICD_ICFGR      trigger type
```

## 13.3 CPU Interface 做什么？

CPU Interface 是每个 CPU 面前的接口，负责：

```text
CPU 当前是否接收中断
当前 CPU 可接收的最低优先级
CPU ack 哪个中断
CPU EOI 哪个中断
```

常见寄存器概念：

```text
GICC_CTLR    CPU interface enable
GICC_PMR     priority mask
GICC_IAR     interrupt acknowledge register
GICC_EOIR    end of interrupt register
```

## 13.4 SGI、PPI、SPI

GIC 中断号分三类：

```text
SGI：Software Generated Interrupt
    ID 0~15
    软件产生，常用于 CPU 间 IPI

PPI：Private Peripheral Interrupt
    ID 16~31
    每个 CPU 私有
    例如 local timer

SPI：Shared Peripheral Interrupt
    ID 32+
    多个 CPU 共享
    普通外设中断大多是 SPI
```

## 13.5 GIC 中断状态机

一个 interrupt source 可能处于：

```text
Inactive：
    没有 pending，也没有 active

Pending：
    中断已经发生，等待 CPU 处理

Active：
    CPU 已经 ack，正在处理

Active + Pending：
    CPU 正在处理上一次中断，同时又来了新的一次
```

这对理解边沿/电平触发、EOI、丢中断、中断风暴都很重要。

## 13.6 GIC 收到中断后的流程

简化流程：

```text
外设拉中断线
    ↓
Distributor 发现对应 interrupt source
    ↓
如果 enabled 且未 masked，设置 pending
    ↓
根据 priority 和 target CPU，投递给某个 CPU interface
    ↓
CPU 收到 IRQ
    ↓
CPU 进入异常向量
    ↓
GIC driver 读取 GICC_IAR
    ↓
得到 hwirq
    ↓
通过 irq_domain 找到 virq
    ↓
generic_handle_irq(virq)
    ↓
调用 irq_desc flow handler
    ↓
调用驱动 handler
    ↓
写 GICC_EOIR，EOI
```

## 13.7 spurious interrupt

GIC 读取 IAR 时可能读到特殊 ID，例如：

```text
1023
```

这通常表示：

```text
没有有效中断
或者中断已经被其他 CPU/路径处理
或者优先级/mask 变化导致当前没有可处理中断
```

驱动里需要识别这种 spurious ID，不能当普通中断处理。

## 13.8 GIC irq_chip

GIC driver 会提供类似：

```c
static struct irq_chip gic_chip = {
    .irq_mask = gic_mask_irq,
    .irq_unmask = gic_unmask_irq,
    .irq_eoi = gic_eoi_irq,
    .irq_set_type = gic_set_type,
    .irq_set_affinity = gic_set_affinity,
};
```

然后通过 irq_domain 的 map 回调设置给某个 virq：

```c
irq_set_chip_and_handler(virq, &gic_chip, handle_fasteoi_irq);
```

所以最终：

```text
generic IRQ layer 负责通用逻辑；
GIC irq_chip 负责写 GIC 寄存器。
```

# 15. softirq 详细总结

wowotech 的 softirq 文章讲 Linux 将中断处理分成 top half 和 bottom half，并重点讨论 softirq 和 tasklet 的差异、context 概念、softirq 执行时机和实现机制。([Wowotech][2])

## 15.1 softirq 是什么？

softirq 是 Linux 的一种下半部机制。

它的目标是：

```text
把 hardirq 中不适合做的耗时工作延后处理；
但又比普通线程更快、更接近中断上下文。
```

典型例子是网络收包：

```text
网卡 hardirq：
    确认收包
    触发 NET_RX_SOFTIRQ
    尽快返回

NET_RX_SOFTIRQ：
    后续处理 skb
    进入协议栈
```

## 15.2 softirq 的特点

```text
运行在 interrupt context
不能睡眠
不能拿 mutex
可以在多个 CPU 上并发执行
同一种 softirq 可以在不同 CPU 同时运行
性能高
并发复杂
```

这和 workqueue 不一样。

```text
softirq：
    interrupt context
    不能睡眠
    高性能
    并发复杂

workqueue：
    process context
    可以睡眠
    由 worker thread 执行
    更适合慢操作
```

## 15.3 softirq 类型

典型 softirq 包括：

```c
enum {
    HI_SOFTIRQ = 0,
    TIMER_SOFTIRQ,
    NET_TX_SOFTIRQ,
    NET_RX_SOFTIRQ,
    BLOCK_SOFTIRQ,
    IRQ_POLL_SOFTIRQ,
    TASKLET_SOFTIRQ,
    SCHED_SOFTIRQ,
    HRTIMER_SOFTIRQ,
    RCU_SOFTIRQ,
    NR_SOFTIRQS
};
```

常见用途：

```text
TIMER_SOFTIRQ：
    处理 timer

NET_RX_SOFTIRQ：
    网络收包

NET_TX_SOFTIRQ：
    网络发包完成

TASKLET_SOFTIRQ：
    普通 tasklet

HI_SOFTIRQ：
    高优先级 tasklet

SCHED_SOFTIRQ：
    调度相关

RCU_SOFTIRQ：
    RCU callback
```

## 15.4 softirq 如何注册？

softirq 是静态注册的，不是普通驱动随便动态添加。

大致形式：

```c
open_softirq(NET_RX_SOFTIRQ, net_rx_action);
```

每个 softirq 编号对应一个处理函数：

```text
softirq_vec[NR_SOFTIRQS]
```

## 15.5 softirq 如何触发？

触发 softirq 通常是：

```c
raise_softirq(NET_RX_SOFTIRQ);
```

本质是设置当前 CPU 的 pending 位：

```text
__softirq_pending
```

可以理解成：

```text
这个 CPU 上有某种 softirq 等待执行。
```

## 15.6 softirq 什么时候执行？

常见时机：

```text
hardirq 返回路径 irq_exit()
local_bh_enable()
ksoftirqd 线程上下文
```

简化流程：

```text
hardirq handler
    ↓
raise_softirq()
    ↓
irq_exit()
    ↓
发现 pending softirq
    ↓
执行 do_softirq()
```

如果 softirq 太多，不能一直在中断返回路径处理，否则用户进程会被饿死。

于是 Linux 会唤醒：

```text
ksoftirqd/N
```

例如：

```text
ksoftirqd/0
ksoftirqd/1
```

## 15.7 为什么会有 ksoftirqd CPU 高？

如果你看到：

```bash
top
# ksoftirqd/3 占用很高 CPU
```

常见原因：

```text
网络包很多
softirq 处理不过来
中断负载集中在某个 CPU
NAPI poll 很重
RCU callback 很多
timer 很多
block softirq 压力大
```

排查常用：

```bash
cat /proc/softirqs
cat /proc/interrupts
mpstat -P ALL 1
sar -I SUM 1
perf top
trace-cmd record -e irq -e napi -e net
```

## 15.8 local_bh_disable/local_bh_enable

`local_bh_disable()` 用于禁止当前 CPU 执行 bottom half。

注意：

```text
它不是全局禁止 softirq；
它只影响当前 CPU；
它通常通过 preempt_count 标记 bottom half disable。
```

常见用途：

```text
保护当前 CPU 上可能被 softirq 并发访问的数据
```

例如网络栈里经常用：

```c
local_bh_disable();
spin_lock(&lock);
/* critical section */
spin_unlock(&lock);
local_bh_enable();
```

# 16. tasklet 详细总结

wowotech 的 tasklet 文章重点讨论 Linux bottom half 机制的选择，指出“越快越好型”有 softirq 和 tasklet，“随遇而安型”有 workqueue 和 threaded irq handler。([Wowotech][10])

## 16.1 tasklet 是什么？

tasklet 是基于 softirq 实现的下半部机制。

你可以理解为：

```text
tasklet = 对 softirq 的进一步封装
```

它让普通驱动更容易使用 bottom half，而不需要自己注册 softirq 类型。

## 16.2 tasklet 的数据结构

典型结构：

```c
struct tasklet_struct {
    struct tasklet_struct *next;
    unsigned long state;
    atomic_t count;
    void (*func)(unsigned long);
    unsigned long data;
};
```

关键字段：

```text
func：
    tasklet 处理函数

data：
    传给 func 的参数

state：
    TASKLET_STATE_SCHED
    TASKLET_STATE_RUN

count：
    disable/enable 计数
```

## 16.3 tasklet 如何使用？

老式写法类似：

```c
static void xxx_tasklet_func(unsigned long data)
{
    struct xxx_dev *dev = (struct xxx_dev *)data;
    /* bottom half work */
}

DECLARE_TASKLET(xxx_tasklet, xxx_tasklet_func, data);
```

中断 handler 中：

```c
static irqreturn_t xxx_irq_handler(int irq, void *dev_id)
{
    tasklet_schedule(&xxx_tasklet);
    return IRQ_HANDLED;
}
```

流程：

```text
hardirq
    ↓
tasklet_schedule()
    ↓
设置 tasklet pending
    ↓
raise_softirq(TASKLET_SOFTIRQ)
    ↓
后续 softirq 执行 tasklet_action()
    ↓
调用 tasklet->func()
```

## 16.4 tasklet 和 softirq 的区别

```text
softirq：
    静态定义
    同一种 softirq 可以在多个 CPU 并发执行
    性能高
    并发复杂
    多用于网络、定时器、RCU 等核心子系统

tasklet：
    动态创建
    基于 softirq
    同一个 tasklet 不会在多个 CPU 上并发执行
    比 softirq 更容易用于普通驱动
```

注意这句话：

```text
同一个 tasklet 不会并发执行；
不同 tasklet 仍然可能在不同 CPU 并发执行。
```

所以如果多个 tasklet 访问同一份数据，仍然需要同步。

## 16.5 tasklet 能睡眠吗？

不能。

因为 tasklet 运行在 softirq context。

所以 tasklet 里不能：

```text
mutex_lock()
msleep()
down_interruptible()
等待 I2C/SPI 同步传输
执行可能阻塞的操作
```

如果你需要睡眠，应使用：

```text
workqueue
threaded irq
kernel thread
```

## 16.6 tasklet 的现代地位

tasklet 在老驱动中很多，但现代内核新代码不太推荐继续引入 tasklet。Linux 社区近年持续推动减少 tasklet，原因包括并发语义复杂、和现代 RT/preempt 模型不够友好、可替代方案更多。LWN 也曾讨论 tasklet 逐步走向终结。([Wowotech][10])

现代选择通常是：

```text
需要极高性能、不能睡眠：
    softirq/NAPI

普通驱动下半部，可能睡眠：
    threaded irq
    workqueue

老代码维护：
    理解 tasklet
```

# 17. workqueue 基本概念

wowotech 的 workqueue 系列从基本概念、CMWQ 概述、创建 workqueue、queue 和处理 work 几个角度展开；官方内核 workqueue 文档也说明，workqueue 是最常用的异步执行上下文机制之一：调用者把 work item 放入队列，由独立 worker 线程执行对应函数。([Wowotech][11])

## 17.1 workqueue 是什么？

workqueue 是 Linux 内核里的异步执行机制。

基本思想：

```text
当前上下文不方便执行某个函数
    ↓
把这个函数封装成 work
    ↓
扔到 workqueue
    ↓
后面由 worker thread 在进程上下文执行
```

这和生活中“待办任务队列”类似：

```text
你正在接电话，不能马上写报告；
你把“写报告”记到待办事项；
稍后由专门工作人员处理。
```

## 17.2 workqueue 和 softirq/tasklet 最大区别

```text
workqueue 运行在进程上下文；
softirq/tasklet 运行在中断上下文。
```

因此 workqueue 可以：

```text
睡眠
拿 mutex
等待 completion
执行阻塞式 I/O
做较慢操作
```

而 softirq/tasklet 不行。

## 17.3 workqueue 常见 API

定义 work：

```c
struct work_struct work;
```

初始化：

```c
INIT_WORK(&work, work_func);
```

提交：

```c
schedule_work(&work);
queue_work(system_wq, &work);
queue_work(my_wq, &work);
```

处理函数：

```c
static void work_func(struct work_struct *work)
{
    /* process context */
}
```

延迟 work：

```c
struct delayed_work dwork;

INIT_DELAYED_WORK(&dwork, delayed_func);
schedule_delayed_work(&dwork, msecs_to_jiffies(100));
```

取消：

```c
cancel_work_sync(&work);
cancel_delayed_work_sync(&dwork);
```

等待完成：

```c
flush_work(&work);
flush_workqueue(wq);
```

## 17.4 一个典型驱动场景

```c
static irqreturn_t xxx_irq_handler(int irq, void *data)
{
    struct xxx_dev *dev = data;

    /* hardirq 里只做最小处理 */
    disable_irq_nosync(irq);
    queue_work(dev->wq, &dev->work);

    return IRQ_HANDLED;
}

static void xxx_work_func(struct work_struct *work)
{
    struct xxx_dev *dev = container_of(work, struct xxx_dev, work);

    mutex_lock(&dev->lock);
    xxx_slow_process(dev);
    mutex_unlock(&dev->lock);

    enable_irq(dev->irq);
}
```

这里：

```text
hardirq：
    快速屏蔽中断并排队 work

workqueue：
    在线程上下文里慢慢处理
```

# 18. 为什么需要 CMWQ？

CMWQ 是 Concurrency Managed Workqueue。

wowotech 的 CMWQ 文章指出，旧 workqueue 机制在 2.6.36 前较简单，每个 workqueue 往往有自己的 worker 线程，随着 CPU 核数和 workqueue 用户增加，会造成资源浪费；CMWQ 引入统一 worker pool，让多个 workqueue 共享后端 worker pool，由内核自动管理并发。官方文档也提到，旧 MT/ST workqueue 存在线程资源浪费、并发能力不足、容易围绕单执行上下文产生死锁等问题，CMWQ 的目标是兼容原 API，同时通过 per-CPU unified worker pools 和动态并发管理减少资源浪费。([Wowotech][12])

## 18.1 旧 workqueue 的问题

老模型大概是：

```text
每个 workqueue 自己创建 worker thread
```

问题：

```text
workqueue 很多 → worker thread 很多 → PID/内存浪费
每个 workqueue 并发能力固定
某些 work 阻塞会拖住同队列其他 work
驱动作者需要自己考虑并发数量
```

例如：

```text
100 个子系统各建一个 per-cpu workqueue；
机器有 128 CPU；
可能产生大量 worker 线程。
```

## 18.2 CMWQ 的思想

CMWQ 把 workqueue 分成两层：

```text
前端：
    workqueue_struct
    给用户/驱动看的队列抽象

后端：
    worker_pool
    真正执行 work 的 worker 线程池
```

也就是说：

```text
驱动看到的是自己的 workqueue；
内核底层可能让多个 workqueue 共享同一个 worker_pool。
```

这样可以：

```text
减少线程数量
提高并发弹性
避免每个 workqueue 自己维护线程池
由内核统一判断是否需要创建更多 worker
```

# 19. CMWQ 核心结构体

## 19.1 work_struct

代表一个具体任务：

```c
struct work_struct {
    atomic_long_t data;
    struct list_head entry;
    work_func_t func;
};
```

核心是：

```text
func：
    要异步执行的函数

entry：
    挂入队列

data：
    pending/running/pool 信息
```

## 19.2 workqueue_struct

用户看到的 workqueue：

```text
struct workqueue_struct
```

它代表：

```text
一个逻辑 workqueue
```

例如：

```c
struct workqueue_struct *wq;
wq = alloc_workqueue("xxx_wq", WQ_UNBOUND, 0);
```

## 19.3 worker

真正执行 work 的内核线程：

```text
kworker/u16:0
kworker/3:1
kworker/0:2H
```

每个 worker 对应一个内核线程。

它的逻辑是：

```text
从 pool 的 worklist 取 work
    ↓
执行 work->func()
    ↓
执行完继续取下一个
    ↓
没活就睡眠
```

## 19.4 worker_pool

worker_pool 是 worker 线程池。

```text
每个 CPU 通常有普通优先级 pool
每个 CPU 通常有 high priority pool
unbound workqueue 还有动态 worker pools
```

## 19.5 pool_workqueue

`pool_workqueue`，简称 pwq，是中间连接层：

```text
workqueue_struct
    ↓
pool_workqueue
    ↓
worker_pool
```

它记录：

```text
某个 workqueue 在某个 worker_pool 上的队列和并发限制
```

# 20. workqueue 类型和 flags

创建 workqueue：

```c
alloc_workqueue(name, flags, max_active);
```

常见 flags：

```text
WQ_UNBOUND
WQ_HIGHPRI
WQ_FREEZABLE
WQ_MEM_RECLAIM
WQ_CPU_INTENSIVE
WQ_POWER_EFFICIENT
```

## 20.1 bound workqueue

默认一般是 bound。

含义：

```text
work 绑定到某个 CPU 的 worker_pool 执行
```

例如在 CPU3 上 queue work，可能由 CPU3 的 kworker 执行。

优点：

```text
cache locality 好
和 CPU 本地数据配合更好
```

缺点：

```text
如果这个 CPU 很忙，work 可能延迟
```

## 20.2 WQ_UNBOUND

unbound workqueue 不绑定具体 CPU。

含义：

```text
work 可以在系统合适的 CPU 上执行
```

适合：

```text
不关心 CPU 亲和性
可能阻塞
可能耗时
希望更灵活调度
```

## 20.3 WQ_HIGHPRI

高优先级 workqueue。

含义：

```text
使用高优先级 worker_pool
kworker nice 更低
```

适合：

```text
延迟敏感的异步任务
```

但不能滥用，否则影响系统公平性。

## 20.4 WQ_FREEZABLE

系统 suspend/freezer 时可以冻结。

适合：

```text
电源管理相关
suspend 时不应继续执行的 work
```

## 20.5 WQ_MEM_RECLAIM

内存回收路径中可能使用的 workqueue 必须考虑这个 flag。

原因：

```text
内存回收时系统可能缺内存
如果 workqueue 无法创建 worker
而内存回收又等待这个 work 完成
就可能死锁
```

`WQ_MEM_RECLAIM` 会配套 rescuer 线程，避免 reclaim 死锁。wowotech 的 CMWQ 概述文章也提到 rescuer thread 用来接手处理，从而解除 deadlock。([Wowotech][12])

## 20.6 WQ_CPU_INTENSIVE

表示 work 是 CPU 密集型。

含义：

```text
这个 work 可能长时间占 CPU
不要让它阻塞同 pool 并发管理
```

## 20.7 WQ_POWER_EFFICIENT

倾向省电。

可能把部分 work 合并到更少 CPU 上执行，减少唤醒 CPU 数量。

# 21. 创建 workqueue：alloc_workqueue 过程

wowotech 的“创建 workqueue 代码分析”文章以 `__alloc_workqueue_key()` 为主线，分析创建 workqueue 实例的代码过程。([Wowotech][13])

简化路径：

```text
alloc_workqueue()
    ↓
__alloc_workqueue_key()
    ↓
分配 workqueue_struct
    ↓
处理 flags
    ↓
设置 max_active
    ↓
分配/初始化 pool_workqueue
    ↓
关联 worker_pool
    ↓
如果 WQ_MEM_RECLAIM，创建 rescuer
    ↓
加入全局 workqueues 链表
```

## 21.1 max_active 是什么？

`max_active` 表示同一个 workqueue 在某个 pool 上最多同时 active 的 work 数量。

例如：

```c
alloc_workqueue("xxx", 0, 1);
```

表示：

```text
最多一个 active work
```

这可以实现类似 ordered/single-thread 行为。

如果：

```c
alloc_workqueue("xxx", WQ_UNBOUND, 0);
```

`0` 表示使用默认并发限制。

## 21.2 ordered workqueue

ordered workqueue 保证 work 顺序执行。

适合：

```text
必须严格串行的异步任务
```

但缺点是：

```text
一个 work 卡住，后面全部卡住
```

# 22. queue work 和处理 work

wowotech 的“workqueue 如何处理 work”文章重点讲两件事：如何把 work 挂入 workqueue，以及 worker 如何处理挂入 workqueue 的 work。([Wowotech][14])

## 22.1 queue_work()

典型路径：

```text
queue_work(wq, work)
    ↓
queue_work_on(WORK_CPU_UNBOUND, wq, work)
    ↓
__queue_work(cpu, wq, work)
    ↓
选择 pool_workqueue
    ↓
如果 work 已经 pending，则不重复入队
    ↓
把 work 插入 pwq/pool 的 worklist
    ↓
唤醒 worker
```

一个重要语义：

```text
同一个 work_struct 如果已经 pending，再次 queue_work 通常不会重复排队。
```

这意味着：

```text
频繁 queue 同一个 work，可能只执行一次；
如果你需要记录事件次数，需要自己计数。
```

## 22.2 worker_thread()

worker 线程大概逻辑：

```text
worker_thread()
    ↓
循环
    ↓
如果没有 work，睡眠
    ↓
有 work，取出一个
    ↓
process_one_work()
    ↓
执行 work->func(work)
    ↓
执行完清状态
    ↓
继续下一个
```

## 22.3 process_one_work()

核心动作：

```text
标记当前 worker 正在执行这个 work
从 worklist 移除
清 pending 标志
调用 work->func()
处理完成后的状态清理
```

## 22.4 work_func 里可以睡眠吗？

可以。

因为 workqueue 运行在 worker thread 中，属于 process context。

所以可以：

```text
mutex_lock()
msleep()
wait_for_completion()
blocking I/O
调用可能调度的接口
```

但注意：

```text
不能在持有 spinlock 时睡眠
不能阻塞系统关键 workqueue 太久
不要把系统 workqueue 当作无限资源
```

## 22.5 cancel_work_sync 和 flush_work 的区别

```text
flush_work(work)：
    等待已经排队或正在执行的 work 完成
    不阻止之后重新 queue

cancel_work_sync(work)：
    尝试取消 pending work
    如果已经在运行，则等待运行完成
    返回后确保该 work 当前不 pending、不 running
```

常见 remove 路径：

```c
free_irq(dev->irq, dev);
cancel_work_sync(&dev->work);
```

否则可能出现：

```text
设备内存已经释放
work 后面又执行
访问已释放内存
use-after-free
```

# 23. threaded irq 和 workqueue 怎么选？

这是驱动里很常见的问题。

## 23.1 threaded irq 适合什么？

适合：

```text
这个工作强绑定于某个 IRQ
需要在中断发生后尽快处理
处理过程中可能睡眠
需要 IRQF_ONESHOT 配合流控
```

例如：

```text
触摸屏中断后通过 I2C 读取坐标
GPIO 按键中断后做 debounce
传感器数据 ready 后通过 SPI 读取
```

## 23.2 workqueue 适合什么？

适合：

```text
工作不一定必须绑定 IRQ 线程
可能被多种路径触发
可以延迟一点
需要统一异步处理
需要 flush/cancel 管理生命周期
```

例如：

```text
状态机后处理
设备 reset
固件加载
耗时配置
定期维护任务
```

## 23.3 softirq/tasklet 适合什么？

适合：

```text
不能睡眠
需要非常快
性能敏感
通常是内核核心子系统，不是普通驱动首选
```

普通驱动现代写法优先级通常是：

```text
能用 threaded irq 就用 threaded irq
复杂异步任务用 workqueue
高性能网络路径用 NAPI/softirq
维护老代码才大量遇到 tasklet
```

# 24. 中断唤醒系统流程

wowotech 的“中断唤醒系统流程”文章讨论外设如何通过 interrupt pin 电平变化唤醒系统，以及设备、SoC、中断控制器、电源管理之间如何配合。([Wowotech][15])

## 24.1 普通中断和唤醒中断不是一回事

普通运行时：

```text
设备产生中断
    ↓
CPU 正在运行
    ↓
中断控制器投递
    ↓
handler 执行
```

系统 suspend 后：

```text
CPU 可能睡了
部分电源域关了
部分中断被 suspend_device_irqs() disable
部分 controller 进入低功耗
设备 pinmux 可能切到 sleep state
```

所以：

```text
普通中断能工作 ≠ suspend 后一定能唤醒。
```

## 24.2 wakeup source

如果一个设备可以唤醒系统，需要配置：

```c
device_init_wakeup(dev, true);
```

表示：

```text
这个 device 可以作为 wakeup source。
```

还可能需要：

```c
enable_irq_wake(irq);
```

表示：

```text
允许这个 IRQ 作为系统唤醒源。
```

## 24.3 enable_irq_wake()

`enable_irq_wake()` 最终会调用 irqchip 的：

```c
irq_set_wake()
```

例如：

```text
GPIO irqchip 配置某个 GPIO 为 wakeup
GIC/PMIC 配置某个中断低功耗可唤醒
```

如果底层 irqchip 不支持 wake，可能失败。

## 24.4 suspend_device_irqs()

系统 suspend 过程中，内核会 disable 大多数普通 IRQ。

但是 wakeup IRQ 需要特殊处理。

大致逻辑：

```text
普通 IRQ：
    suspend 时 disable

wakeup IRQ：
    保留唤醒能力
    可能不按普通路径递送 handler
    先唤醒系统，再 resume 后处理
```

## 24.5 IRQF_NO_SUSPEND

`IRQF_NO_SUSPEND` 表示：

```text
suspend_device_irqs() 阶段不要禁用这个 IRQ。
```

但这不是普通驱动随便用的。

滥用会导致：

```text
suspend 阶段仍然有中断 handler 执行
访问已经 suspend 的设备
破坏电源管理时序
```

## 24.6 中断唤醒失败如何排查？

按这个顺序：

```text
1. 外设是否真的产生了 wake 信号？
2. pinctrl sleep state 是否正确？
3. GPIO 是否配置成输入/中断功能？
4. interrupt-parent 是否正确？
5. 中断触发类型是否正确？level/edge/high/low
6. device_init_wakeup() 是否调用？
7. enable_irq_wake() 是否成功？
8. 对应 irqchip 是否实现 irq_set_wake？
9. suspend 时该 IRQ 是否被错误 disable？
10. 电源域是否把外设或 GPIO controller 关掉了？
11. resume 后是否进入了 handler？
12. /sys/kernel/debug/wakeup_sources 里是否有记录？
```

# 25. /proc/interrupts 怎么看？

`/proc/interrupts` 是理解中断运行情况的核心入口。

示例：

```text
           CPU0       CPU1       CPU2       CPU3
 32:      1000          0          0          0  GICv3  27 Level     arch_timer
 45:         0      23890          0          0  GICv3  89 Level     eth0
120:        20          0          0          0  gpio   12 Edge      power-key
```

字段含义：

```text
IRQ number：
    Linux IRQ number

CPU0/CPU1/...：
    每个 CPU 上处理该 IRQ 的次数

中断控制器：
    GICv3 / gpio / MSI 等

hwirq：
    控制器内部编号

触发类型：
    Level / Edge

name：
    request_irq 时传入的 name
```

如果某个 IRQ 计数疯狂增长：

```text
可能中断风暴
```

如果某个 IRQ 永远不增长：

```text
中断没有触发
触发类型错
设备没产生中断
irq_domain 映射错
中断被 mask
affinity 问题
```

# 27. IRQ affinity

中断可以绑定 CPU。

查看：

```bash
cat /proc/irq/<irq>/smp_affinity
cat /proc/irq/<irq>/smp_affinity_list
```

设置：

```bash
echo 3 > /proc/irq/<irq>/smp_affinity_list
```

含义：

```text
让这个 IRQ 尽量投递到 CPU3。
```

为什么是“尽量”？

因为：

```text
不是所有中断都支持 affinity
per-cpu IRQ 不能随便改
中断控制器需要支持 set_affinity
irqbalance 可能又改回来
```

GIC SPI 通常可以设置 target CPU；PPI 是每 CPU 私有，语义不同。

# 28. 中断上下文限制总结

在 hardirq/softirq/tasklet/workqueue/threaded irq 中，限制不同。

| 上下文           | 能睡眠？ | 能拿 mutex？ | 典型用途               |
| ------------- | ---: | --------: | ------------------ |
| hardirq       |   不能 |        不能 | 最快确认中断、清状态、唤醒下半部   |
| softirq       |   不能 |        不能 | 网络、timer、RCU、block |
| tasklet       |   不能 |        不能 | 老驱动 bottom half    |
| threaded irq  |   可以 |        可以 | 中断相关的慢处理           |
| workqueue     |   可以 |        可以 | 通用异步任务             |
| kernel thread |   可以 |        可以 | 长期后台任务             |

口诀：

```text
hardirq/softirq/tasklet：不能睡眠；
threaded irq/workqueue/kthread：可以睡眠。
```

# 29. 中断风暴

中断风暴是指某个 IRQ 高频触发，系统大量时间耗在中断里。

常见原因：

```text
电平触发中断，驱动没有清设备中断源
触发类型配置错，例如设备是低电平，配置成边沿
硬件异常，interrupt line 一直有效
共享中断 handler 返回不当
中断被 unmask 太早
外设状态机异常
```

现象：

```bash
cat /proc/interrupts
```

某个 IRQ 计数飞速增长。

可能伴随：

```text
CPU sys/irq 很高
ksoftirqd 高
系统卡顿
日志刷屏
```

排查思路：

```text
1. 看 /proc/interrupts 哪个 IRQ 增长最快
2. 找到对应设备和驱动
3. 确认触发类型
4. 检查 handler 是否清设备 status
5. 检查 mask/ack/eoi/unmask 顺序
6. 临时 disable_irq() 验证
7. 看硬件线电平或寄存器 pending 位
```

---

# 30. 丢中断

丢中断表现为：

```text
外设明明发生事件，但 handler 没执行
```

常见原因：

```text
irq_domain 映射错
设备树 interrupts 写错
interrupt-parent 错
触发类型错
中断被 mask/disable
没有 enable 设备内部 interrupt
没有 enable 中断控制器对应位
共享中断 handler 判断错误
边沿触发期间没有正确处理 pending
驱动清状态顺序错误
```

排查：

```text
1. 看设备内部 interrupt status
2. 看中断控制器 pending bit
3. 看 /proc/interrupts 计数
4. 看驱动 handler 是否注册成功
5. 看 request_irq 返回值
6. 看触发类型
7. 看 pinctrl/pinmux
8. 看 irq affinity
```

# 32. 一个完整的中断处理示例

假设有一个 I2C 触摸屏。

设备树：

```dts
touch@5d {
    compatible = "xxx,touch";
    reg = <0x5d>;
    interrupt-parent = <&gpio1>;
    interrupts = <12 IRQ_TYPE_EDGE_FALLING>;
};
```

启动时：

```text
GPIO irqchip driver 创建 gpio1 irq_domain
GIC driver 创建 gic irq_domain
touch driver probe
    ↓
platform_get_irq()
    ↓
解析 interrupt-parent = gpio1
    ↓
gpio1 irq_domain 把 GPIO12 映射成 virq
    ↓
request_threaded_irq()
    ↓
irq_desc->action 挂上 touch handler
```

运行时：

```text
手指触摸屏幕
    ↓
触摸芯片拉低 INT 引脚
    ↓
GPIO controller 检测 falling edge
    ↓
GPIO parent IRQ 上报 GIC
    ↓
GIC 投递给 CPU
    ↓
CPU 进入 IRQ exception
    ↓
GIC driver 读取 hwirq
    ↓
找到 GPIO parent virq
    ↓
GPIO chained handler 读取 GPIO pending
    ↓
发现 GPIO12 触发
    ↓
generic_handle_irq(touch virq)
    ↓
handle_edge_irq()
    ↓
touch primary handler 返回 IRQ_WAKE_THREAD
    ↓
touch irq thread 被唤醒
    ↓
thread_fn 通过 I2C 读取坐标
    ↓
input_report_abs()
    ↓
用户空间收到触摸事件
```

这个例子串起了：

```text
GIC
GPIO irqchip
irq_domain
irq_desc
high level handler
request_threaded_irq
threaded handler
```

# 33. 这组文档真正要掌握的主干

如果你没时间看原文，至少要把下面这些点掌握牢。

## 33.1 中断编号体系

```text
hwirq：
    硬件中断控制器内部编号

virq / IRQ number：
    Linux 内核逻辑编号

irq_domain：
    hwirq 到 virq 的映射器
```

核心问题：

```text
驱动拿到的 irq 通常不是硬件中断号。
```

## 33.2 中断描述体系

```text
irq_desc：
    一个 IRQ 的总管理结构

irq_data：
    保存硬件相关映射和 chip 信息

irq_chip：
    操作中断控制器硬件

irqaction：
    驱动注册的 handler
```

核心关系：

```text
virq → irq_desc → irq_data/irq_chip/action
```

## 33.3 中断处理路径

```text
CPU exception entry
    ↓
arch handler
    ↓
irqchip read hwirq
    ↓
irq_domain map to virq
    ↓
generic_handle_irq
    ↓
desc->handle_irq
    ↓
handle_irq_event
    ↓
driver handler
```

## 33.4 中断流控

```text
handle_level_irq：
    电平触发流控

handle_edge_irq：
    边沿触发流控

handle_fasteoi_irq：
    EOI 型控制器

handle_percpu_irq：
    per-cpu IRQ

handle_nested_irq：
    嵌套中断
```

## 33.5 下半部选择

```text
hardirq：
    最快，不睡眠

softirq：
    高性能，不睡眠，多 CPU 并发

tasklet：
    基于 softirq，同一个 tasklet 不并发，老机制

threaded irq：
    和 IRQ 强相关，可以睡眠

workqueue：
    通用异步任务，可以睡眠
```

## 33.6 GIC

```text
Distributor：
    管全局中断源、enable、priority、target、pending

CPU Interface：
    和每个 CPU 对接，IAR/EOIR/PMR

SGI：
    软件中断/IPI

PPI：
    每 CPU 私有中断

SPI：
    共享外设中断
```

## 33.7 workqueue/CMWQ

```text
work_struct：
    一个异步任务

workqueue_struct：
    用户看到的逻辑队列

worker：
    真正执行 work 的 kworker 线程

worker_pool：
    后端线程池

pool_workqueue：
    workqueue 和 worker_pool 的连接
```

核心思想：

```text
CMWQ 让多个 workqueue 共享 worker_pool，由内核管理并发，减少线程浪费。
```

## 33.8 中断唤醒

```text
普通中断正常 ≠ suspend 后可唤醒

需要：
    device_init_wakeup()
    enable_irq_wake()
    irqchip 支持 irq_set_wake()
    pinctrl sleep state 正确
    电源域没有错误关闭
```

# 34. 最容易混淆的 20 个点

## 1. hwirq 和 virq 是不是一个东西？

不是。

```text
hwirq 是硬件编号；
virq 是 Linux 编号。
```

## 2. request_irq() 传入的是 hwirq 吗？

通常不是。

```text
request_irq() 传入 Linux IRQ number。
```

## 3. irq_domain 是普通驱动创建的吗？

一般不是。

```text
中断控制器驱动创建 irq_domain；
普通设备驱动通过平台 API 拿 virq。
```

## 4. irq_chip 是驱动 handler 吗？

不是。

```text
irq_chip 操作中断控制器；
driver handler 处理设备业务。
```

## 5. handle_level_irq 是驱动写的吗？

不是。

```text
它是 generic IRQ layer 的 flow handler。
```

## 6. ack 和清设备中断源一样吗？

不一样。

```text
ack 是确认中断控制器；
清设备中断源是清外设状态寄存器。
```

## 7. eoi 和 unmask 一样吗？

不一样。

```text
eoi 是告诉中断控制器处理结束；
unmask 是允许中断再次上报。
```

## 8. 电平触发为什么容易中断风暴？

因为：

```text
设备状态不清，中断线一直有效。
```

## 9. 边沿触发为什么可能丢事件？

因为：

```text
边沿是一瞬间；
处理期间新边沿需要 pending 机制保存。
```

## 10. softirq 能睡眠吗？

不能。

## 11. tasklet 能睡眠吗？

不能。

## 12. workqueue 能睡眠吗？

能。

## 13. threaded irq 能睡眠吗？

thread_fn 能，primary handler 不能。

## 14. IRQF_ONESHOT 是干嘛的？

```text
thread_fn 执行期间保持 IRQ masked，避免同一 IRQ 重入。
```

## 15. shared IRQ 为什么 dev_id 不能为 NULL？

因为：

```text
free_irq() 要靠 dev_id 找到对应 irqaction。
```

## 16. ksoftirqd 高说明什么？

说明：

```text
softirq 压力大，中断返回路径处理不过来。
```

## 17. workqueue 里的 work 会重复排队吗？

同一个 `work_struct` 已经 pending 时，通常不会重复入队。

## 18. cancel_work_sync() 是否必要？

设备 remove 时通常很重要，否则可能 work 访问已释放设备。

## 19. enable_irq_wake() 和 enable_irq() 一样吗？

不一样。

```text
enable_irq() 是普通运行时使能 IRQ；
enable_irq_wake() 是允许 IRQ 唤醒系统。
```

## 20. 普通中断正常，为什么 suspend 后不能唤醒？

可能是：

```text
wakeup source 没开
irq_set_wake 不支持
pinctrl sleep 配错
GPIO/PMIC 电源域关了
触发类型错
suspend_device_irqs 阶段被禁用
```
